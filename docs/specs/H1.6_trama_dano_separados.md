## 0. Decisiones de diseño de sistema que cierra esta spec

### 0.1 Alcance mínimo de "algo bloquea daño": `leaderShield` (Escudo del Líder, GDD §2.8) — NO Aliados (H1.15)

El criterio de aceptación pide un test que muestre "Aliado bloquea daño pero no Trama". Aliados como entidad completa (carta en mesa, vida propia, redirección por elección del jugador, Berserker, Arrollar hacia el Líder cuando el Aliado muere) es **H1.15**, todavía no implementada, y no existe ningún sistema de mesa/cartas en el motor (H1.8/H1.12 tampoco existen). Inventar aquí una versión reducida de "Aliado" (con vida propia, aunque sea de un único Aliado fijo) adelantaría diseño de esa historia sin necesidad.

El propio GDD ya describe, en una sección **distinta y anterior** a Aliados (§2.8, "Defensa y Escudos"), un mecanismo de absorción de daño que pertenece exclusivamente al Líder y es más simple que Aliados:
- *"Defensa X crea X fichas de escudo persistentes [...]. Cada punto de daño entrante consume 1 ficha antes de tocar tu vida."*
- *"El daño no tiene Arrollar por defecto: si el daño supera las fichas de escudo, el exceso se pierde [...]. Solo pasa si la habilidad tiene la keyword Arrollar."*
- *"Tope global de escudo del Líder: 5."*
- *"Los Aliados no generan escudo. El escudo es del Líder."* — confirma explícitamente que Escudo y Aliado son dos mecanismos de absorción **distintos**, no uno un subconjunto del otro.

**Decisión:** H1.6 modela `leaderShield` (contador simple, entero, rango `[0, LEADER_SHIELD_MAX]` con `LEADER_SHIELD_MAX = 5`, GDD §2.8) como el "algo que bloquea daño" mínimo que satisface el criterio de aceptación, en vez de un Aliado de juguete. Ventaja adicional no solo de alcance sino de corrección de diseño: la absorción de Escudo es **automática e incondicional** (GDD §2.8: "consume 1 ficha antes de tocar tu vida", sin elección del jugador), mientras que la absorción de Aliado es **opcional/redirigida por el jugador** (GDD §3.3: "el jugador elige libremente si redirigir el daño al Aliado o no — sin gastar acción"). Modelar Escudo automático no necesita ningún `CombatCommand` nuevo (no hay decisión que tomar); modelar Aliado sí necesitaría `REDIRECT_DAMAGE` (ya knowed y correctamente diferido a H1.15 en `architecture_stack.md` §2.2). Esto confirma que Escudo es la vía de menor superficie de cambio para H1.6.

**Explícitamente NO modelado aquí (queda para H1.15):** mesa de cartas Aliado, vida propia de un Aliado, redirección por elección del jugador (`REDIRECT_DAMAGE`), Berserker, Arrollar de exceso *hacia el Líder cuando un Aliado muere* (el Arrollar de **Escudo**, en cambio, sí se modela — ver §0.3 — porque está descrito en la misma sección §2.8, no en §3.7).

**Cómo se puebla `leaderShield` en runtime:** en H1.6 no existe ninguna carta/habilidad "Defensa X" que genere fichas de escudo en combate (eso es contenido, H1.9/H1.12, con su propio `AbilityEffectDefinition` o `CardDefinition` todavía sin diseñar). `leaderShield` se siembra únicamente como valor inicial inyectado en `CombatEngineConfig.initialLeaderShield` (igual patrón que `poolSize`/`initialTurnOwner`, ver §2.2), suficiente para que los tests de esta historia demuestren el pipeline de absorción sin inventar cómo se genera el escudo en juego real — eso es una historia de contenido futura, no bloqueante aquí.

### 0.2 H1.6 conecta `resolveAbilityUmbral` (H1.5) con mutación de estado real — nuevo mapa inyectado `abilityEffects`

Siguiendo el mismo patrón que `abilityCoreCosts` (H1.3) y `abilityCooldowns` (H1.4), se añade a `CombatEngineConfig` un tercer mapa **opcional** `abilityEffects: ReadonlyMap<AbilityId, AbilityEffectDefinition>`. Es opcional (a diferencia de los otros dos, que son obligatorios) porque, a diferencia de coste/cooldown, **no toda habilidad tiene todavía un efecto numérico modelado** — todos los tests de H1.3/H1.4/H1.5 siguen funcionando sin pasar `abilityEffects` (sin romper nada existente), y una habilidad sin entrada en este mapa simplemente consume Núcleo + gasta CD sin mutar `leaderDamage`/`scenarioPlot`/`leaderShield`, exactamente igual que hoy.

`AbilityEffectDefinition` (§2.1) es una unión discriminada con **exactamente dos variantes, mutuamente excluyentes por construcción del tipo** (una `abilityId` mapea a un único valor del `Map`, nunca a los dos a la vez) — esto satisface literalmente el criterio *"Una habilidad nunca hace ambas cosas"* a nivel de tipos, no solo de test:

- `{ kind: 'ATTACK', formula: AbilityUmbralDefinition, arrollar?: boolean }` — Ataque del Enemigo (⚔️, GDD §3.4). Reutiliza `AbilityUmbralDefinition` de H1.5 tal cual (sin cambios en `types/umbral.ts`/`umbral.ts`) porque GDD §12 describe explícitamente "Ataque"/"Ataque +X"/"Ataque ×X" como fórmulas alimentadas por el valor del Núcleo gastado — el caso de uso exacto para el que se diseñó Umbral.
- `{ kind: 'PLOT', amount: number }` — Trama del Enemigo o del Líder (📜, GDD §3.6). **Deliberadamente NO usa `AbilityUmbralDefinition`** — ver §0.3 para la justificación completa de por qué esta fórmula es una magnitud fija, no alimentada por Núcleo.

`CombatEngine.handleActivateAbility` (§3.3), tras validar y consumir el Núcleo (mismo camino que hoy), consulta `abilityEffects.get(command.abilityId)`: si existe, resuelve el efecto (`applyAttackEffect`/`applyPlotEffect`, §3.2) y muta `leaderDamage`/`leaderShield` o `scenarioPlot` en consecuencia, emitiendo un evento nuevo (`LEADER_DAMAGED`/`SCENARIO_PLOT_CHANGED`, §2.4). Este es el primer punto del motor donde se aplica un efecto real de habilidad — exactamente la integración que H1.5 §0.1/§7 dejó preparada (`resolveAbilityUmbral(definition, event.nucleoSpent.value)`).

### 0.3 Por qué "Trama X" es una magnitud fija (`amount: number`), no una `AbilityUmbralDefinition`

GDD §12 (tabla de keywords) describe cada keyword de Ataque mencionando explícitamente "Núcleo" en su fórmula ("Ataque: Daño = valor del Núcleo"; "Ataque +X: Daño = Núcleo + X"; "Ataque ×X: Daño = Núcleo × X"). La fila de **"Trama X"**, en cambio, dice literalmente: *"Mueve el contador del Escenario X pasos (dirección por contexto) — Enemigo sube, jugador baja"* — sin mencionar Núcleo en absoluto, a diferencia de las filas de Ataque y de "Caos (auto-daño)" (que sí lo mencionan explícitamente). La propia spec de H1.5 (§0.3) ya hizo este análisis literal al decidir el alcance de Umbral: *"'Trama X' [...] es un número fijo de la carta/habilidad, no una fórmula alimentada por el valor del Núcleo [...]. Si contenido futuro define una habilidad tipo 'Trama = Núcleo', [...] usa exactamente la misma `UmbralFormula`/`resolveAbilityUmbral` [...] sin necesitar ningún tipo nuevo."*

**Nota de consistencia:** el §7 de la spec de H1.5 (sección prospectiva, más informal) sugería de forma más laxa que "el código que aplique el efecto de una habilidad de Ataque/Trama llama a `resolveAbilityUmbral`" para ambas — esta spec resuelve la aparente tensión a favor del análisis literal más cuidadoso del propio §0.3 de esa misma historia (que interpreta la tabla del GDD keyword por keyword), no del resumen prospectivo del §7. Esto no es una contradicción del Programmer con historias pasadas: ninguna de las dos secciones de H1.5 tocó código ni comprometió una firma; el §7 era una nota de expectativa, no una decisión cerrada.

**Consecuencia:** `AbilityEffectDefinition` de tipo `PLOT` lleva `amount: number` (magnitud fija, siempre positiva, GDD "Trama X" sin signo — la dirección la decide el `side` dueño de la habilidad, ver §0.4). El día que contenido futuro necesite una habilidad "Trama = Núcleo" o "Trama = Núcleo + X", basta con añadir una variante adicional a `AbilityEffectDefinition` (ej. `{ kind: 'PLOT_UMBRAL', formula: AbilityUmbralDefinition }`) reutilizando `resolveAbilityUmbral` sin tocar nada de esta historia — exactamente la puerta que H1.5 §0.3 ya dejó documentada.

### 0.4 Bidireccionalidad de Trama: dirección derivada del `side` dueño de la habilidad, piso en 0, sin techo en esta historia

GDD §3.6: *"Sube cuando el enemigo ejecuta habilidad 📜 [...]. Se frena con: habilidad de Líder anti-Trama, carta de jugador anti-Trama [...]."* GDD §12: *"Trama X [...] Enemigo sube, jugador baja."* Esto es exactamente lo mismo dato que ya existe en `AbilityCooldownDefinition.side` (H1.4: el lado dueño de la habilidad) — **no hace falta un campo de dirección nuevo en `AbilityEffectDefinition`**: `side === 'ENEMY'` → `scenarioPlot` sube; `side === 'LEADER'` → `scenarioPlot` baja. Esto también deja preparado, sin coste adicional, el caso "habilidad de Líder anti-Trama" citado explícitamente por el GDD, aunque el criterio de aceptación de H1.6 solo pida explícitamente el sentido Enemigo→sube.

**Piso en 0 (decisión de esta historia, no bloqueante — asunción razonable, no encontrada explícita en el GDD):** ninguna cita del GDD dice qué pasa si Trama bajara de 0; se aplica el mismo patrón ya usado en el motor para otros contadores que no deben ser negativos (`remainingCooldowns`, saturado en 0 vía `Math.max(0, ...)` desde H1.4). Se documenta aquí como una asunción explícita, no como una cita literal del GDD, para que quede visible si el Director quiere revisarla.

**Sin techo/umbral final en esta historia:** GDD §3.6 menciona un "Umbral final = derrota alternativa", pero el propio backlog asigna la condición de derrota (`Trama ≥ umbral final`) a **H1.18** ("Sistema básico de juego de combate"), y el valor concreto de ese umbral es dato de `ScenarioDefinition` (H1.11, contenido, todavía no existe). H1.6 **no** clampea `scenarioPlot` por arriba ni comprueba ninguna condición de derrota — es responsabilidad de H1.18 leer `scenarioPlot` del snapshot y compararlo contra el umbral de la definición del Escenario activo.

### 0.5 `leaderDamage`: contador acumulado, no vida restante — vida máxima queda fuera de esta historia (H1.18)

El criterio de aceptación usa literalmente el nombre `leaderDamage` (no `leaderHealth`), y el propio comentario de `snapshot.ts` (H1.4, escrito antes de esta historia) ya anticipaba ese nombre exacto. Se modela como contador que **empieza en 0 y solo sube** con daño de Ataque no absorbido (ver §0.1/§3.2) — no como "vida restante que baja desde un máximo". Esto es deliberado y evita una decisión de negocio que no le corresponde a esta historia: **la vida MÁXIMA del Líder (y del Enemigo) y la condición de derrota/victoria (`leaderDamage` acumulado vs. vida máxima; vida del Enemigo ≤ 0) son criterios de aceptación explícitos de H1.18**, que además depende de datos de catálogo (`LeaderDefinition`/`EnemyDefinition`, H1.8-H1.10) que no existen todavía. H1.6 no introduce `leaderMaxHealth` ni `enemyHealth` ni ninguna comprobación de derrota/victoria — sería invadir el alcance de H1.18 sin los datos que esa historia necesita para hacerlo bien.

**Consecuencia de alcance, explícita:** una habilidad de tipo `ATTACK` en `abilityEffects` **debe** pertenecer a un `abilityId` cuyo `AbilityCooldownDefinition.side` sea `'ENEMY'` — el constructor de `CombatEngine` lo valida y lanza si no (§3.1, `validateAbilityEffectsConfig`). Razón: el GDD (§3.7) dice *"el enemigo apunta siempre al Líder con sus habilidades de Ataque"* — es decir, en el modelo actual, **todo** daño de tipo Ataque va de Enemigo a Líder. El caso inverso (Líder ataca al Enemigo, mutando una futura `enemyHealth`) no está modelado en esta historia (no existe `enemyHealth` en el snapshot) y se deja explícitamente fuera de alcance, para H1.18 o una historia de contenido que la preceda. Esto no es una limitación oculta: el constructor falla rápido y con mensaje claro si algo intenta declarar una habilidad `ATTACK` de `side: 'LEADER'`, en vez de mutar silenciosamente el campo equivocado.

### 0.6 Nuevos `CombatCommand`/`CombatEvent`/`CombatCommandError`

- **`CombatCommand`: ninguno nuevo.** La mutación de `leaderDamage`/`leaderShield`/`scenarioPlot` es un efecto colateral de `ACTIVATE_ABILITY` (ya existente, H1.3) cuando `abilityEffects` describe esa habilidad — no hace falta ningún intent nuevo del jugador. (El comentario de `types/commands.ts`, escrito en H1.3, especulaba que H1.6 podría añadir `REDIRECT_DAMAGE` — no es así: `REDIRECT_DAMAGE` sigue perteneciendo íntegramente a H1.15, ver §0.1).
- **`CombatCommandError`: ninguno nuevo.** No hay ninguna validación adicional de `dispatch()` en runtime — los únicos fallos nuevos (`abilityId` de `abilityEffects` no registrado en `abilityCoreCosts`/`abilityCooldowns`; `ATTACK` con `side` `LEADER`; `initialLeaderShield` fuera de rango) son errores de **configuración**, lanzados como `Error` en el constructor, exactamente el mismo patrón que `validateAbilityCooldownsConfig` (H1.4) — no `CombatCommandError`.
- **`CombatEvent`: 2 nuevos** — `LEADER_DAMAGED` y `SCENARIO_PLOT_CHANGED` (§2.4). Nombrados en términos de dominio (igual que H1.3 renombró el `CORE_ROLLED` del sketch de `architecture_stack.md` a `NUCLEO_POOL_ROLLED`) en vez de reusar literalmente `DAMAGE_DEALT`/`PLOT_CHANGED` del sketch original — los nombres nuevos dejan explícito, sin leer el payload, que el objetivo de daño es siempre el Líder y que el contador de Trama es del Escenario (el punto central de esta historia).

### 0.7 Ninguna otra ambigüedad de negocio detectada

Fuera de las asunciones explícitas ya declaradas (piso 0 de `scenarioPlot`, §0.4; alcance mínimo de "bloquea daño" vía `leaderShield` en vez de Aliado, §0.1; `ATTACK` restringido a `side: 'ENEMY'`, §0.5), no se ha encontrado ninguna decisión de diseño de negocio que requiera escalarse al Director en esta historia — todas las demás quedan resueltas citando el GDD/decisions.md/specs previas con confianza.

### 3.2 Métodos privados nuevos: `applyAttackEffect` / `applyPlotEffect`

```ts
  /**
   * NUEVO H1.6. Resuelve un efecto ATTACK (GDD §3.4/§2.8/§3.7): calcula el daño bruto
   * vía `resolveAbilityUmbral` (H1.5) con el valor del Núcleo gastado, lo hace pasar
   * primero por `leaderShield` (absorción automática e incondicional, sin
   * `CombatCommand` de por medio — ver spec §0.1), y solo suma a `leaderDamage` el
   * exceso si la habilidad tiene `arrollar: true` (GDD §2.8: "sin Arrollar por
   * defecto [...] el exceso se pierde"). Muta `this.leaderShield`/`this.leaderDamage`.
   */
  private applyAttackEffect(
    command: Extract<CombatCommand, { type: 'ACTIVATE_ABILITY' }>,
    effectDef: Extract<AbilityEffectDefinition, { kind: 'ATTACK' }>,
    nucleo: NucleoInstance
  ): CombatEvent {
    const resolution = resolveAbilityUmbral(effectDef.formula, nucleo.value);
    const rawAmount = resolution.baseResolvedValue;

    const shieldBefore = this.leaderShield;
    const absorbedByShield = Math.min(shieldBefore, rawAmount);
    this.leaderShield = shieldBefore - absorbedByShield;

    const excess = rawAmount - absorbedByShield;
    const appliedDamage = effectDef.arrollar === true ? excess : 0;
    this.leaderDamage += appliedDamage;

    return {
      type: 'LEADER_DAMAGED',
      abilityId: command.abilityId,
      sourceId: command.sourceId,
      side: command.side,
      nucleoSpent: nucleo,
      rawAmount,
      absorbedByShield,
      appliedDamage,
      leaderShieldAfter: this.leaderShield,
      leaderDamageAfter: this.leaderDamage,
    };
  }

  /**
   * NUEVO H1.6. Resuelve un efecto PLOT (GDD §3.6/§12): magnitud fija
   * (`effectDef.amount`, NO alimentada por Núcleo — ver spec §0.3), dirección derivada
   * del `side` dueño de la habilidad (`ENEMY` → sube, `LEADER` → baja — spec §0.4).
   * Satura `scenarioPlot` en 0 por abajo. Nunca toca `leaderShield`/`leaderDamage`
   * (GDD §3.6: "el daño de Trama es inabsorbible" — es, además, conceptualmente otra
   * cosa que "daño").
   */
  private applyPlotEffect(
    command: Extract<CombatCommand, { type: 'ACTIVATE_ABILITY' }>,
    effectDef: Extract<AbilityEffectDefinition, { kind: 'PLOT' }>
  ): CombatEvent {
    const ownerDef = this.abilityCooldowns.get(command.abilityId) as AbilityCooldownDefinition;
    const direction: 'INCREASE' | 'DECREASE' = ownerDef.side === 'ENEMY' ? 'INCREASE' : 'DECREASE';
    const rawAmount = effectDef.amount;
    const appliedDelta = direction === 'INCREASE' ? rawAmount : -rawAmount;
    this.scenarioPlot = Math.max(0, this.scenarioPlot + appliedDelta);

    return {
      type: 'SCENARIO_PLOT_CHANGED',
      abilityId: command.abilityId,
      sourceId: command.sourceId,
      side: command.side,
      direction,
      rawAmount,
      appliedDelta,
      scenarioPlotAfter: this.scenarioPlot,
    };
  }
```

      ]),
      abilityEffects: effects([
        [ENEMY_PLOT, { kind: 'PLOT', amount: 5 }],
        [LEADER_ANTI_PLOT, { kind: 'PLOT', amount: 2 }],
      ]),
      poolSize: 6,
    });

    // LEADER tiene el turno inicial: sube primero la Trama vía una activación ENEMY no
    // es posible (NOT_YOUR_TURN) — se sube manualmente pasando primero por ENEMY.
    engine.dispatch({ type: 'END_TURN' }); // LEADER -> ENEMY
    const enemyNucleo = engine.getSnapshot().nucleoPool[0]!;
    engine.dispatch({ type: 'ACTIVATE_ABILITY', abilityId: ENEMY_PLOT, sourceId: 'enemy', side: 'ENEMY', nucleoInstanceId: enemyNucleo.id });
    expect(engine.getSnapshot().scenarioPlot).toBe(5);

    engine.dispatch({ type: 'END_TURN' }); // ENEMY -> LEADER
    const leaderNucleo = engine.getSnapshot().nucleoPool[0]!;
    engine.dispatch({ type: 'ACTIVATE_ABILITY', abilityId: LEADER_ANTI_PLOT, sourceId: 'leader', side: 'LEADER', nucleoInstanceId: leaderNucleo.id });
    expect(engine.getSnapshot().scenarioPlot).toBe(3); // 5 - 2
  });

  it('scenarioPlot satura en 0, nunca queda negativo', () => {
    const engine = new CombatEngine({
      randomSource: new SeededRandomSource(7),
      abilityCoreCosts: costs([LEADER_ANTI_PLOT]),
      abilityCooldowns: cooldowns([[LEADER_ANTI_PLOT, { side: 'LEADER', baseCooldown: 1 }]]),
      abilityEffects: effects([[LEADER_ANTI_PLOT, { kind: 'PLOT', amount: 2 }]]),
      poolSize: 6,
    });
    const nucleo = engine.getSnapshot().nucleoPool[0]!;
    // scenarioPlot arranca en 0; una habilidad LEADER lo intenta bajar en 2.
    engine.dispatch({ type: 'ACTIVATE_ABILITY', abilityId: LEADER_ANTI_PLOT, sourceId: 'leader', side: 'LEADER', nucleoInstanceId: nucleo.id });

  });
