# H1.19 — Test harness CLI para jugar combate aislado

> Spec técnica del Architect para la historia H1.19 (Épica E1, `.ai-studio/memory/backlog.md`
> líneas 272-282, última historia del Hito 1). Consume `packages/domain/combat/src/combat-engine.ts`
> (H1.3-H1.18, leído línea a línea, incluido el `handlePlayCard`/`runAutomaticEnemyTurn` reales ya
> implementados), `packages/domain/combat/src/types/*.ts` completos, `packages/domain/catalog/src/
> catalog-loader.ts` + `types/*.ts`, el contenido real 2×2×2 (`packages/data/**`, H1.9-H1.12),
> `packages/data/load-content.test.ts` (patrón de lectura de JSON vía `fs` sin `JSON.parse` dentro
> de `domain/catalog`), `eslint.config.mjs` (boundaries) y el QA de H1.18 (nota explícita sobre la
> falta de un adaptador `CardDefinition` → `PlayableCardDefinition` en producción).

Estado del repo al momento de escribir esta spec (H1.1-H1.18 cerradas):

- **El motor de combate está completo e integrado.** `CombatEngine.dispatch()` procesa los 8
  `CombatCommand` (`ACTIVATE_ABILITY`, `END_TURN`, `PLAY_CONTRATIEMPO`, `PLAY_ALLY`,
  `SET_DAMAGE_REDIRECT`, `SUMMON_MINION`, `RESOLVE_MINION_ACTION`, `PLAY_CARD`). `handleEndTurn`
  ya dispara automáticamente todo el turno del Enemigo (`runAutomaticEnemyTurn`, H1.18 §0.5) si
  `CombatEngineConfig.enemyAbilityAiProfiles`/`dramaturgiaDeck` están poblados. `CombatStateSnapshot`
  refleja `status`/`defeatReason` desde H1.18.
- **`runAutomaticEnemyTurn` (leído línea a línea) ya invoca `handleResolveMinionAction()`
  internamente** cuando `minionsInPlay.length > 0` (combat-engine.ts, líneas ~1868-1871) — el turno
  del Enemigo, IA incluida, es 100% automático una vez `enemyAiEnabled`. **Ningún contenido real
  de H1.9-H1.12 define ningún `MinionDefinition`** (grep negativo sobre `packages/data/enemies/*.json`
  y `scenarios/*.json`: los `effectDescription` que mencionan secuaces son texto no ejecutable,
  documentado como deuda explícita desde H1.10 §0.3) — `SUMMON_MINION` nunca se dispara con el
  contenido de este hito. Ver §0.3 para la decisión de alcance que esto habilita.
- **No existe ningún ensamblador de `CombatEngineConfig` a partir de un `Catalog` real.** Todo test
  de `domain/combat` (H1.3-H1.18) construye sus mapas (`abilityCoreCosts`, `abilityCooldowns`,
  `playableCards`, etc.) a mano, directamente en TypeScript, nunca a partir de `LeaderDefinition`/
  `EnemyDefinition`/`ScenarioDefinition`/`CardDefinition` reales. El QA de H1.18 lo señaló
  explícitamente: *"no existe todavía en producción un adaptador `CardDefinition` →
  `PlayableCardDefinition`... tuve que construirlo a mano en el script de prueba... vale la pena que
  quede como historia explícita antes de integrar UI"*. **Esta historia es esa historia explícita**
  — ver §0.2, decisión de que el adaptador se construye AQUÍ, en `domain/combat` (no en el CLI), por
  ser la única forma de que H2 (React/Phaser) lo reutilice sin duplicarlo.
- **`packages/data/leaders/*.json` ya incluyen `maxHealth`** (`soldado-base.json`/`mago-base.json`,
  añadido en H1.18) — no hace falta tocar contenido en esta historia.
- **No existe ningún workspace de "aplicación"/CLI todavía.** `package.json` raíz solo referencia
  `packages/domain/*` y `packages/data`; no hay `ts-node`/`tsx` en `devDependencies`; el único modo
  de ejecutar TypeScript hoy es `tsc -b` seguido de `node` sobre el JS compilado, o `vitest` para
  tests.

## 0. Decisiones de arquitectura que cierra esta spec

### 0.1 Ubicación: nuevo workspace `packages/cli`, no un script dentro de `packages/data`

**Pregunta que cierra esta decisión:** ¿el harness vive como script añadido a `packages/data`
(que ya depende de `domain-catalog`/`domain-combat`, ver su `package.json`), o como workspace
nuevo?

**Decisión: nuevo workspace `packages/cli`.** Razones:

1. `packages/data` es, por diseño y por `eslint.config.mjs` (`{ type: 'data', pattern:
   'packages/data/**' }`, `{ from: 'data', allow: [] }`), un paquete de **contenido puro** (JSON +
   el test que lo valida contra `CatalogLoader`) — no tiene ninguna otra responsabilidad de
   runtime. Añadirle un script interactivo con su propio bucle de E/S, parser de comandos y
   renderer de texto le da una segunda responsabilidad ajena a "ser el contenido 2×2×2", justo lo
   que la separación de paquetes del monorepo (H1.1) existe para evitar.
2. El propio `eslint.config.mjs` ya deja "entradas futuras" documentadas para paquetes de
   aplicación (`combat-scene`, `ui-shared`, `shell`, todas fuera de `domain/*`/`data`) — un CLI de
   desarrollo encaja en ese mismo patrón de "paquete de aplicación que consume domain/*", aunque
   sea de un tipo más simple (sin Phaser/React). Se añade como una entrada más de ese mismo estilo,
   no como una excepción.
3. Mantiene `packages/data` con cero `dependencies` de runtime más allá de las 3 ya declaradas
   (`domain-catalog`, `domain-shared`, `domain-combat`, usadas solo por su test) — el CLI necesita
   además leer archivos de `packages/data/**` en runtime (no solo en tests), lo cual es una
   responsabilidad de aplicación, no de contenido.

`packages/cli` se añade a `workspaces` en el `package.json` raíz:

```json
"workspaces": [
  "packages/domain/*",
  "packages/data",
  "packages/cli"
]
```

Y a `tsconfig.json` raíz (`references`) y a `eslint.config.mjs` (`boundaries/elements` +
`element-types`, ver §1).

### 0.2 Adaptador Catálogo → `CombatEngineConfig`: vive en `domain/combat`, no en el CLI

**Pregunta que cierra esta decisión (la marcada explícitamente por QA de H1.18):** ¿el adaptador
`CardDefinition`/`LeaderDefinition`/`EnemyDefinition`/`ScenarioDefinition` → `CombatEngineConfig`
(incluyendo, pero sin limitarse a, `PlayableCardDefinition`) se construye dentro de `packages/cli`
(uso exclusivo del harness), o como una función exportada de `domain/combat`?

**Decisión: se construye en `domain/combat`, como nuevo módulo `packages/domain/combat/src/
catalog-adapter.ts`, exportado desde `index.ts` junto al resto de la API pública.** Razones:

1. **Reutilización real, no hipotética.** `docs/architecture_stack.md` ya fija que H2 (Phaser +
   React) construye `CombatEngine` a partir del mismo `Catalog` (elección de Líder/mazo, sorteo
   Enemigo×Escenario de la run). Sin este adaptador en `domain/combat`, H2 tendría que
   reimplementar exactamente esta lógica (o copiarla del CLI, un paquete de aplicación que no
   debería ser una dependencia de otro paquete de aplicación) — poner el adaptador en `domain/
   combat` es la única ubicación que ambos consumidores (`cli` hoy, `shell`/`combat-scene` en H2)
   pueden importar sin crear una dependencia cli→shell o shell→cli.
2. `domain/combat` **ya tiene permiso de boundaries para importar `domain/catalog`** (`{ from:
   'domain-combat', allow: ['domain-shared', 'domain-catalog'] }`, confirmado y ya usado desde
   H1.17 para `PhaseDefinition`/`PhaseChangeCondition` sin mirror) — no hace falta ninguna regla
   nueva de boundaries para este módulo.
3. Cierra la deuda exactamente como la enunció QA: "un adaptador... antes de integrar UI" — un
   módulo de `domain/combat` es infraestructura de producción, no una utilidad de desarrollo
   descartable; construirlo aquí (en vez de en `packages/cli`) es lo que lo convierte en esa pieza
   reutilizable de producción en vez de, otra vez, "un script de prueba a mano".

`packages/cli` **consume** `buildCombatEngineConfig` (§2) — no reimplementa ninguna parte de él.

### 0.3 Alcance de comandos soportados: 6 de los 8 `CombatCommand` — `SUMMON_MINION`/
`RESOLVE_MINION_ACTION` explícitamente fuera de alcance

**Pregunta que cierra esta decisión:** ¿el CLI expone los 8 tipos de `CombatCommand`, o un
subconjunto?

**Decisión: el CLI despacha manualmente 6 comandos — `ACTIVATE_ABILITY`, `END_TURN`, `PLAY_CARD`,
`PLAY_ALLY`, `PLAY_CONTRATIEMPO`, `SET_DAMAGE_REDIRECT` — y nunca despacha `SUMMON_MINION` ni
`RESOLVE_MINION_ACTION` manualmente.** Razones, ninguna es "por simplicidad", todas ancladas en
cómo funciona el motor real:

1. **El CLI controla exclusivamente al Líder.** El Enemigo se juega siempre con
   `enemyAiEnabled: true` (§2, `buildCombatEngineConfig` puebla `enemyAbilityAiProfiles`/
   `dramaturgiaDeck` siempre) — un `END_TURN` del jugador humano dispara automáticamente todo el
   turno de Enemigo (`runAutomaticEnemyTurn`, ya implementado, H1.18 §0.5.1), incluida la resolución
   de Secuaces si los hubiera (`handleResolveMinionAction()` invocado internamente, confirmado en
   combat-engine.ts líneas ~1868-1871). El jugador **nunca** necesita dispatchar
   `RESOLVE_MINION_ACTION` a mano — el motor ya lo hace por él, siempre, en el mismo camino que
   usan los tests de H1.16-H1.18.
2. **`SUMMON_MINION` es exclusivo de `turnOwner === 'ENEMY'`** (`handleSummonMinion`, ver
   `types/commands.ts` — "GDD §3.8... equivalente a lo que hoy hace, sin ejecutar, el
   `effectDescription` de una carta de Dramaturgia"). Ningún camino automático de IA lo invoca hoy
   (grep confirmado: `SUMMON_MINION`/`handleSummonMinion` no aparecen en `runAutomaticEnemyTurn`) —
   es, con el contenido real, un comando sin ningún emisor posible: **ningún `MinionDefinition`
   existe en `packages/data/**`** (§0, estado del repo), así que aunque el CLI lo expusiera no
   habría ningún `minionDefinitionId` válido que un jugador pudiera escribir. Exponerlo sería una
   superficie de comando muerta, documentada explícitamente como deuda para cuando exista contenido
   de Secuaz real (candidato natural para una historia de H2 que también decida cómo la Dramaturgia
   del Enemigo invoca Secuaces en runtime).
3. Esto dejа el CLI con exactamente los comandos que un humano jugando el rol del Líder necesita —
   consistente con el propio backlog, que solo pide explícitamente `play card X` / `activate
   ability Y` / `end turn` como ejemplos de la gramática mínima.

### 0.4 Runner: `tsx` como nueva devDependency raíz, ejecución directa de TypeScript sin build previo

**Pregunta que cierra esta decisión:** ¿`npm run play-combat` ejecuta sobre JS ya compilado (`tsc -b`
+ `node dist/...`), o directamente sobre `.ts` vía un runner?

**Decisión: se añade `tsx` a `devDependencies` del `package.json` raíz, y `play-combat` invoca
`tsx packages/cli/src/main.ts` directamente.** Razones:

1. El backlog pide explícitamente "un script simple" — forzar a quien lo ejecuta a acordarse de
   `npm run build` antes de cada partida (o de que el watch de `tsc -b --watch` esté corriendo)
   añade fricción contraria al espíritu de la historia ("validación manual del motor", iteración
   rápida).
2. `tsx` no introduce ningún paso de bundling ni afecta al `tsconfig`/`tsc -b` existente — sigue
   siendo el único mecanismo de *build* de producción para el resto de paquetes; `tsx` es
   estrictamente una herramienta de desarrollo, exactamente como `vitest` ya lo es para tests (que
   tampoco pasa por `tsc -b`).
3. Alternativa descartada: `ts-node` con loader ESM — más configuración (el repo entero es
   `"type": "module"`, `ts-node`/ESM tiene fricción histórica conocida con resolución de módulos)
   frente a `tsx`, que soporta ESM+TS de fábrica sin configuración adicional. No hay ninguna otra
   razón de peso para preferir `ts-node`.

`package.json` raíz añade:

```json
"scripts": {
  "play-combat": "tsx packages/cli/src/main.ts"
},
"devDependencies": {
  "tsx": "^4.19.0"
}
```

(versión ilustrativa — Programmer fija la última minor estable en el momento de implementar).

`packages/cli/package.json` conserva `"build": "tsc -b"` por paridad con el resto de paquetes
(typecheck en CI, `tsc -b` raíz) aunque `play-combat` no dependa de ese build — ver §1.

### 0.5 Interfaz de texto: índices posicionales para Núcleos/Aliados, nunca los ids internos opacos

**Pregunta que cierra esta decisión:** ¿el jugador escribe `nucleoInstanceId`/`CardInstanceId`
reales (`nucleo-3`, `ally-1`...) en el comando, o una referencia más cómoda?

**Decisión: el renderer imprime cada Núcleo del pool y cada Aliado en mesa con un índice
posicional `[0]`, `[1]`, ... en cada snapshot; la gramática de comandos referencia esos índices
(nunca el id interno), y el parser los resuelve contra el **último snapshot conocido** en el
momento de construir el `CombatCommand`.** Razón: `nucleoInstanceId`/`CardInstanceId` son ids
generados por el motor (`nucleo-0`, `ally-0`, contadores internos, ver `combat-engine.ts`
`nextNucleoId`/`nextCardInstanceId`) — no tienen ningún significado mnemotécnico para un humano
tecleando comandos en vivo; un índice que se re-imprime en cada turno es lo único razonable de
memorizar/leer entre un `print` y el siguiente comando. Ver §4 para la gramática completa y §5
para el layout exacto que expone estos índices.

## 2. `packages/domain/combat/src/catalog-adapter.ts` (NUEVO) — contrato completo

### 2.1 Firma pública

```ts
import type { RandomSource } from '@collector/domain-shared';
import type { Catalog, LeaderDefinition, EnemyDefinition, ScenarioDefinition } from '@collector/domain-catalog';
import type { CombatEngineConfig } from './types/config';
import type { CombatSide } from './types/turn';

export interface BuildCombatEngineConfigParams {
  readonly catalog: Catalog;
  readonly leader: LeaderDefinition;
  readonly enemy: EnemyDefinition;
  readonly scenario: ScenarioDefinition;
  readonly randomSource: RandomSource;
  /** Default 'LEADER' — mismo default que CombatEngineConfig.initialTurnOwner. */
  readonly initialTurnOwner?: CombatSide;
}

/**
 * Ensambla un `CombatEngineConfig` completo a partir de un `Catalog` ya cargado
 * (`CatalogLoader.load()`) y 3 entidades concretas ya resueltas (Líder, Enemigo,
 * Escenario) — mismo nivel de "ya resuelto" que exige `CombatEngineConfig` para
 * `leaderMaxHealth`/`scenarioPlotDefeatThreshold` (H1.18 §0.3/§0.4). Activa SIEMPRE el
 * turno de IA automático (`enemyAbilityAiProfiles`/`dramaturgiaDeck` nunca vacíos) — no
 * hay ningún caso de uso de este adaptador que quiera un Enemigo sin IA.
 *
 * No valida referencias cruzadas (cardId inexistente, etc.) — esa validación YA ocurrió
 * en `CatalogLoader.load()` (H1.8 §4) antes de que `catalog` llegue aquí; un `Catalog`
 * bien formado por contrato nunca puede producir una referencia rota en este adaptador.
 */
export function buildCombatEngineConfig(params: BuildCombatEngineConfigParams): CombatEngineConfig;
```

### 2.2 Sub-ensambladores internos (no exportados, documentados por su contrato — Programmer los
implementa como funciones privadas del módulo o inline, a discreción, siempre que el resultado de
`buildCombatEngineConfig` sea el descrito)

#### a) `abilityCoreCosts` / `abilityCooldowns` / `abilityEffects`

Fuente: `leader.baseAbilities` (4 elementos, `side: 'LEADER'`) + `enemy.abilities` (`side:
'ENEMY'`). Para cada `AbilityDefinition`/`EnemyAbilityDefinition`:

```
abilityCoreCosts.set(ability.id, ability.coreCost)
abilityCooldowns.set(ability.id, { side, baseCooldown: ability.baseCooldown })
if (ability.effect) abilityEffects.set(ability.id, ability.effect)
```

`ability.effect` (`CatalogAbilityEffect`, `domain/catalog/types/ability.ts`) es **estructuralmente
idéntico** a `AbilityEffectDefinition` (`domain/combat/types/ability-effect.ts`): mismas 2
variantes (`ATTACK` con `formula: { baseFormula, bonusFormula? }` + `arrollar?`; `PLOT` con
`amount`), mismos 3 `kind` de `UmbralFormula`/`CatalogUmbralFormula` (`VALUE`/`ADD`/`MULTIPLY`).
Se asigna directamente (cast estructural, sin función de conversión de campos) — documentar en el
código que esta identidad estructural es intencional y ya fue señalada por H1.18 §0.1.1 (el motor
reutiliza `AbilityUmbralDefinition`/`resolveAbilityUmbral` sin ningún tipo nuevo).

Con el contenido real (`soldado-base.json`/`mago-base.json`/`bestia-base.json`/
`espectro-base.json`), esto produce 4+4 = 8 entradas en `abilityCoreCosts`/`abilityCooldowns` (4
del Líder elegido + 4 del Enemigo elegido) por combate montado.

`abilityCombo` se deja como `Set()` vacío (default) — ninguna `AbilityDefinition` del contenido
real tiene un concepto de Combo propio (la keyword `COMBO` vive únicamente en `CardDefinition`,
`card-soldado-base-10`, y **`handlePlayCard` no implementa Combo para cartas** — confirmado por
lectura del motor real, es deuda explícita ya asumida por H1.18, fuera de alcance de este
adaptador).

#### b) `playableCards` / `allyCards` / `contratiempoCards` — un único recorrido de `leader.cardPoolIds`

```
for (const cardId of leader.cardPoolIds):
  const card = catalog.cards.get(cardId)   // garantizado presente por CatalogLoader (§4 H1.8)
  switch (card.type):
    case 'EVENTO':
    case 'EQUIPO':
      playableCards.set(cardId, buildPlayableCardDefinition(card))
    case 'ALIADO':
      allyCards.set(cardId, buildAllyCardDefinition(card))
    case 'CONTRATIEMPO':
      contratiempoCards.set(cardId, buildContratiempoCardDefinition(card))
```

Cada `cardId` cae en exactamente UNO de los 3 mapas — por construcción (un `switch` exhaustivo
sobre `CardType`), nunca hace falta validar exclusividad después.

**Nota de alcance:** solo se recorre `leader.cardPoolIds` (10 cartas del Líder elegido) — las 6
cartas "comunes" (`card-common-01`..`06`, H1.12) **no** están referenciadas por ningún
`cardPoolIds` (confirmado por `load-content.test.ts`, "las 6 cartas comunes NO están referenciadas
por ningún cardPoolIds de Líder") y por tanto nunca se juegan con `PLAY_CARD`/`PLAY_ALLY`/
`PLAY_CONTRATIEMPO` en este harness — consistente con el propio criterio de H1.18 §0.1 punto 3
("sin concepto de mano ni de mazo... jugar una carta de `cardPoolIds` es válido").

**`buildPlayableCardDefinition(card: CardDefinition): PlayableCardDefinition`:**

```
energyCost = card.cost.energy
effect = resolveKeywordEffect(card.keywords):
  buscar la primera keyword de esta lista (a lo sumo una debe estar presente — si el contenido
  violara esto, lanzar Error "carta <id> tiene más de un efecto numérico de PLAY_CARD"):
    ATAQUE           → { kind: 'ATTACK_ENEMY', formula: { baseFormula: { kind: 'VALUE' } } }
    ATAQUE_MAS_X     → { kind: 'ATTACK_ENEMY', formula: { baseFormula: { kind: 'ADD', amount } } }
    ATAQUE_POR_X     → { kind: 'ATTACK_ENEMY', formula: { baseFormula: { kind: 'MULTIPLY', amount } } }
    TRAMA_X          → { kind: 'PLOT', amount }
    DEFENSA_X        → { kind: 'SHIELD', amount }
  si ninguna de estas 5 keywords está presente (ej. carta solo con NEUTRO/UMBRAL/ARROLLAR/COMBO
  como modificador sin keyword numérica propia): effect = undefined
return { energyCost, ...(effect ? { effect } : {}) }
```

`UMBRAL`/`ARROLLAR`/`COMBO`/`NEUTRO` presentes junto a una keyword numérica **no** generan ningún
campo adicional — son, exactamente como documenta H1.18 §0.1.1, "dato en reposo" ya asumido por el
motor (Umbral de carta sin `bonusFormula`; Arrollar de carta sin objetivo válido; Combo de carta no
ejecutado). El adaptador no necesita ninguna lógica extra para ellas.

**`buildAllyCardDefinition(card: CardDefinition): AllyCardDefinition`:**

```
energyCost = card.cost.energy
life = keywords.find(k => k.keyword === 'VIDA_X').amount    // VIDA_X siempre presente en ALIADO (H1.12)
isBerserker = keywords.some(k => k.keyword === 'BERSERKER')
// abilityIds: ausente — ningún ALIADO del contenido real referencia una AbilityId propia
return { energyCost, life, isBerserker }
```

(La keyword `DEFENSOR` en una carta ALIADO, ej. `card-soldado-base-06`, no tiene ningún campo
correspondiente en `AllyCardDefinition` — no modelado por el motor para Aliados del jugador, mismo
criterio de "dato en reposo"; `DEFENSOR` en Secuaces, H1.16, es un concepto distinto con su propio
campo `isDefensor`.)

**`buildContratiempoCardDefinition(card: CardDefinition): ContratiempoCardDefinition`:**

```
energyCost = card.cost.energy
undoScope = keywords.some(k => k.keyword === 'DESHACER_TURNO') ? 'FULL_TURN' : 'DAMAGE_ONLY'
  // exactamente una de DESHACER_DANO/DESHACER_TURNO está presente (invariante ya validada
  // por parseCardDefinition, H1.14 §0.5) — no hace falta validarlo de nuevo aquí.
return { energyCost, undoScope }
```

#### c) `enemyAbilityAiProfiles` / `dramaturgiaDeck`

```
enemyAbilityAiProfiles = new Map(enemy.abilities.map(a => [a.id, a.aiProfile]))
dramaturgiaDeck = [...enemy.dramaturgiaDeck, ...scenario.dramaturgiaDeck].map(c => c.icon)
```

(Mismo criterio explícito de H1.18 §0.5: "`EnemyDefinition.dramaturgiaDeck.concat(

#### d) `enemyPhases` / `scenarioPhases` / `enemyMaxHealth` / `leaderMaxHealth` /
`scenarioPlotDefeatThreshold`

```
enemyPhases = enemy.phases                 // reutilizado tal cual, mismo tipo (H1.17 §0.1)
scenarioPhases = scenario.phases           // ídem
enemyMaxHealth = enemy.maxHealth
leaderMaxHealth = leader.maxHealth
scenarioPlotDefeatThreshold = Math.max(...scenario.plotThresholds.map(t => t.atLeast))
  // mismo criterio exacto que H1.18 §0.4
```

initialTurnOwner: params.initialTurnOwner ?? 'LEADER'
minionDefinitions: new Map()   // §0.3 — ningún contenido real define Secuaces todavía

### 2.3 `packages/domain/combat/src/catalog-adapter.test.ts` (NUEVO) — smoke test obligatorio

Reutiliza el mismo patrón de lectura de `packages/data/**` que `packages/data/load-content.test.ts`

## 4. Gramática de comandos

Una línea de stdin = un comando. Tokenización simple por espacios (`line.trim().split(/\s+/)`);
sin comillas ni escapes — ningún id de catálogo contiene espacios. Comparación case-insensitive
para las palabras clave del comando (`play`/`activate`/`ability`/`ability`/`end`/`turn`/etc.),
case-sensitive para los propios ids (`AbilityId`/`CardId` son tal cual aparecen en el JSON).

| Comando textual | `CombatCommand` resultante | Notas |
|---|---|---|
| `activate ability <abilityId> <nucleoIndex>` | `{ type: 'ACTIVATE_ABILITY', abilityId, sourceId: 'leader', side: 'LEADER', nucleoInstanceId }` | `nucleoIndex` = índice 0-based impreso en la sección "Núcleos" del último snapshot (§5.2); se resuelve a `nucleoPool[nucleoIndex].id`. Fuera de rango → error de parseo, no se dispatcha nada. |
| `end turn` | `{ type: 'END_TURN' }` | Dispara automáticamente todo el turno de Enemigo (§0.3/§0.5.1 H1.18) antes de devolver el control. |
| `play card <cardId>` | `{ type: 'PLAY_CARD', cardId, sourceId: 'leader' }` | Sin `nucleoIndex` — válido si la carta no tiene efecto `ATTACK_ENEMY` (o si el motor decide que no hace falta; si hiciera falta y no se dio, el motor devuelve `PLAY_CARD_NUCLEO_REQUIRED`, ver §7.2). |
| `play card <cardId> <nucleoIndex>` | `{ type: 'PLAY_CARD', cardId, sourceId: 'leader', nucleoInstanceId }` | Forma con Núcleo — usar cuando la carta es de Ataque (`ATAQUE`/`ATAQUE_MAS_X`/`ATAQUE_POR_X`). |
| `play ally <cardId>` | `{ type: 'PLAY_ALLY', cardId, sourceId: 'leader' }` | |
| `play contratiempo <cardId>` | `{ type: 'PLAY_CONTRATIEMPO', cardId, sourceId: 'leader' }` | |
| `set redirect <allyIndex>` | `{ type: 'SET_DAMAGE_REDIRECT', targetAllyInstanceId }` | `allyIndex` = índice 0-based en la sección "Aliados en mesa" del último snapshot (§5.2); se resuelve a `alliesInPlay[allyIndex].instanceId`. |
| `set redirect none` | `{ type: 'SET_DAMAGE_REDIRECT', targetAllyInstanceId: null }` | |
| `help` | *(no se dispatcha)* | Imprime la tabla de comandos disponibles. |
| `status` | *(no se dispatcha)* | Reimprime el último snapshot sin avanzar el combate. |
| `quit` / `exit` | *(no se dispatcha)* | Cierra el proceso limpiamente (`process.exit(0)`), sin exigir que el combate haya terminado. |

`sourceId: 'leader'` es una constante fija para los 5 comandos exclusivos del Líder — el CLI nunca
necesita distinguir entre "fuentes" del lado Líder (sin Aliados con habilidades propias en el
contenido real, §2.2.b). `SET_DAMAGE_REDIRECT` no lleva `sourceId` (mismo contrato que el comando
real, `types/commands.ts`).

### 4.1 `command-parser.ts` — contrato

```ts
import type { CombatCommand } from '@collector/domain-combat';
import type { CombatStateSnapshot } from '@collector/domain-combat';

export type ParsedLine =
  | { readonly kind: 'COMMAND'; readonly command: CombatCommand }
  | { readonly kind: 'LOCAL'; readonly action: 'HELP' | 'STATUS' | 'QUIT' }
  | { readonly kind: 'PARSE_ERROR'; readonly message: string };

/**
 * Traduce una línea cruda de stdin a un `CombatCommand` tipado, o a una acción local del
 * CLI (help/status/quit), o a un error de parseo — nunca lanza. `snapshot` es el ÚLTIMO
 * `CombatStateSnapshot` conocido (§0.5) — se usa exclusivamente para resolver
 * `nucleoIndex`/`allyIndex` a los ids reales que el motor espera; si `snapshot` no tiene
 * suficientes entradas para el índice pedido, se devuelve `PARSE_ERROR` con un mensaje
 * legible (nunca un índice fuera de rango llega a `CombatCommand`).
 */
export function parseLine(line: string, snapshot: CombatStateSnapshot): ParsedLine;
```

## 5. Formato de salida en consola

### 5.1 Cuándo se imprime

1. **Una vez**, inmediatamente tras construir el `CombatEngine` (antes del primer prompt) — el
   snapshot inicial completo (§5.2).
2. Tras cada `COMMAND` exitoso: 1 línea de resumen por cada `CombatEvent` devuelto por `dispatch()`
   (§5.4) seguida del snapshot completo (§5.2).
3. Tras cada `COMMAND` rechazado por el motor (`Result.ok === false`): 1 línea de error (§5.5), sin
   reimprimir el snapshot completo (no cambió).
4. Tras `PARSE_ERROR`: 1 línea de error, sin tocar el motor ni reimprimir nada más.
5. Tras `LOCAL: 'STATUS'`: el snapshot completo (§5.2), sin línea de eventos.
6. Al alcanzar `status !== 'IN_PROGRESS'` (tras el paso 2 de arriba): además del snapshot ya
   impreso, un banner final (§5.6) y cierre del proceso (`process.exitCode = 0`).

### 5.2 Layout exacto del snapshot completo

```
======================================================================
Turno 7 — LÍDER

Líder    : Daño 12/30 | Escudo 2/5 | Energía 3/5 | Nivel 2 (level-ups 1/2)
Enemigo  : Daño 25/60 | Fase 1/2
Escenario: Trama 4 (derrota en 10) | Fase 2/2

Núcleos disponibles:
  [0] AGRESION valor=3   [1] CONTROL valor=1   [2] DEFENSA valor=4 (Umbral)
  [3] CAOS valor=2       [4] RECURSO valor=0 (debuff)

Cooldowns del Líder:
  ability-soldado-base-guardia-firme     (Guardia Firme)      CD 0/1  LISTA
  ability-soldado-base-avance-agresivo   (Avance Agresivo)    CD 1/2
  ability-soldado-base-linea-defensiva   (Línea Defensiva)    CD 3/3
  ability-soldado-base-grito-de-guerra   (Grito de Guerra)    CD 0/4  LISTA

Acciones: 1/2 usadas (combo no concedido este turno)

Aliados en mesa:
  [0] card-soldado-base-06 (Recluta Leal)    vida 3/5
Redirección de daño activa: ninguna

Contratiempo disponible (turno de Enemigo anterior): 2 entradas revertibles

Estado: EN CURSO
----------------------------------------------------------------------
>
```

Reglas de construcción, campo por campo (mapeo directo desde `CombatStateSnapshot`, sin ningún
cálculo que el motor no exponga ya):

- Cabecera: `Turno ${turn.turnNumber} — ${turn.turnOwner === 'LEADER' ? 'LÍDER' : 'ENEMIGO'}`.
- `Líder`: `Daño ${leaderDamage}/${leaderMaxHealth}` (el CLI recibe `leaderMaxHealth` aparte, del
  `CombatEngineConfig` ya construido — **no** viene en el snapshot, ver H1.18 §2.3, se guarda en
  una variable local al arrancar) | `Escudo ${leaderShield}/${LEADER_SHIELD_MAX}` (constante
  exportada de `domain-combat`) | `Energía ${leaderEnergy}/${LEADER_ENERGY_MAX}` (ídem) | `Nivel
  ${leaderState.level} (level-ups ${leaderState.levelUpsSpent}/${LEADER_LEVEL_UPS_MAX})`.
- `Enemigo`: `Daño ${enemyDamage}/${enemyMaxHealth}` (mismo criterio que `leaderMaxHealth`: guardado
  localmente, no viene en el snapshot) | `Fase ${enemyPhase.phaseNumber}/${enemyPhase.totalPhases}`
  (si `totalPhases === 0`, imprimir `Fase — (sin tracking)` en su lugar).
- `Escenario`: `Trama ${scenarioPlot} (derrota en ${scenarioPlotDefeatThreshold})` (guardado
  localmente, mismo criterio) | `Fase ${scenarioPhase.phaseNumber}/${scenarioPhase.totalPhases}`.
- `Núcleos disponibles`: una entrada por elemento de `nucleoPool`, en su orden de array (= el
  índice que el jugador usará), formato `[i] ${color} valor=${value}` + sufijo `(Umbral)` si
  `value >= UMBRAL_BONUS_THRESHOLD` (constante exportada) + sufijo `(debuff)` si `value === 0`.
  Se envuelve en líneas de a lo sumo 3 entradas por línea (estético, no funcional — a discreción
  de Programmer mantener 1 por línea si es más simple).
- `Cooldowns del Líder`/`Cooldowns del Enemigo`: dos secciones separadas (se muestra solo la del
  `turnOwner` actual, o ambas siempre — decisión de Programmer; se recomienda mostrar SIEMPRE
  ambas para que el jugador vea venir la próxima habilidad del Enemigo, aunque no pueda actuar
  sobre ella). Una línea por entrada de `cooldowns` filtrada por `side`, formato `${abilityId}
  (${nameLookup.abilityName(abilityId)})  CD ${remaining}/${baseCooldown}` + sufijo `LISTA` si
  `remaining === 0`.
- `Acciones`: `${actions.actionsTaken}/${actions.actionsAllowed} usadas (combo
  ${actions.comboBonusGranted ? 'concedido' : 'no concedido'} este turno)`.
- `Aliados en mesa`: una línea por elemento de `alliesInPlay` (incluye muertos, `life === 0` — se
  marca `(muerto)` al final de la línea en ese caso), formato `[i] ${cardId}
  (${nameLookup.cardName(cardId)})    vida ${life}/${maxLife}`. Si `alliesInPlay` está vacío:
  `Aliados en mesa: ninguno`.
- `Redirección de daño activa`: `ninguna` si `activeDamageRedirectTargetId === null`; si no,
  `[i] ${cardId}` del Aliado correspondiente (buscar su índice dentro de `alliesInPlay`).
- `Contratiempo disponible`: `${undoableLastEnemyTurn.length} entradas revertibles` si > 0; si es
  0: `ninguno (sin turno de Enemigo revertible ahora mismo)`.
- `Estado`: `EN CURSO` si `status === 'IN_PROGRESS'`; en cualquier otro caso este bloque nunca se
  vuelve a imprimir como snapshot intermedio — el combate ya cerró (ver §5.6).

### 5.3 `renderer.ts` — contrato

```ts
import type { CombatStateSnapshot } from '@collector/domain-combat';
import type { NameLookup } from './name-lookup';

export interface RenderContext {
  readonly nameLookup: NameLookup;
  readonly leaderMaxHealth: number;
  readonly enemyMaxHealth: number;
  readonly scenarioPlotDefeatThreshold: number;
}

/** Devuelve el bloque completo de §5.2 como un único string multi-línea, sin el prompt
 *  final ("> ") — eso lo añade repl.ts al pedir la siguiente línea. */
export function renderSnapshot(snapshot: CombatStateSnapshot, ctx: RenderContext): string;
```

### 5.4 Resumen de eventos (impreso ANTES del snapshot, tras cada `COMMAND` exitoso)

Una línea por `CombatEvent`, formato `» ${humanReadableEventLine(event)}` — no se exige un mapeo
1:1 exhaustivo con los ~20 tipos de `CombatEvent`; basta con un `switch` que cubra los tipos que
el CLI puede producir realmente (todo `CombatEvent` excepto los exclusivos de `SUMMON_MINION`/
`RESOLVE_MINION_ACTION` cuando no hay Secuaces, que en la práctica no aparecen — ver §0.3) y un
`default` genérico `» ${event.type}` para cualquier tipo no cubierto explícitamente, de forma que
el CLI nunca "trague" un evento en silencio aunque el switch no esté exhaustivamente enumerado.
Ejemplos ilustrativos (no exhaustivo, Programmer decide la redacción exacta de cada línea):

```
» ABILITY_ACTIVATED: Guardia Firme (Núcleo AGRESION=3)
» ENEMY_DAMAGED: Estocada inflige 5 de daño al Enemigo (25/60)
» TURN_ENDED: LÍDER → ENEMIGO
» DRAMATURGIA_CARD_DRAWN: ATTACK
» ABILITY_ACTIVATED: Zarpazo (Enemigo)
» LEADER_DAMAGED: 6 de daño (2 absorbido por Escudo, 4 aplicado) — Líder 12/30
» COMBAT_ENDED: DEFEAT (LEADER_HEALTH)
```

### 5.5 Formato de error de comando rechazado por el motor

```
✗ Comando rechazado: NO_ACTIONS_REMAINING (side=LEADER, actionsTaken=2, actionsAllowed=2)
```

Formato genérico: `✗ Comando rechazado: ${error.code} (${Object.entries(error).filter(([k]) => k
!== 'code').map(([k, v]) => `${k}=${v}`).join(', ')})` — vuelca todos los campos del
`CombatCommandError` sin necesitar un mensaje distinto a mano por cada uno de los ~15 códigos (más
que suficiente para un harness de desarrollo; no es texto orientado a jugador final).

### 5.6 Banner final

```
======================================================================
  COMBATE TERMINADO — VICTORIA
======================================================================
```
o
```
======================================================================
  COMBATE TERMINADO — DERROTA (SCENARIO_PLOT)
======================================================================
```

(`defeatReason` solo se imprime si `status === 'DEFEAT'`.) Tras el banner, el proceso termina
(`process.exitCode = 0` — terminar en victoria o derrota son ambos cierres "sin errores" a efectos
del criterio de aceptación "se puede jugar hasta victoria/derrota sin errores").

## 6. `main.ts` / `repl.ts` — flujo end-to-end

```
main():
  args = parseArgv(process.argv)                                   // §3.2
  rawInput = loadRawContent()                                      // §3.1
  catalog = await new CatalogLoader(rawInput).load()                // lanza si el contenido real
                                                                     // está roto — no se captura,
                                                                     // es un error de setup, no de
                                                                     // partida (mismo criterio que
                                                                     // load-content.test.ts)
  leader = catalog.getLeader(args.leaderId)                         // lanza si el id no existe —
  enemy = catalog.getEnemy(args.enemyId)                            // mensaje ya incluye el id
  scenario = catalog.getScenario(args.scenarioId)                   // inválido (CatalogLoader.getOrThrow)

  randomSource = args.seed !== undefined
    ? new SeededRandomSource(args.seed)
    : new DefaultRandomSource()

  config = buildCombatEngineConfig({ catalog, leader, enemy, scenario, randomSource })
  engine = new CombatEngine(config)

  nameLookup = buildNameLookup({ leader, enemy, catalog })
  renderCtx = {
    nameLookup,
    leaderMaxHealth: config.leaderMaxHealth,
    enemyMaxHealth: config.enemyMaxHealth,
    scenarioPlotDefeatThreshold: config.scenarioPlotDefeatThreshold,
  }

  let lastSnapshot = engine.getSnapshot()
  print(renderSnapshot(lastSnapshot, renderCtx))

  runRepl({
    prompt: () => readLineFromStdin('> '),
    onLine: (line) => {
      if (lastSnapshot.status !== 'IN_PROGRESS') { /* no debería llegar aquí, ver abajo */ }

      parsed = parseLine(line, lastSnapshot)
      switch (parsed.kind):
        case 'LOCAL':
          handle HELP/STATUS/QUIT localmente (STATUS reimprime renderSnapshot(lastSnapshot, ...);
          QUIT hace process.exit(0))
          return CONTINUE

        case 'PARSE_ERROR':
          print(`✗ ${parsed.message}`)
          return CONTINUE

        case 'COMMAND':
          result = engine.dispatch(parsed.command)
          if (!result.ok):
            print(renderRejection(result.error))          // §5.5
            return CONTINUE

          for (event of result.value): print(renderEvent(event, nameLookup))   // §5.4
          lastSnapshot = engine.getSnapshot()
          print(renderSnapshot(lastSnapshot, renderCtx))

          if (lastSnapshot.status !== 'IN_PROGRESS'):
            print(renderFinalBanner(lastSnapshot))         // §5.6
            return STOP                                    // cierra el bucle, process.exitCode = 0

          return CONTINUE
    }
  })
```

`runRepl` (`repl.ts`) es un wrapper mínimo sobre el módulo `node:readline/promises` (ya disponible
en Node ≥20, requisito ya fijado por `package.json` raíz `engines.node`) — bucle `while (true) {
line = await rl.question(prompt); ... }`, sin dependencias externas de parsing de terminal.

## 7. Manejo de errores y casos de borde

### 7.1 EOF de stdin (`Ctrl+D` / pipe agotado) antes de terminar el combate

`readline.question()` rechaza/cierra al llegar EOF — el CLI debe capturarlo y terminar
limpiamente (`process.exitCode = 0`, mensaje `"Combate interrumpido (EOF) en estado IN_PROGRESS —
sin ganador."`) en vez de dejar una excepción no capturada. Esto es explícitamente DISTINTO de un
error del motor — es una forma válida de cerrar una sesión de validación manual a medias.

### 7.2 `PLAY_CARD_NUCLEO_REQUIRED` — flujo de reintento

Si el jugador escribe `play card <cardId>` (sin índice) para una carta de Ataque, el motor
devuelve `err({ code: 'PLAY_CARD_NUCLEO_REQUIRED', cardId })` — se imprime como cualquier otro
rechazo (§5.5); el jugador simplemente reintenta con `play card <cardId> <nucleoIndex>`. El CLI
**no** intenta adivinar ni completar el índice automáticamente — mantiene el criterio de "cada
comando textual mapea 1:1 a un intento de `dispatch()`", sin heurísticas ocultas.

### 7.3 Redirección de daño forzada por Berserker

`SET_DAMAGE_REDIRECT` puede aceptarse pero quedar "ignorado en la práctica" si hay un Berserker
vivo (`DAMAGE_REDIRECT_SET.forcedByBerserker === true`, H1.15 §0.3/§0.4) — el CLI debe imprimir ese
evento indicando explícitamente el aviso, ej.: `» DAMAGE_REDIRECT_SET: objetivo [1] (AVISO: hay un
Berserker vivo en mesa, todo el daño irá a él en su lugar)`. No es un error de comando — el propio
motor lo acepta; es una nota informativa de por qué el efecto observado puede no coincidir con lo
pedido.

### 7.4 `packages/cli/src/main.test.ts` — smoke test obligatorio (sin stdin real)

No se testea el bucle de `readline` interactivo (fuera de alcance razonable para un test
automatizado) — en su lugar, un test de integración que:

1. Construye `catalog`/`config`/`engine` exactamente como `main()` (reutilizando
   `buildCombatEngineConfig` con `SeededRandomSource` fija).
2. Alimenta una secuencia de líneas de texto fijas directamente a `parseLine` +
   `engine.dispatch()` (sin pasar por stdin/readline) hasta alcanzar `VICTORY` o `DEFEAT` — un
   guion de combate escrito a mano que se sepa gana o pierde de forma determinista con esa
   semilla.
3. Verifica que `engine.getSnapshot().status !== 'IN_PROGRESS'` al final, y que ningún
   `dispatch()` intermedio del guion fue rechazado inesperadamente (`result.ok === true` en cada
   paso planeado).

Esto cubre el criterio de aceptación "se puede jugar hasta victoria/derrota sin errores" con un
test reproducible, sin depender de una sesión interactiva real.

## 8. Fuera de alcance explícito

- **`SUMMON_MINION`/`RESOLVE_MINION_ACTION` manuales** — ver §0.3. Candidato natural para una
  historia futura de H2 cuando exista contenido de Secuaz real y una decisión de diseño sobre cómo
  la Dramaturgia del Enemigo los invoca en runtime.
- **Deshacer/rehacer comandos del propio Líder** (distinto de Contratiempo, que ya deshace al
  Enemigo) — no pedido por el backlog, no existe ningún mecanismo de motor para ello.
- **Guardado/carga de partida** — el backlog pide "combate completo" en una sola sesión de
  proceso, no persistencia entre ejecuciones.
- **Selección interactiva de Líder/Enemigo/Escenario** (menú) — se resuelve con flags de CLI
  (`--leader`/`--enemy`/`--scenario`, §3.2), no con un prompt adicional; mantiene "script simple"
  del backlog.
- **Combo de cartas (`COMBO` keyword vía `PLAY_CARD`)** — deuda ya asumida por H1.18, no la cierra
  esta historia (el adaptador de §2 simplemente no le atribuye ningún campo especial, consistente
  con que `handlePlayCard` no lo mira).
- **Cualquier cambio a `packages/data/**`** — el contenido real ya tiene todo lo que este harness
  necesita (`maxHealth` de Líder incluido desde H1.18); no se autoran cartas/habilidades nuevas.
