## 0. Qué es "blur" en Phaser aquí — decisión de implementación

Phaser no tiene un filtro de desenfoque gaussiano barato/nativo sobre `Canvas`/`WebGL` estándar sin
pipelines de shader custom (coste/riesgo desproporcionado para esta historia). **"Blur" se implementa
como un overlay oscuro semi-transparente** (mismo lenguaje visual que `COLOR_OVERLAY` que
`apps/shell` ya usa en sus modales — `RunStartModal`/`TurnStartModal`, `rgba(10,10,12,0.78)`) que
cubre TODO el tablero excepto el objeto en foco, que se eleva por encima de ese overlay (`setDepth`)
para seguir viéndose nítido y brillante mientras el resto se atenúa. Es exactamente lo que vision.md
describe con la palabra "blur" en la práctica ("fade/blur el resto del tablero a ~30% opacity") — un
oscurecimiento, no un desenfoque óptico.

## 1. `FocusController` — sesión de foco reentrante-segura (singleton por escena)

Nuevo módulo `packages/combat-scene/src/juice/focus-controller.ts`.

```ts
export interface FocusController {
  /** Incrementa el contador de sesiones de foco activas. Si es la transición 0→1 (primer momento
   *  grande en curso), crea/anima el overlay oscuro (fade-in) y el zoom+pan de cámara hacia el
   *  objeto resuelto por `focusId` — resuelve la Promise cuando esa animación de ENTRADA termina.
   *  Si el contador ya era >0 (un segundo momento grande empieza mientras el primero sigue
   *  resolviéndose — ej. ABILITY_ACTIVATED seguido inmediatamente de ENEMY_DAMAGED que también cruza
   *  umbral), NO repite el fade-in — solo re-apunta la cámara al nuevo `focusId` (pan suave, sin
   *  volver a animar el overlay) y resuelve inmediatamente. */
  begin(scene: Phaser.Scene, focusId: string | undefined): Promise<void>;
  /** Decrementa el contador. Si llega a 0, dispara fade-out del overlay + reset de zoom/pan de
   *  cámara a su estado neutro (`unfocusReset`), resolviendo cuando termina. Si el contador sigue
   *  >0 tras decrementar, no-op inmediato (otro momento grande sigue reclamando el foco). */
  end(scene: Phaser.Scene): Promise<void>;
}

/** Factory con clausura — mismo patrón que `createTurnBannerRecipe` (H4): un `FocusController` por
 *  `CombatScene`, reutilizado entre invocaciones. `createEffectsDirector` recibe la instancia ya
 *  construida (H5.3 §2.2, 5º parámetro) — nunca se instancia más de una vez por escena, mismo
 *  criterio "sin guardia interna, responsabilidad del caller" que el resto del módulo `juice/`. */
export function createFocusController(): FocusController;
```

**Contraste deliberado con la limitación conocida de `hitImpact`/`applyHitStop`** (documentada en
`hit-impact.ts`: "NO es reentrante... no lleva contador de referencias"): `FocusController` SÍ es
reentrante desde el diseño, precisamente porque los momentos grandes son más largos (500-1000ms+) y
más frecuentemente solapados dentro de una misma cadena de eventos (`ABILITY_ACTIVATED` +
`ENEMY_DAMAGED` casi simultáneos) que los `hitImpact` sueltos — el riesgo de flicker visible por doble
fade-in/fade-out es mucho más alto aquí, así que el contador de referencias es la solución correcta
desde el principio, no una deuda técnica aceptada.

## 2. Implementación interna de `FocusController`

### 2.1 Overlay de oscurecimiento

```ts
const FOCUS_OVERLAY_COLOR = 0x0a0a0c; // ≈ --ink
const FOCUS_OVERLAY_ALPHA = 0.7; // deja ~30% de opacidad visible al resto del tablero, vision.md
const FOCUS_OVERLAY_DEPTH = 500; // por debajo de turnBanner (BANNER_DEPTH=1000), por encima del
                                  // tablero normal (paneles/roles/dados/tiles, todos <100 hoy)
const FOCUS_TARGET_ELEVATED_DEPTH = 600; // el objeto en foco sube por encima del overlay
const FADE_IN_MS = 300;  // vision.md "fade overlay oscuro sobre tablero (duration ~300ms)"
const FADE_OUT_MS = 200; // backlog H5.4 "unfocusReset ~200ms"
```

`begin`/primera transición 0→1:
1. Crea (o reutiliza, si ya existe de una sesión previa) un `Phaser.GameObjects.Rectangle` a pantalla
   completa (`COMBAT_SCENE_VIEWPORT.width × height`), color/alpha 0, `setDepth(FOCUS_OVERLAY_DEPTH)`,
   `setScrollFactor(0)` (fijo respecto a cámara — importante porque `focusZoom` mueve la cámara, el
   overlay no debe desplazarse con ella).
2. Resuelve el game object del `focusId` (§3) y guarda su `depth` original; lo eleva a
   `FOCUS_TARGET_ELEVATED_DEPTH` para que quede por encima del overlay.
3. Anima `alpha: 0 → FOCUS_OVERLAY_ALPHA` en `FADE_IN_MS` (`Sine.easeOut`, mismo estilo que el resto
   de fades del proyecto — `turnBanner`).
4. En paralelo, invoca `focusZoom` (§4.1) hacia la posición del `focusId`.
5. Resuelve cuando ambos (fade-in del overlay + zoom) terminan.

`begin`/transición N→N+1 (N≥1): solo repanea la cámara (`focusZoom` con `zoomLevel` YA aplicado,
sin reiniciar el tween de `alpha` del overlay) y resuelve de inmediato.

`end`/transición 1→0:
1. Anima `alpha: FOCUS_OVERLAY_ALPHA → 0` en `FADE_OUT_MS`.
2. En paralelo, invoca `unfocusReset` (§4.3) — cámara vuelve a `zoom = 1`/posición neutra.
3. Restaura el `depth` original del último objeto elevado.
4. Resuelve cuando ambos terminan.

`end`/transición N→N-1 (N>1 tras decrementar): no-op, resuelve inmediatamente (Promise ya resuelta).

### 2.2 Resolución de posición del `focusId` — reutiliza y extiende lo ya existente

```ts
/** Mismo espíritu que `resolveOrCreatePlaceholder` (placeholder.ts) pero SIN crear un placeholder
 *  nuevo si no se encuentra — el foco no debe generar un rectángulo gris fantasma en el centro del
 *  viewport si `focusId` no resuelve a nada conocido; en ese caso, simplemente no se hace pan (zoom
 *  se aplica centrado en el viewport actual). Reutiliza `scene.children.getByName(focusId)`, el
 *  mismo mecanismo de nombrado ya usado por `role-view.ts` (Líder/Enemigo/Escenario),
 *  `board-anchors-view.ts` (Secuaz/Aliado por instanceId) y — NUEVO requisito de esta historia,
 *  documentado también en H5.1 §7 — `nucleo-table-view.ts` (dado por `NucleoInstanceId`). */
function resolveFocusPosition(scene: Phaser.Scene, focusId: string | undefined): { x: number; y: number } | null {
  if (focusId === undefined) return null;
  const obj = scene.children.getByName(focusId) as Phaser.GameObjects.Rectangle | null;
  return obj ? { x: obj.x, y: obj.y } : null;
}
```

**Prerrequisito técnico obligatorio (ya anticipado en H5.1 §7):** `nucleo-table-view.ts`
(`createStaticTile`) debe añadir `rect.setName(die.id)` — sin nombre, un momento grande de
`ABILITY_ACTIVATED` (cuyo `focusId` es el dado gastado, ver §3) no puede resolver su posición y el
zoom cae al centro del viewport en vez de al dado real. Sin este cambio, el resto del sistema de foco
sigue funcionando (degrada a "zoom centrado", no rompe), pero pierde el detalle exacto que pide
vision.md ("el dado elegido se destaca en la mesa").

## 3. Extensión de `resolveJuiceTarget` (`effects-director.ts`) — `focusId` para `ABILITY_ACTIVATED`

`resolveJuiceTarget` (H2.4) hoy no asigna `focusId` a `ABILITY_ACTIVATED` (cae al `default`, sin
foco). H5.4 requiere que lo tenga, para que el zoom se dirija al dado gastado:

```ts
// effects-director.ts — MODIFICADO, nuevo case en resolveJuiceTarget
case 'ABILITY_ACTIVATED':
  return { event, focusId: event.nucleoSpent.id };
```

(`ABILITY_ACTIVATED.nucleoSpent: NucleoInstance` ya existe en el evento — `NucleoInstance.id` es el
mismo `NucleoInstanceId` que `nucleo-table-view.ts` usa como nombre del tile tras el cambio de §2.2.)

## 4. Recetas standalone — para composición manual en `JUICE_CONFIG` cuando haga falta

Aunque `EffectsDirector` aplica el wrap de foco automáticamente (H5.3 §2.2), las 4 piezas visuales se
exponen TAMBIÉN como `JuiceRecipe` individuales — útil para que una entrada futura de `JUICE_CONFIG`
componga un efecto puntual sin depender del wrap automático completo (ej. un flash blanco adicional
sobre un evento que ya está en foco). Todas delegan en la MISMA instancia de `FocusController`
inyectada (evita dos fuentes de verdad sobre el estado de zoom/overlay).

### 4.1 `focusZoom`

```ts
export interface FocusZoomParams {
  readonly zoomLevel?: number;   // por defecto 1.25 (vision.md sugiere "hasta 1.3" — H5.4 usa 1.25
                                  // como default más conservador, evita que el zoom saque el resto
                                  // del tablero-mesa del viewport visible; Programmer puede tunear).
  readonly durationMs?: number;  // por defecto 300
}

/** Tween de `camera.zoom` de su valor actual a `zoomLevel`, con `camera.pan(x, y, durationMs)`
 *  simultáneo hacia la posición resuelta de `target.focusId` (§2.2) — o sin pan si no resuelve
 *  (zoom centrado en el punto de cámara actual). Implementado con `scene.cameras.main.zoomTo(...)`
 *  (tween nativo de Phaser Camera, evita reinventar el tween manual) + `.pan(...)` encadenados. */
export const focusZoom: JuiceRecipe<FocusZoomParams>;
```

### 4.2 `focusBlur`

```ts
export interface FocusBlurParams {
  readonly durationMs?: number; // por defecto 300 (FADE_IN_MS)
}

/** Crea/reutiliza el overlay de oscurecimiento (§2.1) y anima su fade-in. Uso independiente del
 *  `FocusController.begin()` automático: delega en el MISMO overlay singleton (mismo criterio de
 *  "no doble fuente de verdad" que el resto de este módulo) — llamar a esta receta manualmente
 *  mientras `EffectsDirector` YA tiene una sesión de foco activa es un no-op visual (el overlay ya
 *  está a alpha completo), documentado como comportamiento esperado, no un bug. */
export const focusBlur: JuiceRecipe<FocusBlurParams>;
```

### 4.3 `unfocusReset`

```ts
export interface UnfocusResetParams {
  readonly durationMs?: number; // por defecto 200 (FADE_OUT_MS)
}

/** Fade-out del overlay + `camera.zoomTo(1, durationMs)` + `camera.pan` de vuelta al centro neutro
 *  del viewport (`COMBAT_SCENE_VIEWPORT.width/2, height/2`). Restaura el `depth` original del último
 *  objeto elevado por `begin`. */
export const unfocusReset: JuiceRecipe<UnfocusResetParams>;
```

### 4.4 `focusWhiteLens`

```ts
export interface FocusWhiteLensParams {
  readonly durationMs?: number; // por defecto 100
}

/** Flash blanco de pantalla completa — overlay `Rectangle` blanco, `alpha: 0.8 → 0` en `durationMs`,
 *  `setDepth` POR ENCIMA de `FOCUS_TARGET_ELEVATED_DEPTH` (600) para que se vea incluso durante un
 *  momento grande ya enfocado. Efecto puntual de "impacto", NO forma parte de la sesión de
 *  `FocusController` (no incrementa/decrementa su contador) — es un flourish de un solo disparo,
 *  reutilizable en cualquier evento (grande o no) que quiera un golpe visual extra, análogo en
 *  espíritu a `hitImpact`'s flash de tinte pero a pantalla completa en vez de sobre un game object. */
export const focusWhiteLens: JuiceRecipe<FocusWhiteLensParams>;
```

## 7. Casos de test que Programmer debe cubrir

**`focus-controller.test.ts`** (con un `Phaser.Scene` fake/mock de tweens, mismo criterio que
`recipes/test-utils/fake-juice-scene.ts` ya usa para el resto de recetas):

1. `begin()` en contador 0→1: crea el overlay UNA vez, anima `alpha` a `FOCUS_OVERLAY_ALPHA`, eleva el
   `depth` del objeto resuelto por `focusId`.
2. `begin()` llamado una 2ª vez SIN que el 1º `end()` haya ocurrido (contador 1→2): NO vuelve a animar
   `alpha` del overlay (sigue en su valor ya asentado), solo repanea la cámara.
3. `end()` con contador 2→1 (tras el caso 2): no-op, overlay/zoom sin cambio.
4. `end()` con contador 1→0: fade-out del overlay a `alpha 0`, zoom vuelve a 1, `depth` del objeto
   restaurado a su valor original.
5. `focusId` que no resuelve a ningún game object nombrado (ej. `undefined`, o un id inexistente): no
   lanza, no crea placeholder fantasma — el zoom queda centrado en el viewport sin pan.

patrón que `screen-shake.test.ts`/`hit-impact.test.ts` ya establecen — mock de `scene.cameras.main`/
