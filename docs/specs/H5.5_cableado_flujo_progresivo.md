> Integra `TurnDecisionFlow` (H5.2) en `apps/shell/src/screens/CombatScreen.tsx`,
> `apps/shell/src/combat/CombatHud.tsx` y `apps/shell/src/combat/CombatBoardOverlay.tsx`. Mismo
> patrón exacto que la integración de `TargetingSignal`/`useTargetingPrompt` ya hecha en H4

## 2. Hook `useTurnRevealStage` — mismo patrón que `useTargetingPrompt`

Nuevo archivo `apps/shell/src/combat/use-turn-reveal-stage.ts`:

```ts
export function useTurnRevealStage(flow: TurnDecisionFlow | null): TurnRevealStage {
  const [state, setState] = useState<TurnRevealStage>(flow?.signal.getState() ?? { stage: 'CATEGORY' });

  useEffect(() => {
    if (!flow) return undefined;
    setState(flow.signal.getState());
    return flow.signal.subscribe(setState);
  }, [flow]);

  return state;
}
```

`CombatScreen.tsx` obtiene `turnDecisionFlow` del mismo modo que hoy obtiene `targetingSignal`/
`gestureHandle` (`useState<TurnDecisionFlow | null>`, poblado dentro del handler de
`Phaser.Core.Events.READY`, junto a las 2 asignaciones ya existentes):

```ts
const [turnDecisionFlow, setTurnDecisionFlow] = useState<TurnDecisionFlow | null>(null);
// ...dentro del mismo game.events.once(Phaser.Core.Events.READY, () => { ... }):
setTurnDecisionFlow(scene.getTurnDecisionFlow());
```

`CombatHudOverlay` recibe `turnDecisionFlow` como prop nueva (mismo patrón que `gestureHandle`), y
calcula `const stage = useTurnRevealStage(turnDecisionFlow);` una vez, pasado a `CombatHud`/
`CombatBoardOverlay` (§3/§4).

## 3. `CombatHud.tsx` — la fila de "2 acciones pagadas" se convierte en el selector de categoría

### 3.1 Qué se mantiene sin cambio

- Contador de acciones (`Acciones X/2`).
- Franja de "Paso previo (gratis)" — sin cambios, sigue siendo su propio flujo secuencial (H4,
  intocado por esta historia).
- Botón "Fin de turno" — sigue siempre visible, fuera del selector de categoría (§0 de H5.2).

### 3.2 Qué cambia — la fila de 4 controles pasa a leer `stage`

```tsx
export interface CombatHudProps {
  readonly snapshot: CombatStateSnapshot;
  readonly bridge: CombatBridge;
  readonly onEndTurn: () => void;
  readonly leaderName: string;
  readonly leaderAbilities: readonly AbilityViewData[];
  /** NUEVO H5.5 — sustituye el dispatch directo de GENERATE_ENERGY/DRAW_CARD y añade el gating de
   *  PLAY_CARD/ACTIVATE_ABILITY (§0 de H5.2: TurnDecisionFlow es el único punto de dispatch para
   *  las 4 categorías a partir de esta historia). `null` mientras CombatScene no emitió READY. */
  readonly turnDecisionFlow: TurnDecisionFlow | null;
  readonly stage: TurnRevealStage;
}
```

**Fase `CATEGORY`** — los 4 controles pasan de (2 indicadores no accionables + 2 botones) a **4
botones accionables por igual**, todos disparando `turnDecisionFlow?.selectCategory(...)`:

```tsx
{stage.stage === 'CATEGORY' && (
  <div className="combat-hud-actions" style={{ display: 'flex', flexWrap: 'wrap', gap: 'var(--hud-gap)' }}>
    <button
      disabled={!canPlayCard}
      title={disabledReasonFor('PLAY_CARD', snapshot, leaderAbilities) ?? undefined}
      style={{ ...(canPlayCard ? enabledStyle : disabledStyle), ...compactChipOverride }}
      onClick={() => turnDecisionFlow?.selectCategory('PLAY_CARD')}
    >
      {isCompact ? 'Carta' : 'Jugar Carta'}
    </button>
    <button /* Activar Habilidad — mismo patrón, selectCategory('ACTIVATE_ABILITY') */ />
    <button /* Generar Energía — selectCategory('GENERATE_ENERGY'), dispatch inmediato (H5.2 §2.1) */ />
    <button /* Robar Carta — selectCategory('DRAW_CARD') */ />
  </div>
)}
```

**Cambio de comportamiento respecto a H4 (documentado explícitamente, no es un efecto colateral no
buscado):** "Jugar Carta"/"Activar Habilidad" dejan de ser SPANs de solo-lectura (H4 los dejaba así
porque su gesto real vivía directamente en el canvas, sin selección de categoría previa) y pasan a
ser BOTONES reales — es precisamente lo que H5.2 pide: la elección de categoría ahora es un gesto
explícito en el HUD, no implícito en tocar una carta directamente.

**Fase `DETAIL`** — la fila de 4 controles se sustituye por una única cabecera de contexto + botón de
retroceso:

```tsx
{stage.stage === 'DETAIL' && (
  <div style={{ display: 'flex', justifyContent: 'space-between', alignItems: 'center', gap: SPACING.sm }}>
    <span style={{ ...TYPE.labelUpper, color: COLOR_TEXT_SECONDARY }}>
      {stage.category === 'PLAY_CARD' ? 'Elige una carta' : 'Elige una habilidad'}
    </span>
    <button
      style={{ ...chipStyle(true), background: 'transparent' }}
      onClick={() => turnDecisionFlow?.cancelDetail()}
    >
      ← Atrás
    </button>
  </div>
)}
```

`disabledReasonFor`/`freeStepAvailabilityFor` (H4, sin cambio de firma) siguen siendo la única fuente
de verdad de disponibilidad — reutilizados tal cual desde `CombatHud.tsx`.

## 4. `CombatBoardOverlay.tsx` — gating de `HandCardRow`/`AbilityRow` (Líder)

```tsx
export interface CombatBoardOverlayProps {
  // ...props existentes...
  /** NUEVO H5.5. */
  readonly stage: TurnRevealStage;
}
```

```tsx
{/* Mano — SOLO visible/interactiva en DETAIL/PLAY_CARD. En CATEGORY (o DETAIL/ACTIVATE_ABILITY),
    la fila queda oculta por completo (sin renderizar), no solo deshabilitada — la revelación
    progresiva significa que el jugador NO ve las cartas hasta elegir la categoría, no que las ve
    atenuadas (eso sería "todo de golpe pero gris", el mismo problema que motivó H5). */}
{gestureHandle && stage.stage === 'DETAIL' && stage.category === 'PLAY_CARD' && (
  <HandCardRow snapshot={snapshot} ctx={ctx} gestureHandle={gestureHandle} />
)}

{/* Habilidades del Líder — mismo criterio, gated a DETAIL/ACTIVATE_ABILITY. */}
<AbilityRow
  snapshot={snapshot}
  abilities={ctx.leaderAbilities}
  side="LEADER"
  rowY={LEADER_ABILITIES_ROW_Y}
  interactive={gestureHandle !== null && stage.stage === 'DETAIL' && stage.category === 'ACTIVATE_ABILITY'}
  visible={stage.stage === 'DETAIL' && stage.category === 'ACTIVATE_ABILITY'} // NUEVO prop, ver §4.1
  {...(gestureHandle ? { gestureHandle } : {})}
/>

{/* Habilidades del Enemigo — SIN cambio: siempre visibles, nunca interactivas (información de
    lectura del tablero, no parte de la decisión del jugador — H5.2 §5 "no gestiona QUÉ pintar" es
    coherente con que la info de SOLO LECTURA del oponente no entra en el sistema de revelación). */}
<AbilityRow snapshot={snapshot} abilities={ctx.enemyAbilities} side="ENEMY" rowY={ENEMY_ABILITIES_ROW_Y} interactive={false} />
```

### 4.1 `AbilityRow.tsx` — nuevo prop `visible`

```ts
export interface AbilityRowProps {
  // ...props existentes...
  /** NUEVO H5.5 — por defecto `true` (retrocompatible: el uso de Enemigo no lo pasa, sigue
   *  siempre visible). `false` oculta la fila completa (no renderiza nada, mismo criterio que
   *  `HandCardRow` arriba — "oculto", no "atenuado"). */
  readonly visible?: boolean;
}
```

**Por qué `AbilityRow` gana un prop en vez de que `CombatBoardOverlay` condicione su montaje con
`{condition && <AbilityRow .../>}`** (que sería más simple y coherente con cómo se trata
`HandCardRow` arriba): se prefiere el prop porque `AbilityRow` mantiene internamente el estado de
`AbilityTile`/popover de long-press (`H4_targeting_habilidades_y_ficha_personaje.md` §3) — desmontar
y remontar el componente completo en cada transición de fase perdería ese estado interno de forma
más disruptiva que ocultarlo vía CSS/`return null` temprano dentro del propio componente. Detalle de
implementación, sin impacto en el contrato de `CombatBoardOverlay`.

## 5. Panel de "Núcleos siempre visibles" — sin gating, nunca cambia con `stage`

Explícito por claridad: `MinionRow`/`AllyRow`/`nucleo-table-view` (dados)/`RoleBlock` de Líder-Enemigo-
Escenario/`CombatLogPanel` — **ninguno de estos se oculta ni se atenúa por `stage`**. La revelación
progresiva de H5.2/H5.5 aplica ÚNICAMENTE a las 2 categorías con tramo de detalle (mano de cartas,
habilidades del Líder) — todo lo demás es lectura de estado del tablero, siempre visible, coherente
con vision.md ("la mesa de dados... nunca deja de estar ahí, en el centro").
