## 1. Decisión de arquitectura: testing sin canvas/WebGL real

### 1.1 Problema

Las 4 recetas de H2.5 deben usar Phaser de verdad (`scene.tweens.add`, `scene.add.particles`,
`scene.cameras.main.shake`, `scene.time.delayedCall`). Instanciar un `Phaser.Game` real requiere un
`<canvas>` con contexto WebGL/2D; `jsdom` (usado hoy por `vitest.config.ts` como entorno por defecto del
paquete) **no** implementa `HTMLCanvasElement.getContext()` de forma utilizable por Phaser sin el paquete
nativo `canvas` (dependencia binaria, con overhead de compilación en cada entorno de CI) — y aun con eso, un
`Phaser.Game` real completo introduce timers/RAF/loop de render que complican aserciones deterministas.

### 1.2 Decisión: Scene fake mínima que registra llamadas, sin Phaser real en los tests unitarios

Los tests unitarios de recetas **no instancian `Phaser.Game`/`Phaser.Scene` reales**. En su lugar, cada
receta se testea contra un **`FakeJuiceScene`**: un objeto plano que implementa solo la superficie de la API
de Phaser que las recetas consumen (`tweens.add`, `tweens.chain`, `add.rectangle`, `add.text`,
`add.particles`, `cameras.main.shake`, `time.delayedCall`), sin renderizar nada — mismo criterio ya usado en
`effects-director.test.ts` (H2.4) de mockear la superficie mínima en vez de instanciar el sistema real.

- `FakeJuiceScene` vive en `packages/combat-scene/src/juice/recipes/test-utils/fake-juice-scene.ts` (nuevo,
  compartido por los 4 archivos de test de recetas).
- Cada método fake **registra la llamada** (config recibida: duración, ease, propiedades animadas, color de
  tint, magnitud de shake, etc.) en un array/objeto espía, y **resuelve la animación inmediatamente** (llama
  a `onComplete`/`callback` de forma síncrona o en un `Promise.resolve().then(...)`), para que el test pueda
  hacer `await recipe.play(...)` sin depender de un reloj real ni de `requestAnimationFrame`.
- Los tests verifican **configuración**, no **renderizado**: "¿se llamó a `tweens.add` con `angle: { from: 0,
  to: 360 * n }` y `duration` dentro de un rango razonable?", "¿se llamó a `cameras.main.shake(duration,
  intensity)` con `intensity` proporcional a `appliedDamage`?", "¿se llamó a `add.particles` tras el tween
  del dado?", "¿`play()` resuelve su `Promise` solo después de que el `onComplete` del tween fake se
  dispare?". Esto es exactamente lo que `docs/architecture_stack.md` §2.4 exige (testabilidad sin navegador
  real) y preserva el mismo espíritu de H2.4 §6.

```ts
// packages/combat-scene/src/juice/recipes/test-utils/fake-juice-scene.ts (contrato, no implementación)
export interface RecordedTween { targets: unknown; config: Record<string, unknown>; }
export interface RecordedShake { duration: number; intensity: number; }
export interface RecordedParticles { x: number; y: number; textureKey: string; config: Record<string, unknown>; }

export interface FakeJuiceScene {
  readonly scene: Phaser.Scene; // objeto fake, castead con `as unknown as Phaser.Scene` al llamar recipe.play
  readonly recordedTweens: RecordedTween[];
  readonly recordedShakes: RecordedShake[];
  readonly recordedParticles: RecordedParticles[];
  readonly recordedDelayedCalls: Array<{ delayMs: number; callback: () => void }>;
  /** Dispara manualmente el `onComplete` de la tween/chain en la posición `index` (orden de creación) —
   *  permite al test controlar cuándo "termina" la animación fake para verificar el orden de resolución
   *  de la Promise devuelta por la receta. */
  completeTween(index: number): void;
  /** Ejecuta inmediatamente el callback registrado por `time.delayedCall` en la posición `index`. */
  runDelayedCall(index: number): void;
}

export function createFakeJuiceScene(): FakeJuiceScene;
```

Por defecto (modo "auto-complete"), `createFakeJuiceScene()` resuelve cada tween/shake/delayedCall tan
pronto se crea (en el siguiente microtask), para que la mayoría de tests de "¿se llamó con estos params?"
no necesiten invocar `completeTween` manualmente. Los tests que verifican **orden/temporización** (p.ej.
`hitImpact` con `hitStop`, o que `diceRoll` resuelve su promesa solo tras el `particleBurst`) usan una
variante `createFakeJuiceScene({ autoComplete: false })` y llaman `completeTween`/`runDelayedCall` a mano
para verificar secuencia — mismo patrón que `vi.fn()` + array `callOrder` ya usado en
`effects-director.test.ts`.

### 1.3 Qué NO cubre este enfoque (y por qué se acepta)

Este enfoque **no** verifica que Phaser real interprete correctamente `tweens.add({ angle: {...} })` o que
`Camera.shake()` produzca el efecto visual esperado en un canvas real — verifica que **la receta invoca la
API de Phaser con los parámetros correctos**. La verificación de que Phaser en sí funciona como se espera
queda delegada a: (a) la documentación oficial de Phaser 3.80 (Tweens/Particles/Camera son API estable y
ampliamente probada, no código de este repo), y (b) la verificación manual complementaria de §7
(Playwright + dev-server, visual, no automatizada en cada CI run). Es un trade-off explícito y pragmático:
cobertura de lógica en CI rápido y determinista, sin arrastrar una dependencia binaria (`canvas` npm) ni
introducir tests frágiles por temporización real de animaciones.

## 2. Placeholders — cómo las recetas operan sin H2.6/H2.8

Ninguna receta asume que existe ya un sprite de tablero/carta/Núcleo real (eso es H2.6-H2.8). Cada receta
crea (o reutiliza si ya existe con ese nombre) un **game object placeholder** simple, nombrado con el mismo
`focusId`/constante de rol que H2.4 §3.3 ya fijó (`FOCUS_ID_LEADER`, `FOCUS_ID_ENEMY`, `FOCUS_ID_SCENARIO`,
o el `focusId` real del evento), para que cuando H2.8 introduzca los sprites reales con esos mismos nombres,
las recetas ya funcionen sin cambios (misma convención documentada en H2.4 §3.3, ahora consumida).

```ts
// packages/combat-scene/src/juice/recipes/placeholder.ts (contrato, no implementación completa)
import type Phaser from 'phaser';

/** Posición de placeholder por rol — coordenadas fijas de referencia (viewport virtual 1080×1920 de
 *  H2.6 §criterio ScaleManager) hasta que H2.8 sustituya por posiciones reales de tablero. */
export const PLACEHOLDER_POSITIONS: Record<string, { x: number; y: number }> = {
  leader: { x: 540, y: 1700 },
  enemy: { x: 540, y: 300 },
  scenario: { x: 540, y: 960 },
};
const DEFAULT_POSITION = { x: 540, y: 960 };

/** Busca un game object ya nombrado `focusId` en la escena (`scene.children.getByName`); si no existe,
 *  crea un placeholder genérico (`Phaser.GameObjects.Rectangle`, 96×96, color gris neutro) en la posición
 *  conocida para ese rol (o `DEFAULT_POSITION` si `focusId` no es uno de los 3 roles fijos — p.ej. un
 *  `cardInstanceId`/`allyInstanceId` real todavía sin sprite). Nunca crea un duplicado por el mismo nombre
 *  dos veces: reutiliza si ya existe (relevante en tests que llaman a la receta más de una vez sobre la
 *  misma `FakeJuiceScene`/escena real). */
export function resolveOrCreatePlaceholder(
  scene: Phaser.Scene,
  focusId: string | undefined,
): Phaser.GameObjects.Rectangle;
```

- `cardFlip` usa el mismo helper pero crea un rectángulo más alto que ancho (proporción de carta, p.ej.
  120×180) en vez del cuadrado 96×96 genérico — variante `resolveOrCreateCardPlaceholder` en el mismo
  archivo, misma lógica de reuso por nombre.
- Cuando `focusId` es `undefined` (p.ej. `NUCLEO_POOL_ROLLED`, que afecta a todo el pool, no a un game
  object único — H2.4 §3.3), `diceRoll` no usa este helper: crea un placeholder efímero por cada núcleo del
  `event.pool` (ver §3.1), destruido al terminar la animación, sin nombre estable (no hay `focusId` con el
  que nombrarlo, y no hace falta: el pool entero se re-crea en cada `NUCLEO_POOL_ROLLED`).

### 3.1 `diceRoll` — dados rodando + `particleBurst` embebido

**Dispara con:** `NUCLEO_POOL_ROLLED` (único mapeo en `JUICE_CONFIG` hoy, §4 de H2.4). `target.event.pool`
es `NucleoInstance[]` (id, color, value); `target.focusId` es `undefined` (afecta a todo el pool, no a un
game object único — H2.4 §3.3).

**Params (`Params` genérico de la receta):**
```ts
interface DiceRollParams {
  /** Posición base del pool en pantalla; por defecto el layout fijo de §2 si no se pasa. */
  readonly poolOrigin?: { x: number; y: number };
}
```
(No se añade a `JuiceStep.params` en `JUICE_CONFIG` en esta historia — el valor por defecto basta para el
MVP; queda como parámetro opcional para cuando H2.8 necesite pasar la posición real del layout de Núcleos.)

**Comportamiento:**
1. Por cada `NucleoInstance` de `event.pool` (iteración interna de la receta — H2.4 §4 ya documenta que
   `EffectsDirector` dispara `diceRoll` una única vez por evento, no una vez por núcleo): crear un
   placeholder efímero (`Rectangle` 64×64, color derivado de `nucleo.color` vía una tabla fija
   color→hex) en una posición distribuida horizontalmente alrededor de `poolOrigin` (o el default de §2),
   separación fija (p.ej. 80px) entre cada dado del pool.
2. Tween por dado (`scene.tweens.add`): `angle` de `0` a `360 * (2 + valor aleatorio determinista simple)`
   (rotación vistosa, 2-3 vueltas) y `scale` de `1.2 → 1` (efecto de "caída"/asentamiento), `duration: 500`,
   `ease: 'Cubic.easeOut'`. Todos los dados del pool animan en paralelo entre sí (son un único step
   `parallel` a nivel de `JuiceConfig`, pero internamente la receta puede lanzar N tweens paralelos — no
   contradice el `mode` de `JuiceStep`, que gobierna la relación entre **steps**, no dentro de una receta).
3. Al completarse el tween de cada dado (`onComplete`), disparar el **`particleBurst` embebido**:
   `scene.add.particles(dado.x, dado.y, '__WHITE' /* textura base 1x1 de Phaser, sin asset propio */, {
   speed: { min: 50, max: 120 }, lifespan: 250, quantity: 8, scale: { start: 0.4, end: 0 }, tint:
   <mismo color hex del dado> })`; el emitter se auto-destruye (`emitter.explode(...)` de un solo disparo, o
   `once: true` + `scene.time.delayedCall(300, () => emitter.destroy())`).
4. La `Promise` que retorna `play()` resuelve cuando **todos** los tweens de todos los dados han completado
   (no espera a que las partículas terminen de desvanecerse — el `particleBurst` es un remate visual fire
   -and-forget, consistente con que sea "sutil" según `backlog.md`).
5. Los placeholders efímeros de dado se destruyen (`gameObject.destroy()`) tras completar su tween (no
   quedan objetos huérfanos en la escena).

### 3.2 `cardFlip` — volteo de carta

**Dispara con:** `CARD_PLAYED`, `CONTRATIEMPO_PLAYED`, `ALLY_ENTERED_PLAY`, `MINION_SUMMONED`,
`DRAMATURGIA_CARD_DRAWN` (todos `parallel`, H2.4 §4). `target.focusId` es el id de la carta (`sourceId`,
`allyInstanceId`, `instanceId`) salvo en `DRAMATURGIA_CARD_DRAWN` (`undefined` — usa posición default de
mazo de Dramaturgia).

**Comportamiento:**
1. `resolveOrCreateCardPlaceholder(scene, target.focusId)` — placeholder proporción carta (120×180),
   posición: la conocida por rol si `focusId` es uno de los 3 roles fijos, o una posición de "mano/mesa"
   genérica (`{ x: 540, y: 1600 }`) para cualquier otro id de carta concreto.
2. Tween en dos tramos, encadenados con `scene.tweens.chain(...)` (API de Phaser ≥3.60, sustituye a
   `Timeline` legado) o dos `tweens.add` secuenciales vía `onComplete`:
   - Tramo A: `scaleX` de `1 → 0`, `duration: 150`, `ease: 'Sine.easeIn'`.
   - Al llegar a `scaleX === 0` (punto medio del flip): callback que cambia el aspecto del placeholder
     (`gameObject.setFillStyle(otroColor)` o, si ya es un `Text`, cambia el string) — simula "cambio de
     textura a mitad" (criterio de `backlog.md`) sin necesitar un sprite real todavía.
   - Tramo B: `scaleX` de `0 → 1`, `duration: 150`, `ease: 'Sine.easeOut'`.
3. `play()` resuelve cuando el tramo B completa (~300ms total).

### 3.3 `hitImpact` — flash + punch + `hitStop` opcional embebido

**Dispara con:** `LEADER_DAMAGED`, `ENEMY_DAMAGED` (ambos seguidos de `screenShake`, `sequential`),
`SCENARIO_PLOT_CHANGED`, `ALLY_DAMAGED` (ambos solos, sin `screenShake`) — H2.4 §4. `target.focusId` es
`'leader'`/`'enemy'`/`'scenario'`/`allyInstanceId` según el evento.

**Params:**
```ts
interface HitImpactParams {
  /** Si `false`, omite el hitStop. Por defecto `true` — decisión: hitStop ocurre siempre que hitImpact se
   *  dispare, salvo que un `JuiceStep.params` futuro lo desactive explícitamente para un evento concreto
   *  (p.ej. si más adelante se decide que `ALLY_DAMAGED` no lo necesita por ser "menos importante" que un
   *  golpe al Líder/Enemigo — no es el caso hoy, todas las entradas actuales lo usan). */
  readonly hitStop?: boolean;
  /** Duración del hitStop en ms. Por defecto 60. */
  readonly hitStopDurationMs?: number;
}
```

**Comportamiento:**
1. `resolveOrCreatePlaceholder(scene, target.focusId)`.
2. Punch de escala: `scene.tweens.chain` (o tweens secuenciales) `scale`: `1 → 1.1 (40ms) → 0.95 (60ms) → 1
   (60ms)`, total ~160ms (< 200ms, criterio de `backlog.md`).
3. Flash de tinte en paralelo al punch: si el placeholder es un `Rectangle`/`Graphics`
   (`setFillStyle`/`fillColor`) en vez de un `Sprite` (no hay textura real todavía), el "tinte" se simula
   cambiando `fillColor` a blanco (`0xffffff`) y de vuelta al color original tras ~80ms
   (`scene.time.delayedCall`); cuando H2.8 sustituya el placeholder por un `Sprite` real, la receta debe
   preferir `setTintFill(0xffffff)` / `clearTint()` si el game object expone esos métodos (chequeo de
   feature, `'setTintFill' in gameObject`), sin romper el caso placeholder.
4. **`hitStop` embebido** (si `params.hitStop !== false`): pausar brevemente el ritmo del combate como
   "peso" del golpe. Implementación: `scene.tweens.timeScale = 0` (pausa **todas** las tweens activas de la
   escena, incluida la del punch en curso) durante `hitStopDurationMs`, reanudado con
   `scene.tweens.timeScale = 1`. La reanudación se programa con un `setTimeout` de JavaScript plano (no
   `scene.time.delayedCall`, que también quedaría congelado si se escalase `scene.time` en vez de
   `scene.tweens` — de ahí escalar específicamente `tweens.timeScale`, no `time.timeScale` global, para que
   el propio temporizador de reanudación no se autocongele). Se ejecuta **antes** del punch (primer paso:
   congelar todas las tweens ya en curso de otros efectos concurrentes un instante, dar peso, luego lanzar
   el punch).
5. `play()` resuelve cuando el tween de punch completa (el hitStop, al afectar `tweens.timeScale` global
   brevemente, ya se resolvió por su propio `setTimeout` antes de que el punch complete, dado que
   `hitStopDurationMs` (60ms) es menor que la duración total del punch).

### 3.4 `screenShake` — sacudida de cámara nativa

**Dispara con:** `LEADER_DAMAGED`, `ENEMY_DAMAGED` (`sequential`, tras `hitImpact`), `PHASE_CHANGED`
(`sequential`, solo) — H2.4 §4.

**Params:**
```ts
interface ScreenShakeParams {
  /** Duración del shake en ms. Por defecto 200. */
  readonly durationMs?: number;
  /** Intensidad base (0-1, escala nativa de Phaser Camera.shake). Por defecto 0.01. */
  readonly baseIntensity?: number;
}
```

**Comportamiento — magnitud parametrizable por cantidad de daño:**
1. La intensidad final **no** es un valor estático de `JuiceStep.params` (que hoy no lleva ninguno en
   `JUICE_CONFIG` para `screenShake` — H2.4 §4): la receta la calcula leyendo `target.event` directamente
   (mismo patrón ya habilitado por `JuiceTarget.event`, H2.4 §0.3), evitando que `EffectsDirector`/
   `JuiceConfig` necesiten conocer nada sobre montos de daño:
   - Si `event.type === 'LEADER_DAMAGED' | 'ENEMY_DAMAGED'`: `intensity = baseIntensity * clamp(appliedDamage
     / 10, 0.5, 2)` (un golpe de referencia ~10 de daño da la intensidad base; golpes mayores/menores
     escalan proporcional, acotado a un rango razonable para que un golpe de 1 de daño no deje de sacudir
     nada y uno de 50 no rompa la cámara).
   - Si `event.type === 'PHASE_CHANGED'` (u otro tipo sin campo de daño): usa `baseIntensity` sin escalar
     (beat narrativo, no proporcional a un número).
2. `scene.cameras.main.shake(params.durationMs ?? 200, intensity)`.
3. `play()` resuelve en el callback `onComplete` del propio `shake()` de Phaser (tercer argumento de
   `Camera.shake(duration, intensity, force, callback)`), envuelto en una `Promise`.

## 5. Tests unitarios (por receta, Vitest + `FakeJuiceScene`)

Cada archivo `*.test.ts` sigue el mismo espíritu que `effects-director.test.ts` (H2.4): entorno `node`
(`// @vitest-environment node`, no hace falta `jsdom` porque `FakeJuiceScene` no toca el DOM), sin
`Phaser.Game`/`Phaser.Scene` reales.

- **`dice-roll.test.ts`:**
  - Emitir `play()` con un `event.pool` de 2 núcleos → verifica que se registraron 2 tweens (uno por dado)
    con `angle`/`scale` configurados, y que se registró al menos 1 `add.particles` por dado tras completar
    su tween respectivo (usando `completeTween` en modo `autoComplete: false` para verificar el orden:
    partículas solo después del tween del dado correspondiente).
  - Verifica que la `Promise` de `play()` resuelve solo tras completar los tweens de **ambos** dados (no
    antes).
- **`card-flip.test.ts`:**
  - Verifica que se registran los dos tramos de tween (`scaleX` 1→0 y 0→1) y que el callback de "cambio de
    aspecto" a mitad de camino se invoca entre ambos tramos (orden verificado con `completeTween` manual).
  - Verifica reuso: llamar `play()` dos veces con el mismo `focusId` no crea dos placeholders distintos
    (mismo nombre, mismo objeto).
- **`hit-impact.test.ts`:**
  - Verifica secuencia de escalas del punch (`1→1.1→0.95→1`) y duración total `< 200ms` (suma de
    `duration`s configurados en los tweens registrados).
  - Verifica que con `params.hitStop` por defecto (`undefined`, tratado como `true`) se registra la
    manipulación de `scene.tweens.timeScale` (fake expone un espía de asignaciones a esa propiedad) antes del
    punch, y que se revierte a `1` tras `hitStopDurationMs`.
  - Verifica que con `params: { hitStop: false }` no hay ninguna manipulación de `timeScale`.
- **`screen-shake.test.ts`:**
  - Con un evento `LEADER_DAMAGED` de `appliedDamage: 20` → `recordedShakes[0].intensity` es mayor que con
    `appliedDamage: 5` (proporcionalidad, sin fijar el valor exacto de la fórmula en el test — evita tests
    frágiles acoplados a la constante exacta, solo verifica la relación de orden).
  - Con un evento `PHASE_CHANGED` → `intensity === baseIntensity` (sin escalar).

Para confirmar visualmente que al menos una receta se dispara de verdad sobre placeholders en un navegador

   arrastrar navegadores Chromium/Firefox/WebKit a todo el monorepo cuando solo hace falta para esta
   verificación puntual).

   (`JuiceSmokeScene`, temporal, solo para este archivo — no `HelloCombatScene`, para no acoplar esta
   verificación al contenido 2×2×2 real) que en su `create()` invoca directamente `diceRoll.play(this, {
   event: <evento NUCLEO_POOL_ROLLED de ejemplo> }, {})` (sin pasar por `EffectsDirector`/`CombatBridge` —
   no hace falta para este smoke test visual) y toma un screenshot con `page.screenshot()` tras esperar el
   tiempo de la animación.
