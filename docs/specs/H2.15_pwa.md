## 1. Decisión de enfoque técnico — `vite-plugin-pwa`, no manifest/SW manuales

**Decisión: usar `vite-plugin-pwa` (estrategia `generateSW`, basada en Workbox) como dependencia nueva de
`apps/shell`.** Se descarta escribir el manifest y el service worker a mano.

**Por qué:**
- El proyecto ya delega a Vite el bundling/hashing de assets (`vite build`); un service worker manual
  tendría que conocer de antemano los nombres de archivo con hash generados por Vite en cada build para
  poder precachearlos por URL exacta — o reimplementar un mecanismo de descubrimiento de assets que Workbox
  ya resuelve de fábrica vía `injectManifest`/`generateSW` (lee el manifest de build de Vite y genera la
  lista de precache automáticamente, con invalidación por hash de contenido).
- `architecture_stack.md` §4.1 ya lo recomienda explícitamente ("recomendado vía plugin de build tipo Vite
  PWA/Workbox") — no es una desviación del plan técnico, es completarlo.
- Para un vertical slice técnico (no production-grade todavía), escribir Cache API a mano (`caches.open`,
  `cache.addAll`, invalidación de versión manual) es más código propio que mantener por el mismo resultado
  observable; `vite-plugin-pwa` es la opción estándar de la industria para proyectos Vite+PWA y reduce la
  superficie de bugs de un sistema de cacheo (invalidación es notoriamente fácil de hacer mal a mano).
- Contrapartida asumida conscientemente: una dependencia nueva (`vite-plugin-pwa` + `workbox-*` transitivas)
  y algo de "magia" de generación de código en build — aceptable porque el contrato de salida (manifest
  enlazado + SW registrado + precache activo) es exactamente lo que pide el criterio de aceptación, y
  Programmer no necesita entender Workbox internamente para completar la historia, solo su configuración
  declarativa (§2).

## 2. Configuración de `vite-plugin-pwa` — `apps/shell/vite.config.ts`

### 2.1 Dependencia nueva

`apps/shell/package.json` — añadir a `devDependencies`:

```json
"vite-plugin-pwa": "^0.20.0"
```

(Versión compatible con Vite 5, ya fijado en el proyecto; Programmer confirma la última versión estable de
la serie 0.20.x compatible con `vite@^5.0.0` al instalar.)

### 2.2 Plugin en `vite.config.ts` — contrato de configuración

```ts
// apps/shell/vite.config.ts (extensión — añade el plugin, resto sin cambio)
import { defineConfig } from 'vite';
import react from '@vitejs/plugin-react';
import tsconfigPaths from 'vite-tsconfig-paths';
import { VitePWA } from 'vite-plugin-pwa';

export default defineConfig({
  plugins: [
    react(),
    tsconfigPaths(),
    VitePWA({
      // generateSW: Workbox genera el SW completo a partir de config declarativa (§1) — sin
      // service-worker.js propio que mantener a mano.
      strategies: 'generateSW',
      registerType: 'autoUpdate', // el SW nuevo reemplaza al anterior sin prompt de usuario (MVP: sin UI de "hay una versión nueva, recarga").
      manifest: {
        // Sustituye por completo a apps/shell/src/pwa/manifest.webmanifest (§2.3 — se retira el stub).
        name: 'The Collector',
        short_name: 'Collector',
        description: 'Juego de cartas PVE roguelite — cruza universos TCG en combates asimétricos.',
        display: 'standalone',
        orientation: 'portrait-primary', // móvil primero (decisions.md), PC sigue funcionando en navegador normal
        start_url: '/',
        scope: '/',
        theme_color: '#1a1a2e', // tono oscuro coherente con temática "coleccionista/cajas", placeholder ajustable sin bloquear la historia
        background_color: '#1a1a2e',
        icons: [
          { src: '/icons/icon-192.png', sizes: '192x192', type: 'image/png' },
          { src: '/icons/icon-512.png', sizes: '512x512', type: 'image/png' },
          { src: '/icons/icon-512-maskable.png', sizes: '512x512', type: 'image/png', purpose: 'maskable' },
        ],
      },
      workbox: {
        // Precache del app shell: JS/CSS/HTML del build de Vite + iconos + los 9 JSON de catálogo ya
        // copiados a public/data (mismo criterio "cache-first para el catálogo hoy empaquetado" de
        // decisions.md/architecture_stack.md §4.1 — Workbox precachea todo lo que matchea el glob,
        // sirviéndolo cache-first por defecto una vez precacheado).
        globPatterns: ['**/*.{js,css,html,png,svg,woff2}'],
        // Runtime caching explícito para los datos de catálogo bajo /data/**: NetworkFirst, no
        // CacheFirst — aunque hoy el catálogo va empaquetado (precacheado igualmente por el glob
        // anterior), esta regla dejafuncionando de fábrica la migración futura a catálogo remoto
        // (architecture_stack.md §4.1: "network-first/stale-while-revalidate para el catálogo si en
        // el futuro se sirve remoto") sin tener que tocar esta config de nuevo.
        runtimeCaching: [
          {
            urlPattern: /\/data\/.*\.json$/,
            handler: 'NetworkFirst',
            options: {
              cacheName: 'collector-catalog-data',
              networkTimeoutSeconds: 3,
            },
          },
        ],
        navigateFallback: '/offline.html', // §4 — fallback offline mínimo
        navigateFallbackDenylist: [/^\/data\//], // no interceptar peticiones de datos como si fueran navegación
      },
      devOptions: {
        enabled: false, // SW no se activa en `npm run dev` (§0.1) — coherente con que la verificación (§6) usa build+preview, evita interferencias de caché durante desarrollo activo
      },
    }),
  ],
  server: { port: 5173 },
});
```

**Nota de precisión sobre `globPatterns` y `public/data`:** Vite copia el contenido de `public/` tal cual a
la raíz del build (`dist/data/**`), sin pasar por el pipeline de hash de assets de `src/`. `vite-plugin-pwa`
en modo `generateSW` incluye por defecto todo lo servido desde `dist/` que matchee `globPatterns` — Programmer
debe confirmar en build real (§6.1) que `dist/data/**/*.json` efectivamente entra al precache manifest
generado (visible en el log de build de `vite-plugin-pwa`, que imprime el conteo de entradas precacheadas).

### 2.3 Retirada del stub anterior

- `apps/shell/src/pwa/manifest.webmanifest` — **se elimina**. `VitePWA({ manifest: {...} })` genera el
  `manifest.webmanifest` final directamente en el build (`dist/manifest.webmanifest`) e inyecta el
  `<link rel="manifest">` en `index.html` automáticamente — no queda ningún archivo manual que mantener.
- `apps/shell/src/pwa/service-worker.js` — **se elimina**. El SW real se genera en `dist/sw.js` por
  `vite-plugin-pwa`/Workbox — ninguna línea del stub sobrevive.
- Carpeta `apps/shell/src/pwa/` queda vacía tras esto; se elimina la carpeta completa (nada más vivía ahí
  según H2.2).

## 3. Iconos — placeholder mínimo, 3 archivos, arte real diferido

**Decisión:** 3 PNG estáticos en `apps/shell/public/icons/`, generados una única vez y comprometidos al
repo (no generados en build):

- `icon-192.png` — 192×192, fondo sólido `#1a1a2e` + inicial "C" centrada en un color de contraste simple.
- `icon-512.png` — 512×512, mismo diseño escalado.
- `icon-512-maskable.png` — 512×512, mismo diseño pero con el contenido central dentro del "safe zone"
  circular del 80% (requisito de iconos `purpose: maskable` para que Android no recorte la "C" en máscaras
  circulares/redondeadas).

Programmer puede generarlos con cualquier herramienta disponible (script Node de un solo uso con `sharp`/
sharp-less alternativas, o cualquier editor de imagen) — el contenido visual exacto **no es parte del
criterio de aceptación de esta historia** (arte real es tarea de una futura iteración de contenido, ya
señalado en la propia descripción del backlog: "vite-plugin-pwa... no bloquees esperando arte final"). El
único contrato que importa es: 3 archivos PNG válidos en las rutas/tamaños exactos referenciados por
`manifest.icons` (§2.2), para que el navegador pueda mostrar un icono real al instalar (no un icono roto).

## 4. Fallback offline — `apps/shell/public/offline.html`

Página HTML estática mínima, sin JS/React (se sirve directamente por el SW cuando `navigateFallback` se
activa — ver §2.2 — es decir, cuando el usuario navega a una ruta de la app sin red y sin esa entrada en
caché):

```html
<!-- apps/shell/public/offline.html (contrato completo — HTML estático, sin build) -->
<!doctype html>
<html lang="es">
  <head>
    <meta charset="UTF-8" />
    <meta name="viewport" content="width=device-width, initial-scale=1.0" />
    <title>The Collector — sin conexión</title>
  </head>
  <body>
    <p>The Collector está offline. Algunos datos pueden no estar disponibles hasta recuperar conexión.</p>
  </body>
</html>
```

Coincide textualmente con el criterio de aceptación del backlog ("página simple mostrando 'app offline,
algunos datos pueden no estar disponibles'"). Vite copia este archivo tal cual a `dist/offline.html`, y
`navigateFallback: '/offline.html'` (§2.2) hace que Workbox lo sirva cuando una navegación de página falla
por red y no hay entrada de precache exacta para esa ruta.

### 6.3 `apps/shell/e2e-pwa/pwa-installability.spec.ts` (nuevo)

Criterio pragmático de "instalabilidad verificable sin Lighthouse real en CI": confirmar por código los 3
requisitos mínimos que Chrome exige para considerar una PWA instalable (manifest válido enlazado, SW
registrado y activo, icons presentes) — sin depender de la heurística interna de Lighthouse ni de un evento
`beforeinstallprompt` (no disparable de forma fiable en modo headless/CI).

```ts
// apps/shell/e2e-pwa/pwa-installability.spec.ts (contrato completo)
import { test, expect } from '@playwright/test';

test('manifest enlazado en <head> con los campos mínimos de instalabilidad', async ({ page }) => {
  await page.goto('/');

  const manifestHref = await page.locator('link[rel="manifest"]').getAttribute('href');
  expect(manifestHref).toBeTruthy();

  const manifestResponse = await page.request.get(new URL(manifestHref!, page.url()).toString());
  expect(manifestResponse.ok()).toBe(true);
  const manifest = await manifestResponse.json();

  expect(manifest.name).toBe('The Collector');
  expect(manifest.display).toBe('standalone');
  expect(manifest.start_url).toBeTruthy();
  expect(manifest.icons.length).toBeGreaterThanOrEqual(2);
  // al menos un icono >= 512x512 — requisito de Chrome para instalabilidad
  expect(manifest.icons.some((icon: { sizes: string }) => icon.sizes === '512x512')).toBe(true);
});

test('service worker se registra y llega a estado "activated" sin errores', async ({ page }) => {
  const pageErrors: Error[] = [];
  page.on('pageerror', (error) => pageErrors.push(error));

  await page.goto('/');

  const swState = await page.evaluate(async () => {
    const registration = await navigator.serviceWorker.ready;
    return registration.active?.state ?? null;
  });

  expect(swState).toBe('activated');
  expect(pageErrors).toEqual([]);
});

test('segunda carga funciona con el SW activo, sin errores de consola', async ({ page }) => {
  await page.goto('/');
  await page.waitForFunction(() => navigator.serviceWorker.controller !== null);

  const pageErrors: Error[] = [];
  page.on('pageerror', (error) => pageErrors.push(error));

  await page.reload();

  expect(pageErrors).toEqual([]);
  await expect(page.getByText('The Collector')).toBeVisible();
});
```

**Instalabilidad real en dispositivo** (icono correcto, prompt nativo de "Añadir a pantalla de inicio",
comportamiento standalone real): verificación **manual**, no automatizada — Programmer confirma una vez en
Chrome Android/desktop (`chrome://inspect` o menú "Instalar app") contra el build servido por `vite preview`
o un deploy real, y lo anota como evidencia de cierre de la historia (captura de pantalla o nota en el PR),
mismo criterio "no gate de CI" que el resto de verificación E2E de `apps/shell`.
