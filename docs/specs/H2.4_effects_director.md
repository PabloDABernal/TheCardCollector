## 0. Qué resuelve esta historia (y qué NO)

### 0.1 Dentro de alcance de H2.4

1. Definir en `packages/combat-scene/src/juice/` los contratos `JuiceRecipe`, `JuiceStep`, `JuiceConfig` y la
   clase/factory `EffectsDirector` (§2-§3).
2. Construir la tabla `JUICE_CONFIG`: mapeo declarativo y **completo** (los 23 tipos reales de `CombatEvent`,
   no un subconjunto) de `CombatEvent['type']` → `JuiceStep[]` (§4) — completo aunque casi todas las
   entradas de esta historia usen recetas **stub**, para que H2.5 no tenga que rediseñar la tabla, solo
   sustituir implementaciones de receta manteniendo los mismos `recipeId`.
3. Implementar **4 recetas STUB** (una por cada nombre de receta ya decidido en decisions.md: `diceRoll`,
   `cardFlip`, `hitImpact`, `screenShake`) que cumplen `JuiceRecipe` pero no tocan Phaser de verdad (sin
   tweens, sin partículas, sin `Camera.shake()`) — solo registran la llamada (p.ej. `console.debug`) y
   retornan una `Promise` ya resuelta. Sirven para demostrar end-to-end que el mapeo evento→receta funciona;
   **no** son el criterio de "feel" (eso es H2.5).
4. Un test unitario (Vitest, sin canvas real, sin `jsdom`) que mockea `CombatBridge.subscribeSceneEvents`,
   emite eventos de ejemplo y verifica que `EffectsDirector` invoca exactamente las recetas correctas según
   `JUICE_CONFIG`, en el orden/modo (`parallel`/`sequential`) declarado.
5. Actualizar `packages/combat-scene/package.json`/`tsconfig.json` para declarar la nueva dependencia real a
   `@collector/combat-bridge` (primer import real de este paquete desde `combat-scene`, anticipado pero no
   materializado por H2.1/H2.3 — ver `docs/specs/H2.3_combat_bridge.md` §5, ya permitido por ESLint
   boundaries, solo falta el import real y la entrada de `package.json`).
6. Actualizar `docs/architecture_stack.md` §3.2 para reflejar el contrato real (ver §0.3).

### 0.2 Fuera de alcance de H2.4 (frontera explícita con otras historias)

- **Ninguna receta real.** `diceRoll`/`cardFlip`/`hitImpact`/`screenShake` con tweens, partículas
  (`particleBurst` embebido en `diceRoll`), tinte/punch/`hitStop` (embebido en `hitImpact`) y
  `Camera.shake()` real son H2.5. Esta historia solo fija **el contrato** (`JuiceRecipe`) y **los IDs**
  (`diceRoll`, `cardFlip`, `hitImpact`, `screenShake`) que H2.5 debe implementar — H2.5 sustituye
  únicamente el contenido de `packages/combat-scene/src/juice/recipes/*.ts`, sin tocar `JUICE_CONFIG`,
  `EffectsDirector` ni `JuiceRecipe` salvo que un ajuste de "feel" real exija cambiar parámetros de la tabla.
- **Ninguna instanciación real desde `CombatScene`.** `CombatScene.create()` invocando
  `createEffectsDirector(...).attach(bridge, this)` es H2.6 — esta historia deja el módulo listo para ser
  consumido, testeado en aislamiento con un `Phaser.Scene` simulado (nunca uno real).
- **Ninguna resolución a game objects reales.** El `focusId` que `EffectsDirector` calcula por evento (§3.3)
  es una cadena (`string`) que H2.8 (view layer) usará para localizar el `Phaser.GameObject` correspondiente
  (p.ej. `scene.children.getByName(focusId)` o un registro propio `Map<string, GameObject>` que H2.8
  construya) — esta historia no crea ese registro ni ningún sprite/dado/carta real.
- **Ningún audio.** `JuiceStep.soundId` (H2.13) no se añade todavía — se deja la puerta abierta en el
  comentario del tipo, no en el tipo mismo (evita un campo opcional sin uso real, mismo criterio aplicado en
  H2.1/H2.2/H2.3 de "no construir sin consumidor").

### 0.3 Decisión de arquitectura: `CombatEvent` real y `JuiceTarget`, no el sketch de `architecture_stack.md` §3.2

**Decisión: `JuiceConfig`/`JuiceRecipe` se definen contra el `CombatEvent` real de
`packages/domain/combat/src/types/events.ts` (23 variantes, cerradas desde H1.3 hasta H1.18), no contra el
sketch `CORE_ROLLED`/`DAMAGE_DEALT`/`PLOT_CHANGED`/etc. de `architecture_stack.md` §2.2/§3.2`, que quedó
obsoleto en cuanto H1.3 formalizó nombres de dominio reales (`NUCLEO_POOL_ROLLED` en vez de `CORE_ROLLED`,
`LEADER_DAMAGED`/`ENEMY_DAMAGED`/`ALLY_DAMAGED` en vez de un único `DAMAGE_DEALT`, etc.) — el propio
`events.ts` línea 12-19 ya documenta esta divergencia respecto al sketch original.

**Por qué:** `EffectsDirector` consume el stream real de `subscribeSceneEvents` (H2.3), que emite instancias
reales de `CombatEvent` — construir `JuiceConfig` contra nombres de evento que no existen en runtime haría
que la tabla nunca disparase nada. Esta spec es la que cierra esa deuda documentada desde H1.3, igual que
H2.3 cerró la ubicación de `CombatBridge` (spec H2.3 §0.3).

**Segunda decisión, `JuiceTarget`:** `architecture_stack.md` §3.2 esbozaba `play(scene, target: JuiceTarget,
params)` sin definir `JuiceTarget`. Se define aquí como:

```ts
interface JuiceTarget {
  readonly event: CombatEvent;   // evento completo, por si la receta necesita más contexto (montos, ids)
  readonly focusId?: string;     // id semántico del game object principal afectado (ver §3.3), si aplica
}
```

`focusId` es una cadena estable (p.ej. `'leader'`, `'enemy'`, `'scenario'`, o el `cardInstanceId`/
`allyInstanceId`/`minionDefinitionId` del evento) — nunca una referencia a un `Phaser.GameObject` real,
porque `EffectsDirector` no sabe (ni debe saber) cómo `combat-scene/view` (H2.8) nombra o localiza sus
game objects. Las recetas reales de H2.5/H2.8 son quienes resuelven `focusId → GameObject` dentro de
`scene`.

**Consecuencia para `architecture_stack.md`:** esta spec actualiza §3.2 para reflejar el contrato real (ver
diff al final de este documento, §7) — mismo patrón que la enmienda de H2.3 §0.3 para `CombatBridge`.

## 2. Contrato de `JuiceRecipe`

```ts
// packages/combat-scene/src/juice/juice-recipe.ts
import type Phaser from 'phaser';
import type { CombatEvent } from '@collector/domain-combat';

export interface JuiceTarget {
  /** Evento completo que disparó este step — la receta lee de aquí cualquier dato que necesite
   *  (montos, ids, side, etc.) sin que `EffectsDirector` tenga que conocer la forma de cada evento. */
  readonly event: CombatEvent;
  /** Id semántico y estable del game object principal afectado por este step (ver §3.3
   *  `resolveJuiceTarget`) — p.ej. 'leader', 'enemy', 'scenario', o un `cardInstanceId`/
   *  `allyInstanceId` real. `undefined` cuando el evento no tiene un único foco (p.ej.
   *  `NUCLEO_POOL_ROLLED` afecta a todo el pool, no a un único game object). La resolución de
   *  `focusId` a un `Phaser.GameObject` real es responsabilidad de la receta (H2.5) contra el
   *  registro de game objects que construya `combat-scene/view` (H2.8) — fuera de alcance aquí. */
  readonly focusId?: string;
}

export type JuiceStepMode = 'parallel' | 'sequential';

/** Una entrada de `JuiceConfig`: qué receta ejecutar, con qué parámetros, y cómo relacionarse en el
 *  tiempo con los steps anteriores del mismo evento (ver `EffectsDirector.resolveEvent`, §3.2). */
export interface JuiceStep {
  readonly recipeId: string;
  readonly params?: Record<string, unknown>;
  readonly mode: JuiceStepMode;
}

/** Contrato que toda receta (stub en H2.4, real en H2.5) debe cumplir. `id` debe coincidir
 *  exactamente con el `recipeId` usado en `JUICE_CONFIG` — es la clave de `JuiceRecipeRegistry`
 *  (§3.1) que `EffectsDirector` usa para resolver cada `JuiceStep` a una implementación concreta. */
export interface JuiceRecipe<Params = Record<string, unknown>> {
  readonly id: string;
  play(scene: Phaser.Scene, target: JuiceTarget, params: Params): Promise<void>;
}

/** Registro id→implementación, inyectado en `createEffectsDirector` (§3.1). H2.4 inyecta
 *  `STUB_RECIPE_REGISTRY` (`recipes/index.ts`); H2.6/H2.9 inyectarán el registro real de H2.5 sin
 *  tocar `EffectsDirector` ni `JUICE_CONFIG`. */
export type JuiceRecipeRegistry = Record<string, JuiceRecipe>;
```

Nota: `params` es `Record<string, unknown>` en el registro (tipo borrado) — cada receta concreta puede
tipar su propio `Params` genérico en su implementación real (H2.5, p.ej. `screenShake` con `{ magnitude:
number; durationMs: number }`), pero `JuiceRecipeRegistry`/`EffectsDirector` no necesitan conocer esos tipos
concretos: son datos de configuración (`JuiceStep.params`) validados solo por la propia receta al leerlos.

## 3. Contrato de `EffectsDirector`

### 3.1 Interfaz pública

```ts
// packages/combat-scene/src/juice/effects-director.ts
import type Phaser from 'phaser';
import type { CombatBridge, CombatEvent, Unsubscribe } from '@collector/combat-bridge';
import type { JuiceConfig } from './juice-config';
import type { JuiceRecipeRegistry } from './juice-recipe';

export interface EffectsDirector {
  /** Se suscribe a `bridge.subscribeSceneEvents` y resuelve cada evento contra `JuiceConfig`
   *  (inyectado en la construcción, ver `createEffectsDirector`). Retorna el `Unsubscribe` de esa
   *  suscripción (H2.6 lo invoca en el `shutdown`/`destroy` de `CombatScene`). Llamar `attach` dos
   *  veces sobre la misma instancia crea dos suscripciones independientes — el caller (H2.6) es
   *  responsable de no hacerlo por accidente; no hay guardia interna contra doble-attach en esta
   *  historia (YAGNI — un único `CombatScene.create()` por combate en todo el alcance de H2). */
  attach(bridge: CombatBridge, scene: Phaser.Scene): Unsubscribe;
}

/** Único punto de construcción — mismo patrón que `createCombatBridge` (H2.3): sin `new
 *  EffectsDirector(...)` expuesto, deja puerta abierta a validación/instrumentación futura sin
 *  romper la firma pública. */
export function createEffectsDirector(
  config: JuiceConfig,
  recipes: JuiceRecipeRegistry,
): EffectsDirector;
```

### 3.2 Comportamiento interno — `resolveEvent` y semántica `parallel`/`sequential`

Al recibir un `CombatEvent` vía el listener de `subscribeSceneEvents`:

1. `const steps = config[event.type] ?? [];` — si no hay entrada (o está vacía) para ese tipo de evento,
   `EffectsDirector` no hace nada (caso normal y frecuente, ver tabla §4 — la mayoría de eventos no tienen
   receta en esta historia).
2. `const target = resolveJuiceTarget(event);` (§3.3).
3. Recorrer `steps` en orden, manteniendo un buffer `pending: Promise<void>[]` de efectos en curso:
   - Si `step.mode === 'parallel'`: `pending.push(recipe.play(scene, target, step.params ?? {}))` — se
     dispara inmediatamente, sin esperar a que termine, y sin bloquear el recorrido del array.
   - Si `step.mode === 'sequential'`: primero `await Promise.all(pending)` (esperar a que termine todo lo
     disparado en paralelo hasta ahora), vaciar `pending`, luego `await recipe.play(scene, target, step.params
     ?? {})` antes de continuar con el siguiente step.
   - `recipe = recipes[step.recipeId]` — si `step.recipeId` no existe en el registro inyectado, es un error
     de configuración: se lanza (`throw`) inmediatamente al resolver ese step, nunca falla en silencio (una
     `JuiceConfig` mal escrita — recipeId con typo — debe fallar ruidosamente en desarrollo/test, no
     tragarse el error).
   - Al terminar de recorrer todos los `steps`, `await Promise.all(pending)` una última vez (flush de
     cualquier `parallel` final) antes de considerar el evento resuelto.
4. El listener registrado en `bridge.subscribeSceneEvents` **no** propaga ni espera la promesa resultante de
   `resolveEvent` hacia el bus de eventos (el bus de dominio es síncrono, fire-and-forget respecto a la capa
   de juice) — `resolveEvent` corre en segundo plano; errores no capturados dentro de una receta deben
   propagarse como excepción no manejada visible en consola/test (no se añade `try/catch` silencioso — un
   fallo de receta debe ser ruidoso durante desarrollo).
5. `attach` no bloquea nunca el flujo del juego: un `CombatEvent` que no tiene entrada en `JuiceConfig` (la
   mayoría, §4) resuelve el paso 1 en O(1) y retorna sin crear ninguna promesa.

### 3.3 `resolveJuiceTarget(event): JuiceTarget`

Función pura (no exportada como parte del contrato público del paquete, solo usada internamente por
`effects-director.ts`, pero testeada indirectamente vía el test de §5) que calcula el `focusId` (si aplica)
a partir de la forma concreta de cada variante de `CombatEvent`:

| Tipo de evento | `focusId` |
|---|---|
| `NUCLEO_POOL_ROLLED` | `undefined` (afecta a todo el pool, no a un núcleo concreto) |
| `LEADER_DAMAGED` | `'leader'` (constante, ver `FOCUS_ID_LEADER`) |
| `ENEMY_DAMAGED` | `'enemy'` (constante, ver `FOCUS_ID_ENEMY`) |
| `ALLY_DAMAGED` | `event.allyInstanceId` |
| `SCENARIO_PLOT_CHANGED` | `'scenario'` (constante, ver `FOCUS_ID_SCENARIO`) |
| `CARD_PLAYED` | `event.sourceId` |
| `CONTRATIEMPO_PLAYED` | `event.sourceId` |
| `ALLY_ENTERED_PLAY` | `event.allyInstanceId` |
| `MINION_SUMMONED` | `event.instanceId` |
| `DRAMATURGIA_CARD_DRAWN` | `undefined` (no hay instancia de carta concreta todavía en este evento — solo `icon`) |
| `PHASE_CHANGED` | `event.source === 'ENEMY' ? 'enemy' : 'scenario'` |
| (resto de tipos, sin receta mapeada en `JUICE_CONFIG` §4) | no se calcula — `resolveJuiceTarget` solo se invoca cuando `steps.length > 0` |

`FOCUS_ID_LEADER = 'leader'`, `FOCUS_ID_ENEMY = 'enemy'`, `FOCUS_ID_SCENARIO = 'scenario'` se exportan como
constantes de `effects-director.ts` — H2.8 debe nombrar (`GameObject.setName(...)` o registro propio) los
game objects del tablero/Líder/Enemigo/Escenario usando exactamente estos 3 strings para que las recetas
reales de H2.5 puedan resolverlos sin re-inventar convención.

## 4. `JUICE_CONFIG` — tabla completa evento→receta

```ts
// packages/combat-scene/src/juice/juice-config.ts
import type { CombatEvent } from '@collector/domain-combat';
import type { JuiceStep } from './juice-recipe';

export type JuiceConfig = Record<CombatEvent['type'], JuiceStep[]>;
```

Tabla completa (23 tipos reales de `packages/domain/combat/src/types/events.ts`) — entradas sin receta se
declaran como array vacío `[]` (no se omiten, para que quede explícito en el código que la ausencia de
receta es una decisión, no un olvido):

| `CombatEvent['type']` | Steps (`recipeId` / `mode`) | Razonamiento |
|---|---|---|
| `NUCLEO_POOL_ROLLED` | `diceRoll` / `parallel` | Rodar los dados del pool — el "verbo" visual literal de este evento; H2.5 itera `event.pool` internamente para animar cada núcleo. |
| `ABILITY_ACTIVATED` | `[]` | El consumo de Núcleo en sí no tiene receta propia — el efecto observable (daño/Trama) ya dispara su propia receta vía el evento específico que le sigue en el mismo `dispatch()`. Evita doble animación del mismo gesto. |
| `TURN_ENDED` | `[]` | Cambio de turno es HUD (indicador de turno), no juice de canvas. |
| `COOLDOWNS_TICKED` | `[]` | Visual de CD es H2.10 (view layer, no receta de `EffectsDirector`). |
| `LEADER_DAMAGED` | `hitImpact` / `sequential`, luego `screenShake` / `sequential` | Golpe recibido por el Líder: impacto + shake es el combo de referencia (forcetable.net/strawtable.net) para un golpe importante. `hitImpact` primero (incluye `hitStop` embebido) para dar peso antes del shake de cámara. |
| `SCENARIO_PLOT_CHANGED` | `hitImpact` / `sequential` | Cambio de Trama es un "golpe" al Escenario, pero sin sacudir la cámara del jugador (reservado a daño al Líder/Enemigo, para no banalizar el shake). |
| `COMBO_TRIGGERED` | `[]` | Sin receta dedicada en el MVP — el efecto que lo originó ya animó vía su propio evento; un brillo de "combo" queda para iteración futura de feel, fuera del alcance de las 4 recetas base. |
| `CONTRATIEMPO_PLAYED` | `cardFlip` / `parallel` | Jugar una carta de Contratiempo es, visualmente, el mismo gesto que jugar cualquier carta. |
| `ALLY_ENTERED_PLAY` | `cardFlip` / `parallel` | Aliado entrando en mesa es una carta jugada desde mano. |
| `ALLY_DAMAGED` | `hitImpact` / `sequential` | Golpe absorbido por Aliado: impacto sí, shake de cámara no (reservado a golpes al protagonista/Enemigo). |
| `DAMAGE_REDIRECT_SET` | `[]` | Cambio de intención de redirección, sin efecto visual propio (H2.7/H2.8 lo reflejan como resaltado de selección, no juice). |
| `MINION_SUMMONED` | `cardFlip` / `parallel` | Secuaz entrando en mesa es una carta jugada. |
| `MINION_ACTION_RESOLVED` | `[]` | Envoltorio informativo — el efecto real (daño/Trama) ya se animó vía su propio evento (mismo criterio que `ABILITY_ACTIVATED`). |
| `MINION_ACTION_SKIPPED` | `[]` | No hay nada que animar (no hay Secuaces en mesa). |
| `MINION_PASSIVE_EFFECTS_APPLIED` | `[]` | Agregado pasivo de inicio de turno enemigo — sin receta dedicada en el MVP (candidato a iteración futura si el feel lo pide, no una de las 4 recetas base). |
| `PHASE_CHANGED` | `screenShake` / `sequential` | Beat narrativo importante (cambio de fase de Enemigo/Escenario) — shake de cámara marca el momento sin necesitar una 5ª receta. |
| `LEADER_LEVELED_UP` | `[]` | Sin receta dedicada en el MVP — candidato a receta de "shimmer"/glow propia, fuera de las 4 base (decisions.md no la incluye). |
| `CARD_PLAYED` | `cardFlip` / `parallel` | El "verbo" visual literal de jugar cualquier carta desde mano. |
| `ENEMY_DAMAGED` | `hitImpact` / `sequential`, luego `screenShake` / `sequential` | Golpe del Líder al Enemigo — mismo peso visual que `LEADER_DAMAGED`, por simetría. |
| `LEADER_SHIELD_GAINED` | `[]` | Ya cubierto por el `cardFlip` de `CARD_PLAYED` que precede a este evento en el mismo `dispatch()` — no se duplica el gesto. |
| `DRAMATURGIA_CARD_DRAWN` | `cardFlip` / `parallel` | Revelar una carta de Dramaturgia del Enemigo es, visualmente, un flip de carta. |
| `DRAMATURGIA_DECK_RESHUFFLED` | `[]` | Evento de contabilidad interna del mazo de Dramaturgia, sin contraparte visual en el MVP. |
| `COMBAT_ENDED` | `[]` | Resultado final se comunica vía modal React (HUD no-juice, `architecture_stack.md` §2.3), no vía canvas/`EffectsDirector`. |

11 de 23 tipos quedan mapeados a receta (supera el "al menos 3" de `backlog.md`); los 12 restantes quedan
`[]` explícitamente, documentando la decisión de no animarlos en el MVP en vez de dejarlos ambiguos.

     const listeners: Array<(e: CombatEvent) => void> = [];
     return {
       bridge: {
