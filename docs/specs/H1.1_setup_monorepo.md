# H1.1 — Setup del monorepo y tooling de base

> Spec técnica del Architect para la historia H1.1 (Épica E1, `.ai-studio/memory/backlog.md`). Consume `docs/architecture_stack.md` §1-2 y §7.1 (tooling concreto queda abierto ahí; esta spec lo cierra). Sin código de producto — solo estructura, contratos y configuración como texto. El Programmer implementa contra esto sin decisiones de diseño pendientes.

Estado del repo al momento de escribir esta spec: no existe todavía ningún `package.json`, `tsconfig`, ni carpeta `packages/`/`apps/` — es un setup desde cero.

---

## 0. Resumen de decisiones de tooling (cierra `architecture_stack.md` §7.1)

| Decisión | Elección | Alternativa descartada |
|---|---|---|
| Gestor de workspaces | **npm workspaces** | pnpm |
| Test runner | **Vitest** | Jest |
| Linter | **ESLint 9 (flat config)** + `typescript-eslint` + `eslint-plugin-boundaries` | reglas `no-restricted-imports` manuales por paquete |
| Node | **≥ 20 LTS, recomendado 22 LTS** (pinnear en `.nvmrc`) | — |
| TypeScript | **5.6.x o superior** (última estable 5.x al momento de instalar) | — |
| Enforcement de dirección de dependencia | 3 capas: (a) `package.json` deps, (b) TS project references + `paths`, (c) ESLint boundaries | una sola capa |

### 0.1 Por qué npm workspaces (no pnpm)

- El proyecto es *just for fun* (`vision.md`), sin CI/CD todavía y sin necesidad de instalaciones ultrarrápidas ni de un `node_modules` estrictamente aislado por paquete (la ventaja principal de pnpm).
- npm workspaces viene **incluido con Node** desde npm 7 — cero herramientas adicionales que instalar, actualizar o que un colaborador nuevo tenga que aprender antes de poder correr el proyecto. Menor fricción de onboarding, coherente con la ausencia de infraestructura compleja hoy.
- Si el proyecto crece y el tiempo de instalación/CI se vuelve un problema real, migrar a pnpm más adelante es un cambio de tooling aislado (no toca `domain` ni contratos) — se puede diferir sin coste hoy.

### 0.2 Por qué Vitest (no Jest)

- El monorepo es TypeScript puro en `packages/domain/*`, sin DOM (criterio explícito de `architecture_stack.md` §2.4: "cero renderizado, cero Phaser, cero DOM"). Vitest corre en modo `environment: 'node'` de forma nativa y rápida (transform vía esbuild), sin necesitar `ts-jest`/Babel ni configuración adicional de transformadores para TS/ESM.
- Jest requiere configuración extra no trivial para TS + ESM (`ts-jest` o `babel-jest`, `moduleNameMapper` para workspaces, flags experimentales de ESM) que añade fricción de mantenimiento sin beneficio aquí.
- Vitest tiene soporte de **workspace nativo** (`vitest.workspace.ts`), coherente con la estructura de múltiples paquetes de este monorepo, y comparte config con un futuro `vite` en `apps/shell`/`packages/combat-scene` (mismo motor, menos piezas distintas que aprender cuando lleguen esas historias).
- API compatible con Jest (`describe`/`it`/`expect`) — no hay curva de aprendizaje adicional para Programmer/QA.

## 2. `package.json` raíz — contrato

```jsonc
{
  "name": "the-collector",
  "private": true,
  "type": "module",
  "engines": { "node": ">=20.0.0", "npm": ">=10.0.0" },
  "workspaces": [
    "packages/domain/*",
    "packages/data"
  ],
  "scripts": {
    "build": "tsc -b",
    "build:clean": "tsc -b --clean",
    "test": "vitest run",
    "test:watch": "vitest",
    "lint": "eslint .",
    "typecheck": "tsc -b --pretty"
  },
  "devDependencies": {
    "typescript": "^5.6.0",
    "vitest": "^2.0.0",
    "eslint": "^9.0.0",
    "typescript-eslint": "^8.0.0",
    "eslint-plugin-boundaries": "^4.0.0"
  }
}
```

Notas de contrato:
- `"workspaces"` **no** incluye `"packages/*"` a secas — `packages/domain` es un directorio contenedor, no un paquete en sí. Cuando lleguen `combat-scene`/`ui-shared`, se añaden como entradas explícitas nuevas (`"packages/combat-scene"`, `"packages/ui-shared"`), y `apps/shell` en su propio array `"apps/*"`. No usar wildcards amplios que puedan capturar paquetes fuera de la capa esperada.
- `"type": "module"` a nivel raíz — todo el monorepo es ESM. Cada paquete hijo hereda esto salvo que declare lo contrario (ninguno debe hacerlo).
- Ninguna dependencia de `react` ni `phaser` en la raíz ni en ningún `package.json` de `packages/domain/*` o `packages/data`. Esto es la primera capa de enforcement: si no está instalado, no se puede importar ni en tiempo de tipos ni en runtime.

## 3. `package.json` por paquete de dominio — contrato

Ejemplo para `packages/domain/shared/package.json` (los otros dos siguen el mismo patrón, cambiando `name`, y añadiendo `dependencies` internas cuando aplique):

```jsonc
{
  "name": "@collector/domain-shared",
  "version": "0.0.0",
  "private": true,
  "type": "module",
  "main": "./dist/index.js",
  "types": "./dist/index.d.ts",
  "exports": {
    ".": { "types": "./dist/index.d.ts", "default": "./dist/index.js" }
  },
  "scripts": {
    "build": "tsc -b"
  }
}
```

`packages/domain/combat/package.json` añade:
```jsonc
"dependencies": { "@collector/domain-shared": "*", "@collector/domain-catalog": "*" }
```
(combat depende de catalog para resolver `CardDefinition`/`AbilityDefinition` por id — ver `architecture_stack.md` §6, arista `catalog --> combatEngine`).

`packages/domain/catalog/package.json` añade:
```jsonc
"dependencies": { "@collector/domain-shared": "*" }
```
(catalog **no** depende de `combat` — la dependencia va en un solo sentido: combat conoce catalog, catalog nunca conoce combat).

`packages/data/package.json` es mínimo, sin `dependencies`, sin `src`, solo:
```jsonc
{ "name": "@collector/data", "version": "0.0.0", "private": true }
```
No participa todavía de `tsc -b` (no tiene código TS) — se le añadirá `tsconfig.json` cuando H1.8 defina el schema de validación de los JSON.

Con npm workspaces, usar `"*"` como rango de versión para dependencias internas es el patrón estándar (no existe protocolo `workspace:` en npm; npm resuelve al paquete local del workspace automáticamente por nombre).

## 4. `tsconfig` — estructura y enforcement de dirección de dependencia

### 4.1 `tsconfig.base.json` (raíz, no se compila directamente)

```jsonc
{
  "compilerOptions": {
    "target": "ES2022",
    "lib": ["ES2022"],
    "module": "ESNext",
    "moduleResolution": "Bundler",
    "resolveJsonModule": true,
    "strict": true,
    "noUncheckedIndexedAccess": true,
    "exactOptionalPropertyTypes": true,
    "esModuleInterop": true,
    "forceConsistentCasingInFileNames": true,
    "skipLibCheck": true,
    "isolatedModules": true,
    "verbatimModuleSyntax": true,
    "declaration": true,
    "declarationMap": true,
    "composite": true,
    "noEmitOnError": true
  }
}
```

Punto clave: **`"lib": ["ES2022"]` sin `"DOM"`** a nivel base. Ningún `tsconfig.json` de `packages/domain/*` debe sobreescribir `lib` para añadir `"DOM"` — eso es lo que impide, a nivel de tipos, usar globals de navegador (`window`, `document`, etc.) dentro de `domain`. Cuando se cree `packages/combat-scene`/`apps/shell` en historias futuras, su `tsconfig.json` sí añade `"DOM"` y `"DOM.Iterable"` — pero eso vive fuera del alcance de H1.1.

### 4.2 `tsconfig.json` raíz (solution file, orquesta el build con project references)

```jsonc
{
  "files": [],
  "references": [
    { "path": "packages/domain/shared" },
    { "path": "packages/domain/catalog" },
    { "path": "packages/domain/combat" }
  ]
}
```

`npm run build` / `npm run typecheck` ejecutan `tsc -b` contra este archivo, que compila los tres paquetes en el orden correcto según sus referencias internas.

### 4.3 `tsconfig.json` por paquete — ejemplo `packages/domain/shared/tsconfig.json`

```jsonc
{
  "extends": "../../../tsconfig.base.json",
  "compilerOptions": {
    "rootDir": "src",
    "outDir": "dist",
    "types": []
  },
  "include": ["src"]
}
```

`packages/domain/catalog/tsconfig.json` (depende de `shared`):

```jsonc
{
  "extends": "../../../tsconfig.base.json",
  "compilerOptions": { "rootDir": "src", "outDir": "dist", "types": [] },
  "include": ["src"],
  "references": [{ "path": "../shared" }]
}
```

`packages/domain/combat/tsconfig.json` (depende de `shared` y `catalog`):

```jsonc
{
  "extends": "../../../tsconfig.base.json",
  "compilerOptions": { "rootDir": "src", "outDir": "dist", "types": [] },
  "include": ["src"],
  "references": [{ "path": "../shared" }, { "path": "../catalog" }]
}
```

**Por qué esto es una segunda capa de enforcement, no solo organización:** con `"composite": true` + project references, `tsc -b` exige que cualquier paquete importado esté declarado explícitamente en `"references"`. Si en el futuro alguien escribe en `domain/shared` un `import ... from '@collector/domain-combat'` (dependencia inversa prohibida), la compilación falla porque `shared` nunca declara una referencia a `combat` — el error aparece en build, no en code review.

### 4.4 Resolución de paquetes en dev (sin build previo)

Para que Vitest pueda ejecutar tests de `domain/combat` importando `@collector/domain-shared` **sin exigir un `npm run build` previo** (loop de desarrollo rápido), cada `vitest.config.ts` de paquete debe usar el plugin `vite-tsconfig-paths` (devDependency raíz) apuntando al `tsconfig.json` del propio paquete, y el `tsconfig.base.json` debe declarar:

```jsonc
// añadido a tsconfig.base.json, sección compilerOptions
"baseUrl": ".",
"paths": {
  "@collector/domain-shared": ["packages/domain/shared/src/index.ts"],
  "@collector/domain-catalog": ["packages/domain/catalog/src/index.ts"],
  "@collector/domain-combat": ["packages/domain/combat/src/index.ts"]
}
```

Esto permite que Vitest resuelva contra el código fuente directamente (rápido, sin `dist/`), mientras que `tsc -b` (build "de verdad") sigue resolviendo vía `package.json`/`exports` + project references. Ambos caminos coexisten sin conflicto porque npm workspaces ya symlinkea los paquetes en `node_modules` por nombre.

## 5. Reglas de ESLint — dirección de dependencia (tercera capa de enforcement)

### 5.1 Elección de plugin

Se usa **`eslint-plugin-boundaries`** en vez de `no-restricted-imports` repetido por paquete, porque permite declarar **una sola vez, en la raíz**, el grafo de capas completo de `architecture_stack.md` §1/§6 (`domain`, `data`, `combat-scene`, `ui-shared`, `shell`) y qué tipo puede importar a qué tipo — igual que el diagrama mermaid, pero ejecutable. Cuando lleguen `combat-scene`/`ui-shared`/`shell` en historias futuras, solo se añaden entradas nuevas a la tabla de tipos, no reglas nuevas dispersas.

### 5.2 Contrato de `eslint.config.mjs` (raíz)

```js
// eslint.config.mjs — forma del contrato, Programmer ajusta sintaxis exacta de la versión instalada
import boundaries from 'eslint-plugin-boundaries';
import tseslint from 'typescript-eslint';

export default tseslint.config(
  ...tseslint.configs.recommended,
  {
    plugins: { boundaries },
    settings: {
      'boundaries/elements': [
        { type: 'domain-shared',  pattern: 'packages/domain/shared/**' },
        { type: 'domain-catalog', pattern: 'packages/domain/catalog/**' },
        { type: 'domain-combat',  pattern: 'packages/domain/combat/**' },
        { type: 'data',           pattern: 'packages/data/**' },
        // entradas futuras (no crear aún en H1.1, dejar documentado el patrón):
        { type: 'combat-scene',   pattern: 'packages/combat-scene/**' },
        { type: 'ui-shared',      pattern: 'packages/ui-shared/**' },
        { type: 'shell',         pattern: 'apps/shell/**' }
      ]
    },
    rules: {
      'boundaries/element-types': ['error', {
        default: 'disallow',
        rules: [
          { from: 'domain-shared',  allow: [] },
          { from: 'domain-catalog', allow: ['domain-shared', 'data'] },
          { from: 'domain-combat',  allow: ['domain-shared', 'domain-catalog'] },
          { from: 'data',           allow: [] },
          { from: 'combat-scene',   allow: ['domain-shared', 'domain-catalog', 'domain-combat'] },
          { from: 'ui-shared',      allow: [] },
          { from: 'shell',          allow: ['domain-shared', 'domain-catalog', 'domain-combat', 'combat-scene', 'ui-shared'] }
        ]
      }],
      // cinturón de seguridad explícito, independiente de boundaries:
      'no-restricted-imports': ['error', {
        paths: [
          { name: 'react', message: 'domain/data no puede depender de react.' },
          { name: 'phaser', message: 'domain/data no puede depender de phaser.' }
        ]
      })
    }
  },
  {
    // override: el cinturón de seguridad de react/phaser solo aplica dentro de domain/data
    files: ['packages/domain/**/*.ts', 'packages/data/**/*.ts'],
    rules: {
      'no-restricted-imports': ['error', { paths: [{ name: 'react' }, { name: 'phaser' }] }]
    }
  }
);
```

Nota: la tabla `rules` de `boundaries/element-types` de arriba ya es válida y completa para el alcance real de H1.1 (`domain-shared`, `domain-catalog`, `domain-combat`, `data`) — las filas de `combat-scene`/`ui-shared`/`shell` se dejan escritas porque no cuesta nada declararlas ahora y evita que Programmer tenga que recordar añadirlas correctamente en historias futuras; como esos paquetes no existen aún, esas filas simplemente no matchean ningún archivo todavía.

## 6. `packages/domain/shared` — contenido mínimo obligatorio

Este paquete es la base que H1.2 (`RandomSource`) y H1.3 (motor de turnos) van a consumir. Debe quedar ya con estos tres módulos (firmas exactas; Programmer implementa el cuerpo):

### 6.1 `src/ids.ts` — branded IDs

```ts
export type Brand<T, B extends string> = T & { readonly __brand: B };

export type CardInstanceId = Brand<string, 'CardInstanceId'>;
export type AbilityId      = Brand<string, 'AbilityId'>;
export type LeaderId       = Brand<string, 'LeaderId'>;
export type EnemyId        = Brand<string, 'EnemyId'>;
export type ScenarioId     = Brand<string, 'ScenarioId'>;

// helper de construcción, evita castear "as" repartido por todo el código
export function createId<B extends string>(brand: B, value: string): Brand<string, B>;
```

Justificación: evita mezclar por error un `AbilityId` con un `CardInstanceId` (ambos serían `string` sin branding), un error de clase entera de bugs que interesa prevenir antes de que H1.3+ empiece a manejar decenas de ids distintos.

### 6.2 `src/result.ts` — `Result<T, E>`

```ts
export type Result<T, E = Error> =
  | { readonly ok: true;  readonly value: T }
  | { readonly ok: false; readonly error: E };

export function ok<T>(value: T): Result<T, never>;
export function err<E>(error: E): Result<never, E>;
export function isOk<T, E>(r: Result<T, E>): r is { ok: true; value: T };
export function isErr<T, E>(r: Result<T, E>): r is { ok: false; error: E };
```

Justificación: `CombatEngine.dispatch()` (`architecture_stack.md` §2.2, `CombatCommandResult`) necesita un tipo de resultado explícito para comandos inválidos (CD bloqueado, Núcleo inexistente, etc.) sin recurrir a excepciones — más fácil de testear de forma exhaustiva en H1.13.

### 6.3 `src/event-bus.ts` — `EventBus<TEvent>` genérico

```ts
export type Unsubscribe = () => void;

export interface EventBus<TEvent> {
  subscribe(listener: (event: TEvent) => void): Unsubscribe;
  emit(event: TEvent): void;
}

export function createEventBus<TEvent>(): EventBus<TEvent>;
```

Justificación: `CombatEngine.subscribe()` y `CombatBridge` (`architecture_stack.md` §2.2-2.3) necesitan un mecanismo pub/sub; en vez de que cada paquete reinvente el suyo, se define una vez en `shared` y tanto `domain/combat` (con `TEvent = CombatEvent`) como futuros paquetes lo reutilizan con su propio tipo de evento.

### 6.4 `src/index.ts`

Re-exporta `ids.ts`, `result.ts`, `event-bus.ts` — es el único punto de entrada público del paquete (`@collector/domain-shared`), consistente con el `"exports"` de su `package.json` (§3).

No se implementa `RandomSource` en H1.1 — es el entregable explícito de H1.2. No crear el archivo todavía para no dejar una interfaz a medias sin su historia.

## 8. Test dummy obligatorio (criterio de aceptación "test dummy pasa")

En vez de un test trivial sin valor (`expect(true).toBe(true)`), el test dummy debe ejercitar el primer entregable real de `domain/shared` — el `EventBus` — para que "tooling funciona" y "el primer contrato de dominio funciona" se validen en el mismo gesto.

Archivo: `packages/domain/shared/src/event-bus.test.ts`

```ts
import { describe, it, expect, vi } from 'vitest';
import { createEventBus } from './event-bus';

describe('EventBus', () => {
  it('entrega eventos emitidos a los listeners suscritos', () => {
    type DummyEvent = { type: 'PING'; value: number };
    const bus = createEventBus<DummyEvent>();
    const listener = vi.fn();

    bus.subscribe(listener);
    bus.emit({ type: 'PING', value: 42 });

    expect(listener).toHaveBeenCalledWith({ type: 'PING', value: 42 });
  });

  it('deja de recibir eventos tras invocar unsubscribe', () => {
    type DummyEvent = { type: 'PING' };
    const bus = createEventBus<DummyEvent>();
    const listener = vi.fn();

    const unsubscribe = bus.subscribe(listener);
    unsubscribe();
    bus.emit({ type: 'PING' });

    expect(listener).not.toHaveBeenCalled();
  });
});
```

Criterio de "hecho" para esta parte: `npm run test` ejecuta ambos casos y ambos pasan en verde, sin levantar DOM ni ningún framework de UI.
