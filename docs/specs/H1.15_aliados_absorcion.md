## 0. Decisiones de diseño que cierra esta spec

### 0.1 "Aliado en mesa" es un concepto nuevo de `domain/combat`, no de `domain/catalog`

`CardDefinition` (catalog) modela contenido estático; `domain/combat` ya tiene precedente de "espejar" un subconjunto de esos datos en su propio tipo, resuelto externamente e inyectado en `CombatEngineConfig` (mismo patrón que `abilityCoreCosts`/`abilityCooldowns`/`abilityEffects`/`contratiempoCards`, H1.3/H1.4/H1.6/H1.14). Se introduce `AllyCardDefinition` (nuevo, `types/ally.ts`) con exactamente los 3 datos que el motor necesita: `energyCost`, `life`, `isBerserker`. El motor nunca importa `domain/catalog` ni parsea keywords — quien construya el mapa (tests hoy, integración de H1.18 mañana) traduce `CardDefinition.cost.energy`/`keywords` a este tipo, igual que ya se hace para Contratiempo (`undoScope` derivado de `DESHACER_DANO`/`DESHACER_TURNO`).

Un Aliado **en mesa** es una instancia concreta (`AllyInPlay`), no la `CardDefinition`. Se identifica con `CardInstanceId` — primer uso real de ese tipo ya declarado en `domain/shared` (ver nota de estado del repo).

### 0.2 Cómo entra un Aliado en mesa: comando `PLAY_ALLY`, mínimo y dedicado — mismo molde que `PLAY_CONTRATIEMPO`

GDD §2.2 punto 4: *"Bajar una carta de mano → gasta Energía. La carta va a mesa o resuelve su efecto."* Esto es una de las 2 acciones del turno del Líder. `domain/combat` todavía no modela mano/mazo genéricos (`PLAY_CARD` genérico sigue siendo H1.18, igual que dejó explícito la spec H1.14 §0.6). Igual que Contratiempo necesitó su propio comando dedicado antes de que `PLAY_CARD` existiera, **esta historia introduce `PLAY_ALLY`**: comando mínimo, exclusivo del Líder, que consume 1 acción + Energía y añade una entrada a `alliesInPlay`. Cuando H1.18 diseñe `PLAY_CARD` genérico, `PLAY_ALLY` puede quedar como caso especial de ese flujo o ser reemplazado — no es una decisión que deba tomar esta historia.

`PLAY_ALLY` **sí consume una acción** (`actionsTakenThisTurn += 1`) — es "bajar una carta de mano", una de las 2 acciones del Líder (GDD §2.2). Esto es distinto de `SET_DAMAGE_REDIRECT` (§0.3), que expresamente NO consume acción.

### 0.3 Redirección de daño: comando declarativo `SET_DAMAGE_REDIRECT`, no un flag dentro del efecto de Ataque del Enemigo

El daño del Enemigo se resuelve de forma síncrona e inmediata dentro de `handleActivateAbility` (side `ENEMY`) — el motor no tiene (ni esta historia introduce) ningún mecanismo de "pausa e input intermedio" a media resolución de un comando. Por tanto, la elección del jugador de "a qué Aliado redirijo" **no puede ser una respuesta en tiempo real al golpe concreto** con la arquitectura actual — tiene que ser una **declaración vigente de antemano** que el motor consulta en el momento de resolver cualquier Ataque entrante, hasta que se cambie.

Se introduce el comando `SET_DAMAGE_REDIRECT { targetAllyInstanceId: CardInstanceId | null }`:
- Fija (o retira, con `null`) qué Aliado en mesa recibe todo el daño de Ataque entrante mientras el valor siga vigente.
- **No consume `actionsTakenThisTurn`, no comprueba `turnOwner`, no gasta Energía ni Núcleo.** Es deliberadamente la interpretación más literal de "sin gastar acción" — y además queda disponible para declararse durante el turno del Enemigo si algún día un harness (H1.19) permite interjecciones a media IA; hoy, en la práctica, se declara durante el turno del Líder como una postura de pie para el/los turnos de Enemigo que vengan.
- Única validación: si `targetAllyInstanceId !== null`, debe existir un `AllyInPlay` con ese id y `life > 0` en este instante (si no, `REDIRECT_TARGET_NOT_FOUND`).
- **No rechaza explícitamente elegir un objetivo mientras haya un Berserker vivo.** El valor se guarda igual, pero es ignorado en tiempo de resolución (§0.4) — más simple que introducir un código de error para una restricción que de todos modos no cambia a dónde va el daño. El evento `DAMAGE_REDIRECT_SET` incluye `forcedByBerserker` informativo para que UI/tests puedan detectarlo sin necesidad de que el comando falle.

### 0.4 Prioridad de resolución del objetivo de un Ataque entrante

Cuando se resuelve un efecto `ATTACK` del Enemigo (`applyAttackEffect`), el motor decide el objetivo **en este orden, siempre**:

1. **Berserker vivo en mesa** (`isBerserker && life > 0`) → todo el golpe va a él, sin excepción (GDD §3.7: "toda la absorción va a él obligatoriamente"). Si hay más de uno (caso no esperado por contenido actual, pero no prohibido por tipos), se elige el primero por orden de entrada en mesa — determinista.
2. Si no hay Berserker vivo: el Aliado fijado por `SET_DAMAGE_REDIRECT` (`activeDamageRedirectTargetId`), **si sigue vivo**. Si el id guardado ya no corresponde a un Aliado vivo (murió, o nunca existió — no debería pasar por validación, pero se tolera de forma defensiva), se limpia (`null`) y se cae al caso 3.
3. Si no hay objetivo: el golpe sigue el camino existente de H1.6 sin cambios — `leaderShield` primero, exceso a `leaderDamage` solo con Arrollar.

**Decisión explícita no cubierta literalmente por el GDD:** cuando el golpe SÍ va a un Aliado (casos 1 o 2), **`leaderShield` no interviene en absoluto** — el Aliado absorbe como si fuera el único objetivo, y el excedente (si Arrollar) pasa directo a `leaderDamage`, saltándose el Escudo. Se elige esta lectura porque el GDD describe la redirección como "el golpe entero va al Aliado en su lugar", no como una absorción en cascada Aliado→Escudo→Líder (que el GDD nunca menciona). Si el Game Designer prefiere una cascada distinta, es un ajuste de una sola función (`resolveDamageTarget`/`applyAttackEffect`) — no bloquea esta historia, y ninguno de los criterios de aceptación depende de la interacción Aliado+Escudo simultánea.

### 0.5 Vida del Aliado: nuevo dato de contenido `VIDA_X` (keyword), NO una fórmula del motor

GDD §3.7: *"Vida del Aliado escala con su coste en Energía."* Esto es una **guía de balance para quien autora contenido**, no una fórrmula que el motor deba computar — igual que Umbral/Defensa/Trama son magnitudes autoradas por keyword+`amount`, no derivadas de otro campo. Se añade `VIDA_X` al vocabulario cerrado de `KeywordId` (booleano→con amount, mismo trato que `DEFENSA_X`/`TRAMA_X`) y una validación cruzada nueva en `schema.ts`, mismo patrón que `CONTRATIEMPO_SCOPE_KEYWORDS` (H1.14 §0.5): **exactamente 1 `VIDA_X` si y solo si `type === 'ALIADO'`**. El motor (`domain/combat`) sigue sin saber nada de keywords — recibe `AllyCardDefinition.life` ya resuelto por quien construye el mapa de configuración, exactamente igual que `energyCost` ya se deriva de `cost.energy` sin que el motor lo calcule.

`isBerserker: boolean` en `AllyCardDefinition` se deriva externamente de la presencia de la keyword `BERSERKER` ya existente — no requiere ningún cambio de vocabulario.

### 0.6 Muerte de un Aliado: nunca se elimina del array, vida queda en 0

Un Aliado que llega a 0 de vida **permanece** en `alliesInPlay` con `life: 0` — nunca se hace `splice`/filtrado. Razones:
1. **Reversibilidad de Contratiempo** (§0.7): si el turno de Enemigo que lo mató se deshace, hace falta poder devolverle la vida exacta que tenía — mucho más simple restaurar un campo que "resucitar" un elemento borrado de un array (reconstruir identidad, posición, etc.).
2. Deja abierta, sin comprometerse a ella aquí, la puerta a que una capa de presentación (Phaser/React, fuera de alcance de Hito 1) anime la muerte o la UI muestre un "cementerio" — decisión de historias futuras.

`resolveDamageTarget`/cualquier lectura de "Aliados vivos" siempre filtra por `life > 0` explícitamente; nunca asume que el array solo contiene vivos.

### 0.7 Integración con Contratiempo (H1.14): el log de reversión debe distinguir objetivo `LEADER` vs `ALLY`

**Esto es obligatorio, no opcional** — GDD §2.7 dice que Contratiempo deshace "efectos del turno enemigo anterior (daño...)"; con Aliados en juego, un Ataque revertido puede haber golpeado a un Aliado, no al Líder. `UndoableEnemyActionLogEntry.effect` (variante `ATTACK`) se extiende con un discriminante `target: 'LEADER' | 'ALLY'`, cada uno con los campos "antes"/"después" que necesita para revertir (§2.2). `handlePlayContratiempo` (§4.6) rama sobre ese discriminante: si `'LEADER'`, comportamiento idéntico a H1.14 (restaura `leaderDamage`/`leaderShield`); si `'ALLY'`, restaura `leaderDamage` (por si hubo derrame de Arrollar) Y la vida del Aliado (`life = allyLifeBefore`, incluso si eso lo "resucita" desde 0 — coherente con §0.6).

**Simplificación documentada, no bloqueante:** si la muerte de ese Aliado había limpiado `activeDamageRedirectTargetId` (§0.4, caso 2), Contratiempo no restaura automáticamente esa postura — el jugador puede volver a declararla gratis con `SET_DAMAGE_REDIRECT` en el mismo turno. No se automatiza porque añade complejidad de estado por un beneficio marginal (un comando extra, gratis, sin restricción).

### 0.8 Confirmación explícita: esta historia NO reabre el bug latente de `handlePlayContratiempo` (nota de deuda técnica del Director)

QA de H1.14 detectó que `handlePlayContratiempo` usa el `leaderDamageBefore` de la ÚLTIMA entrada `ATTACK` del log del turno enemigo (el bucle sobre `revertedEntries` sobrescribe `this.leaderDamage` en cada iteración), y que hoy es inalcanzable porque `ENEMY_BASE_ACTIONS_PER_TURN = 1` y Combo está bloqueado para `side: 'ENEMY'` (`validateAbilityComboConfig` exige `side === 'LEADER'`).

**Verificado explícitamente:** ni `PLAY_ALLY` ni `SET_DAMAGE_REDIRECT` son comandos de `side: 'ENEMY'`, ninguno de los dos llama a `handleActivateAbility` ni empuja nada a `currentEnemyTurnLog` — ese log solo recibe entradas desde el bloque `if (command.side === 'ENEMY')` dentro de `handleActivateAbility`, código que esta historia no toca en su condición de entrada. **`ENEMY_BASE_ACTIONS_PER_TURN` sigue en 1 y `abilityCombo` sigue exigiendo `side: 'LEADER'`** — esta historia no modifica ninguna de las dos constantes/validaciones. Por tanto, el número máximo de entradas `ATTACK` de `side: 'ENEMY'` en `currentEnemyTurnLog` por turno **sigue siendo como mucho 1**, igual que antes de H1.15. El bug del "último vs. primero" sigue siendo código muerto, exactamente igual que lo dejó H1.14 — no se corrige aquí (fuera de alcance) ni se reabre. Se recomienda a Coordinator mantenerlo registrado como bug de baja prioridad en el backlog para cuando H1.16 (Secuaces) sí introduzca una segunda acción de `side: 'ENEMY'` por turno.

### 0.9 Fuera de alcance explícito

- **Activar la habilidad propia de un Aliado** (GDD §2.2 punto 4: "Activar una habilidad de un Aliado en mesa... gasta tu acción + 1 Núcleo") ya es soportado, sin cambios, por `ACTIVATE_ABILITY` existente con `side: 'LEADER'` y `sourceId` = id del Aliado (`turn.ts` ya lo documenta: "Aliados/Secuaces actúan DENTRO del turno de uno de estos dos lados... son `sourceId` distintos"). Esta historia no valida que el `sourceId` enviado corresponda a un Aliado realmente vivo en mesa — ese hueco de validación ya existía antes de H1.15 (nunca se comprobó para ningún `sourceId`) y no es un criterio de aceptación de esta historia. Se deja anotado para que Coordinator lo considere como fast-follow, posiblemente junto con H1.18.
- **Mano/mazo del jugador, robo, Auto-reglas** (GDD §2.2) — H1.18, sin cambios respecto a lo ya declarado fuera de alcance por H1.14.
- **Secuaces del Enemigo** — H1.16, sistema completamente distinto (presencia pasiva, IA propia). No se toca aquí.
- **Interacción exacta Aliado+Escudo simultáneos** — ver nota en §0.4; no bloqueante, ningún AC lo ejercita.

### 4.7 `handlePlayAlly` — método nuevo

```ts
private handlePlayAlly(
  command: Extract<CombatCommand, { type: 'PLAY_ALLY' }>
): CombatCommandResult {
  const def = this.allyCards.get(command.cardId);
  if (!def) {
    return err({ code: 'ALLY_CARD_UNKNOWN', cardId: command.cardId });
  }

  if (this.turnOwner !== 'LEADER') {
    return err({ code: 'NOT_YOUR_TURN', expected: 'LEADER', actual: this.turnOwner });
  }

  if (this.actionsTakenThisTurn >= this.actionsAllowedThisTurn) {
    return err({
      code: 'NO_ACTIONS_REMAINING',
      side: this.turnOwner,
      actionsTaken: this.actionsTakenThisTurn,
      actionsAllowed: this.actionsAllowedThisTurn,
    });
  }

  if (this.leaderEnergy < def.energyCost) {
    return err({
      code: 'ALLY_INSUFFICIENT_ENERGY',
      cardId: command.cardId,
      required: def.energyCost,
      available: this.leaderEnergy,
    });
  }

  // Mutación.
  this.leaderEnergy -= def.energyCost;
  this.actionsTakenThisTurn += 1;

  const instanceId = this.nextCardInstanceId();
  const ally: AllyInPlay = {
    instanceId,
    cardId: command.cardId,
    isBerserker: def.isBerserker,
    maxLife: def.life,
    life: def.life,
  };
  this.alliesInPlay = [...this.alliesInPlay, ally];

  // Calentamiento (GDD §2.5) de las habilidades propias del Aliado, si las tiene.
  for (const abilityId of def.abilityIds ?? []) {
    const cdDef = this.abilityCooldowns.get(abilityId);
    if (cdDef) this.remainingCooldowns.set(abilityId, cdDef.baseCooldown);
  }

  const event: CombatEvent = {
    type: 'ALLY_ENTERED_PLAY',
    cardId: command.cardId,
    sourceId: command.sourceId,
    allyInstanceId: instanceId,
    maxLife: def.life,
    isBerserker: def.isBerserker,
    leaderEnergyAfter: this.leaderEnergy,
  };
  this.eventBus.emit(event);

  return ok([event]);
}

if (raw.type === 'ALIADO') {
  if (lifeKeywords.length !== 1) {
    fail(
      context,
      `type ALIADO exige exactamente 1 keyword VIDA_X, encontradas ${lifeKeywords.length} (GDD §3.7)`
    );
  }
} else if (lifeKeywords.length > 0) {
  fail(
    context,
    `keyword VIDA_X solo es válida en cartas type ALIADO (GDD §3.7), pero esta carta es type ${String(raw.type)}`
  );
}
```
