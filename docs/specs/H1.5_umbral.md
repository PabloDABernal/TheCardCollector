# H1.5 — Umbral (fórmula alimentada por valor de Núcleo 1-4, incluye debuff a 0)

> Spec técnica del Architect para la historia H1.5 (Épica E1, `.ai-studio/memory/backlog.md` líneas 71-83). Consume GDD §2.3, §2.4, §12 (`docs/GDD_The_Collector_v2.md`), decisions.md ("Costes de habilidad solo por color/genérico", "Piso del valor de Núcleo: permitir 0 como debuff extremo"), y el código real de H1.1-H1.4 ya implementado y testeado en `packages/domain/combat/src/` y `packages/domain/shared/src/` (verificado línea a línea al escribir esta spec, no solo las specs anteriores). Mismo nivel de detalle que `docs/specs/H1.4_cooldowns.md`: sin código de producto salvo firmas/pseudocódigo exacto y tests literales.

## 0. Decisiones de diseño de sistema que cierra esta spec

### 0.1 Alcance: capa de funciones puras de resolución de fórmula, sin tocar `CombatEngine`

H1.5 implementa **únicamente** una función pura: dada una fórmula (`UmbralFormula`) y un `NucleoValue`, devuelve el número resultante y si se activó el bonus Umbral. **No** conecta ese resultado a ninguna mutación de estado real (vida del Líder, contador de Trama del Escenario, fichas de Defensa...) — eso pertenece a H1.6 (que introduce `leaderDamage`/`scenarioPlot` en `CombatStateSnapshot`) y a historias de integración posteriores (H1.18).

Justificación explícita, no es una suposición mía: la propia spec de H1.4 ya adelantó esta frontera en su §9 ("Qué queda listo para las historias siguientes"): *"H1.5 (Umbral): completamente ortogonal a esta historia — Umbral opera sobre `nucleoSpent.value` del evento `ABILITY_ACTIVATED`, nunca sobre `AbilityCooldownDefinition`/`remainingCooldowns`. No hay intersección de código."* — es decir, el propio diseño de H1.4 ya asumía que Umbral consume el valor del evento `ABILITY_ACTIVATED` **desde fuera** del motor, no que el motor lo aplique internamente. `CombatEngine` de hoy no tiene ningún punto donde "resolver el efecto de una habilidad" — añadir uno ahora, sin `leaderDamage`/`scenarioPlot` (H1.6) ni `AbilityDefinition` real (H1.8), sería inventar un enganche que se tendría que rehacer en cuanto esas historias lleguen.

**Consecuencia de arquitectura (respuesta al punto 4 del encargo):** H1.5 **no añade ningún `CombatCommand` ni `CombatEvent` nuevo, ni toca `combat-engine.ts`**. Es un módulo nuevo de funciones puras + tipos en `packages/domain/combat/src/`, testeado de forma aislada (le pasas un `NucleoValue` literal en el test, no hace falta levantar un `CombatEngine`). Cuando H1.6/H1.18 necesiten aplicar el resultado a `leaderDamage`/`scenarioPlot`, importarán y llamarán a estas funciones con `event.nucleoSpent.value` (evento `ABILITY_ACTIVATED`, ya existente desde H1.3) como argumento — sin ningún cambio retroactivo aquí.

### 0.2 Cómo se modela "el Núcleo puede valer 0 por un debuff" — sin implementar el debuff en sí

`NucleoValue = number` (H1.3) ya deja la puerta abierta a exactamente esto; **H1.5 no añade ningún mecanismo nuevo para que un Núcleo del pool real llegue a valer 0** — ninguna historia del backlog actual (revisado completo, `H1.1`-`H1.19`) es dueña de "una carta/habilidad/pasivo/escenario fija un Núcleo del pool a 0" (GDD §2.3.1, "Dados Virtuales"); eso queda pendiente de una historia futura de contenido/mecánica que todavía no existe en el backlog.

Lo que sí resuelve H1.5, y es exactamente lo que pide el criterio de aceptación: las funciones de resolución de Umbral (§2-§3) **aceptan cualquier `NucleoValue`, incluido `0`, sin lanzar ni bloquear** — el "0" en los tests de esta historia se construye como un valor literal pasado directamente a la función (no a través del pool real, que en H1.3/H1.4 sigue generando solo `[1,4]`). Esto dejar la función lista para cuando exista esa historia futura, sin inventar ahora cómo se produce el 0 en runtime.

**Nota para el Director (no bloquea esta historia, pero queda pendiente):** no hay ninguna historia en el backlog dedicada a "cómo un debuff fija un Núcleo del pool a 0 en runtime". Cuando se diseñe el contenido que use esa mecánica (probablemente alrededor de H1.9/H1.12 o una historia nueva de la épica de contenido), hará falta una historia explícita para el mecanismo de mutación del pool — no se ha inventado aquí.

### 0.3 Fórmulas exactas que soporta Umbral en esta historia

GDD §12 (tabla de keywords) define, ligadas al valor del Núcleo gastado:

| Keyword GDD | Fórmula | Modelo en esta historia |
|---|---|---|
| **Ataque** | Daño = valor del Núcleo | `{ kind: 'VALUE' }` |
| **Ataque +X** | Daño = Núcleo + X | `{ kind: 'ADD', amount: X }` |
| **Ataque ×X** | Daño = Núcleo × X | `{ kind: 'MULTIPLY', amount: X }` |

La representación (`UmbralFormula`, §2) es **deliberadamente agnóstica de a qué contador del juego alimenta el resultado** (daño al Líder, movimiento de Trama, fichas de Defensa...) — esa asociación ("esta habilidad usa esta fórmula para mutar `leaderDamage`") es un dato de la definición de la habilidad/carta (H1.8 `AbilityDefinition`, fuera de alcance aquí) y de cómo H1.6/H1.18 la apliquen. Esto es intencional y reutilizable: la misma máquina de fórmulas sirve igual si mañana una carta define "Ataque = Núcleo", "Trama = Núcleo +1" o cualquier otro efecto numérico que se quiera alimentar por Umbral — no hace falta un tipo por keyword.

**Explícitamente fuera de alcance (no modelado, no hay tipo/variante para esto):**
- **"Ataque -X / /X"** (GDD §12: *"Modificadores negativos — Capa futura"*): el propio GDD lo marca como no implementado todavía. `UmbralFormula` no tiene variantes `SUBTRACT`/`DIVIDE`. Añadirlas es responsabilidad de la historia que active esa keyword.
- **"Caos (auto-daño)"** (GDD §12: *"Toda habilidad 🟣 inflige auto-daño = valor del Núcleo gastado — Regla de color, no de carta"*): es una regla ligada al **color** del Núcleo gastado (CAOS), no una fórmula de la habilidad en sí, y su aplicación (mutar la vida propia) requiere `leaderDamage` (H1.6). Cuando se implemente, reutilizará `computeUmbralFormulaValue({ kind: 'VALUE' }, nucleoValue)` (§3) sin necesitar ningún tipo nuevo — se deja como nota, no se implementa aquí.
- **"Trama X"** (GDD §12, fila propia de la tabla: *"Mueve el contador del Escenario X pasos [...] Enemigo sube, jugador baja"*): leída literalmente en la tabla de keywords, es un **número fijo de la carta/habilidad**, no una fórmula alimentada por el valor del Núcleo — no aparece descrita con "Núcleo" en su fila, a diferencia de "Ataque"/"Ataque +X"/"Ataque ×X" y de "Caos (auto-daño)", que sí lo mencionan explícitamente. La mención de "Trama X" en la descripción de esta historia en el backlog se lee como un ejemplo de "keyword con parámetro numérico" en general, no como una keyword concreta que consuma Umbral. **Si contenido futuro define una habilidad tipo "Trama = Núcleo" o "Trama = Núcleo + X"**, no hace falta ningún tipo nuevo: usa exactamente la misma `UmbralFormula`/`resolveAbilityUmbral` de esta historia (de nuevo, la representación es agnóstica del contador que alimenta). No se ha encontrado necesidad de escalar esto al Director — se resuelve citando la tabla del GDD con confianza.

### 0.4 Resolviendo una posible lectura errónea del criterio de aceptación: "el Núcleo a 0 resuelve el efecto a 0" no significa "toda fórmula da literalmente 0"

El criterio de aceptación dice: *"Núcleo modificado a 0: [...] su efecto numérico resuelve a 0"*. Leído junto al GDD, esto es literalmente cierto para `VALUE` (`0` → `0`) y para `MULTIPLY` (`0 × X = 0` para cualquier `X`), pero **no** para `ADD` (`0 + X = X`, no `0`) — la propia fórmula del GDD (*"Ataque +X: Daño = Núcleo + X"*) no dice en ningún sitio que `X` desaparezca cuando el Núcleo vale 0, y no hay ninguna cita del GDD que soporte "clampear" el resultado de `ADD` a 0 cuando el Núcleo es 0 (eso reescribiría la fórmula misma, no es un caso especial de Umbral).

**Resolución sin ambigüedad, citando el GDD con confianza (no se escala al Director):** el criterio de aceptación describe el caso general/ilustrativo con la keyword base ("Ataque"), donde en efecto el resultado cae a 0; para `ADD` lo que cae a 0 es exclusivamente **la contribución escalada por el Núcleo**, no el término fijo `X` de la propia carta. La implementación (§3) aplica la aritmética literal de cada fórmula sin ningún caso especial para `nucleoValue === 0` — el "0" nunca se trata como sentinela, es un número como cualquier otro. Los tests de §4 fijan explícitamente ambos comportamientos (MULTIPLY/VALUE caen a 0, ADD no) para que no quede duda de implementación.

### 0.5 Bonus Umbral (≥3): representación y alcance

GDD §12: *"Umbral — Si el Núcleo gastado es ≥3, efecto extra"*. Se modela con dos piezas:

1. **La activación del bonus** (`bonusActivated: boolean`) es universal y no depende de la fórmula: `nucleoValue >= UMBRAL_BONUS_THRESHOLD` (constante = 3), sea cual sea la fórmula base de la habilidad.
2. **El valor numérico del "efecto extra" en sí**, cuando ese efecto extra también se expresa como un número alimentado por el mismo Núcleo (ej. "Umbral: +2 Ataque adicional"), se modela con un **segundo `UmbralFormula` opcional** (`bonusFormula`) en `AbilityUmbralDefinition` (§2) — reutiliza el mismo tipo, no uno nuevo. **Qué mecánica de juego representa ese bonus** (más daño, ficha de escudo, robar carta, etc. — GDD no da más detalle en §12 sobre la naturaleza exacta del "efecto extra" de cada habilidad concreta, es dato de contenido) es responsabilidad de la definición de la habilidad/carta (H1.8/H1.9), completamente fuera de alcance de esta historia. `resolveAbilityUmbral` (§3) se limita a decir "el bonus se activó" y, si la habilidad declaró un `bonusFormula`, cuánto vale numéricamente — nunca decide qué significa ese número.

No se introduce ningún mapa inyectado tipo `abilityCoreCosts`/`abilityCooldowns` en `CombatEngineConfig` para esto: como `CombatEngine` no aplica todavía ningún efecto de habilidad (§0.1), no hay ningún punto de `dispatch` que necesite resolver un `AbilityUmbralDefinition` por `abilityId` en tiempo de ejecución. Los tests y el futuro código de H1.6/H1.18 construyen/resuelven un `AbilityUmbralDefinition` explícito y lo pasan directamente a `resolveAbilityUmbral` — igual que hoy nadie necesita un mapa para invocar `satisfiesCoreCost` (`nucleo-color.ts`, H1.3).

### 0.6 Ninguna otra ambigüedad de negocio detectada

Fuera de la nota del §0.2 (mecanismo de debuff a 0, pendiente de una historia futura no bloqueante) y de la aclaración aritmética del §0.4 (resuelta citando el GDD con confianza), no se ha encontrado ninguna decisión de diseño de negocio que requiera escalarse al Director en esta historia.

## 2. `packages/domain/combat/src/types/umbral.ts` — NUEVO

```ts
import type { NucleoValue } from './nucleo';

/**
 * Umbral del valor de Núcleo gastado a partir del cual se activa el "efecto extra"
 * de la keyword Umbral — GDD §2.3, §12: "Si el Núcleo gastado es ≥3, efecto extra".
 * 50% de frecuencia en el rango base 1-4 (GDD §12, nota de la tabla).
 */
export const UMBRAL_BONUS_THRESHOLD = 3;

/**
 * Fórmula genérica que consume el valor de un Núcleo gastado para producir un efecto
 * numérico (GDD §12: "Ataque", "Ataque +X", "Ataque ×X"). Deliberadamente agnóstica de
 * a qué contador del juego alimenta el resultado (daño al Líder, movimiento de Trama
 * del Escenario, fichas de Defensa...) — esa asociación es responsabilidad de la
 * definición de la habilidad/carta (H1.8, fuera de alcance) y de quien aplique el
 * resultado (H1.6/H1.18, fuera de alcance) — ver spec H1.5 §0.1/§0.3.
 *
 * - 'VALUE'    → resultado = valor del Núcleo, sin modificar (keyword "Ataque" base).
 * - 'ADD'      → resultado = valor del Núcleo + `amount` (keyword "Ataque +X").
 * - 'MULTIPLY' → resultado = valor del Núcleo × `amount` (keyword "Ataque ×X").
 *
 * Explícitamente NO se modelan variantes de resta/división ("Ataque -X / /X", GDD §12:
 * "Modificadores negativos — Capa futura") — fuera de alcance de H1.5, ver spec §0.3.
 */
export type UmbralFormula =
  | { readonly kind: 'VALUE' }
  | { readonly kind: 'ADD'; readonly amount: number }
  | { readonly kind: 'MULTIPLY'; readonly amount: number };

/**
 * Dato de una habilidad/carta relevante para Umbral: su fórmula base y, opcionalmente,
 * una segunda fórmula que solo se resuelve si se activa el bonus Umbral (≥3, GDD §12).
 * `bonusFormula` reutiliza el mismo tipo `UmbralFormula` — no hay un tipo separado para
 * el "efecto extra"; qué representa ese número en el juego (más daño, escudo, etc.) es
 * dato de contenido (H1.8/H1.9), fuera de alcance aquí — ver spec §0.5.
 *
 * Construido explícitamente por quien llama a `resolveAbilityUmbral` (tests, o más
 * adelante H1.6/H1.18 a partir de `AbilityDefinition`) — NO vive en
 * `CombatEngineConfig`: `CombatEngine` no aplica todavía ningún efecto de habilidad
 * (ver spec §0.1/§0.5).
 */
export interface AbilityUmbralDefinition {
  readonly baseFormula: UmbralFormula;
  readonly bonusFormula?: UmbralFormula;
}

/**
 * Resultado de resolver un `AbilityUmbralDefinition` contra un `NucleoValue` concreto
 * (posiblemente 0 — decisions.md "Piso del valor de Núcleo: permitir 0 como debuff
 * extremo"; ver spec §0.2).
 */
export interface AbilityUmbralResolution {
  /** Valor del Núcleo gastado que alimentó esta resolución. Se repite aquí (en vez de
   *  obligar al caller a recordarlo aparte) para que el resultado sea autocontenido en
   *  logs/tests/UI futura. */
  readonly nucleoValue: NucleoValue;
  /** Resultado de aplicar `baseFormula` — siempre presente, sin condición de Umbral. */
  readonly baseResolvedValue: number;
  /** `true` si `nucleoValue >= UMBRAL_BONUS_THRESHOLD` (GDD §12). Nunca depende de la
   *  fórmula ni de si hay `bonusFormula` definida. */
  readonly bonusActivated: boolean;
  /**
   * Resultado de aplicar `bonusFormula`, solo si `bonusActivated` es `true` Y la
   * definición declaró `bonusFormula`. `undefined` en cualquier otro caso (bonus no
   * activado, o activado pero sin `bonusFormula` — habilidad cuyo "efecto extra" no es
   * numérico, ej. "roba una carta").
   */
  readonly bonusResolvedValue?: number;
}
```

## 3. `packages/domain/combat/src/umbral.ts` — NUEVO (lógica pura)

```ts
import type { NucleoValue } from './types/nucleo';
import type { AbilityUmbralDefinition, AbilityUmbralResolution, UmbralFormula } from './types/umbral';
import { UMBRAL_BONUS_THRESHOLD } from './types/umbral';

/**
 * Aplica literalmente la aritmética de `formula` sobre `nucleoValue`. Pura, sin
 * estado, sin ningún caso especial para `nucleoValue === 0` — el 0 se trata como
 * cualquier otro número (ver spec H1.5 §0.4: para 'ADD', 0 + amount = amount, NUNCA se
 * clampea a 0).
 */
export function computeUmbralFormulaValue(formula: UmbralFormula, nucleoValue: NucleoValue): number {
  switch (formula.kind) {
    case 'VALUE':
      return nucleoValue;
    case 'ADD':
      return nucleoValue + formula.amount;
    case 'MULTIPLY':
      return nucleoValue * formula.amount;
  }
}

/**
 * GDD §12: "Si el Núcleo gastado es ≥3, efecto extra". Universal: no depende de la
 * fórmula de la habilidad, solo del valor bruto del Núcleo gastado.
 */
export function isUmbralBonusActive(nucleoValue: NucleoValue): boolean {
  return nucleoValue >= UMBRAL_BONUS_THRESHOLD;
}

/**
 * Resuelve una `AbilityUmbralDefinition` completa (fórmula base + bonus opcional)
 * contra el valor de un Núcleo gastado. Punto de entrada principal de esta historia —
 * ver spec H1.5 §0.1: NO consulta ni muta `CombatEngine`; quien la invoque le pasa el
 * `NucleoValue` (ej. `nucleoSpent.value` de un evento `ABILITY_ACTIVADO`, H1.3)
 * explícitamente.
 */
export function resolveAbilityUmbral(
  definition: AbilityUmbralDefinition,
  nucleoValue: NucleoValue
): AbilityUmbralResolution {
  const bonusActivated = isUmbralBonusActive(nucleoValue);
  return {
    nucleoValue,
    baseResolvedValue: computeUmbralFormulaValue(definition.baseFormula, nucleoValue),
    bonusActivated,
    bonusResolvedValue:
      bonusActivated && definition.bonusFormula !== undefined
        ? computeUmbralFormulaValue(definition.bonusFormula, nucleoValue)
        : undefined,
  };
}
```

**Nota de implementación:** el `switch` de `computeUmbralFormulaValue` no lleva `default` — el union `UmbralFormula` es exhaustivo (mismo patrón que `satisfiesCoreCost` en `nucleo-color.ts` y `dispatch()` en `combat-engine.ts`); si TypeScript se queja de "not all code paths return a value" añadir `default: return assertNever(formula)` no es necesario mientras el union no crezca sin actualizar el switch — seguir el estilo ya usado en el resto del código base (ver `satisfiesCoreCost`, que tampoco lleva `default`).

## 4. `packages/domain/combat/src/umbral.test.ts` — NUEVO

```ts

import { computeUmbralFormulaValue, isUmbralBonusActive, resolveAbilityUmbral } from './umbral';
import { UMBRAL_BONUS_THRESHOLD, type AbilityUmbralDefinition, type UmbralFormula } from './types/umbral';

describe('computeUmbralFormulaValue — keyword "Ataque" (VALUE)', () => {
  it.each([
    [0, 0],
    [1, 1],
    [2, 2],
    [3, 3],
    [4, 4],
  ])('nucleoValue=%i → resultado=%i (Daño = valor del Núcleo, GDD §12)', (nucleoValue, expected) => {
    const formula: UmbralFormula = { kind: 'VALUE' };
    expect(computeUmbralFormulaValue(formula, nucleoValue)).toBe(expected);
  });
});

describe('computeUmbralFormulaValue — keyword "Ataque +X" (ADD, amount=2)', () => {
  it.each([
    [0, 2],
    [1, 3],
    [2, 4],
    [3, 5],
    [4, 6],
  ])('nucleoValue=%i → resultado=%i (Daño = Núcleo + 2, GDD §12)', (nucleoValue, expected) => {

  });

  it('nucleoValue=0 NO colapsa a 0: devuelve exactamente `amount` (spec §0.4 — no hay clamp especial para 0)', () => {
    const formula: UmbralFormula = { kind: 'ADD', amount: 7 };

  });

  it.each([
    [0, 0],
    [1, 3],
    [2, 6],
    [3, 9],
    [4, 12],

  });

    const formula: UmbralFormula = { kind: 'MULTIPLY', amount: 999 };

  });

  it.each([
    [0, false],
    [1, false],
    [2, false],
    [3, true],
    [4, true],
  ])('nucleoValue=%i → bonusActivated=%s', (nucleoValue, expected) => {
    expect(isUmbralBonusActive(nucleoValue)).toBe(expected);
  });

    expect(UMBRAL_BONUS_THRESHOLD).toBe(3);
  });

    [0, 2, false],
    [1, 3, false],
    [2, 4, false],
    [3, 5, true],
    [4, 6, true],

      const result = resolveAbilityUmbral(definition, nucleoValue);
      expect(result).toEqual({
        nucleoValue,
        baseResolvedValue: expectedBase,

describe('resolveAbilityUmbral — habilidad CON bonusFormula ("Ataque = Núcleo; Umbral: +5 adicional")', () => {

    baseFormula: { kind: 'VALUE' },
    bonusFormula: { kind: 'ADD', amount: 5 },

  it.each([

    [1, 1, false, undefined],
    [2, 2, false, undefined],
    [3, 3, true, 8],

    (nucleoValue, expectedBase, expectedBonusActivated, expectedBonusResolvedValue) => {
      const result = resolveAbilityUmbral(definition, nucleoValue);
      expect(result).toEqual({
        nucleoValue,
        baseResolvedValue: expectedBase,
        bonusActivated: expectedBonusActivated,
        bonusResolvedValue: expectedBonusResolvedValue,

describe('resolveAbilityUmbral — criterio de aceptación literal: Núcleo modificado a 0', () => {
  it('con fórmula base "Ataque" (VALUE), el efecto numérico resuelve a 0 y Umbral NO se activa', () => {
    const definition: AbilityUmbralDefinition = {
      baseFormula: { kind: 'VALUE' },
      bonusFormula: { kind: 'ADD', amount: 10 },
    };
    const result = resolveAbilityUmbral(definition, 0);
    expect(result.baseResolvedValue).toBe(0);
    expect(result.bonusActivated).toBe(false);

  });
});

describe('resolveAbilityUmbral — el valor del Núcleo NUNCA bloquea/lanza, ni con valor 1 ni con 0', () => {
  it.each([0, 1, 2, 3, 4])(

    (nucleoValue) => {
      const definition: AbilityUmbralDefinition = { baseFormula: { kind: 'ADD', amount: 1 } };
      expect(() => resolveAbilityUmbral(definition, nucleoValue)).not.toThrow();
    }
  );

  it('Núcleo valor 1: no bloquea la habilidad, solo produce un efecto reducido (no un error/rechazo)', () => {
    const definition: AbilityUmbralDefinition = { baseFormula: { kind: 'MULTIPLY', amount: 4 } };
    const result = resolveAbilityUmbral(definition, 1);
    expect(result.baseResolvedValue).toBe(4); // reducido frente a, ej., nucleoValue=4 → 16
    expect(result.bonusActivated).toBe(false);
```
