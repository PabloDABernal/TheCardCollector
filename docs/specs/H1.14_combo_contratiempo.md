## 0. Decisiones de diseño que cierra esta spec

### 0.1 El conteo de acciones por turno se introduce aquí, por primera vez, como prerrequisito de Combo

GDD §2.1: *"2 acciones por turno, alternos"*; §3.4: *"Una acción por turno"* (Enemigo); §2.6: *"puedes encadenar una 3ª acción"*. Ninguna de estas tres reglas está hoy implementada — `handleActivateAbility` nunca ha comprobado cuántas veces se activó ya en el turno. Se introduce:

- `LEADER_BASE_ACTIONS_PER_TURN = 2`, `ENEMY_BASE_ACTIONS_PER_TURN = 1` (constantes, GDD §2.1/§3.4 — no son dato de contenido variable, igual que `LEADER_SHIELD_MAX`/`UMBRAL_BONUS_THRESHOLD`/`ABILITY_BASE_COOLDOWN_MIN`).
- `COMBO_MAX_BONUS_ACTIONS_PER_TURN = 1` (GDD §2.6 dice "una 3ª acción", singular — no "cadena indefinida"; ver §0.2 sobre el tope).
- Un contador runtime `actionsTakenThisTurn` y un límite runtime `actionsAllowedThisTurn` (inicializado a la base del `turnOwner` actual, ampliable por Combo), ambos reseteados en cada `END_TURN` para el lado que **recibe** el turno.

**Verificado contra la suite de tests existente:** ninguna prueba de `combat-engine.test.ts`/`combat-engine.cooldowns.test.ts`/`combat-engine.damage-plot.test.ts` dispatcha más de 2 `ACTIVATE_ABILITY` de `LEADER` ni más de 1 de `ENEMY` dentro del mismo turno — introducir el límite no rompe ningún test existente salvo el migrado explícitamente en §5.1.

### 0.2 Combo: modelado a nivel de "acción genérica", no atado a `ACTIVATE_ABILITY` en exclusiva

GDD §2.6 dice *"Algunas **habilidades/cartas** tienen la keyword Combo"* — no solo cartas. Hoy el único tipo de acción ejecutable por el motor es `ACTIVATE_ABILITY` (activar habilidad de Líder/Aliado/Enemigo); `PLAY_CARD` genérico (para las 2 cartas Evento con `COMBO` ya en catálogo) es H1.18. Se diseña el mecanismo para que sea trivialmente reutilizable cuando `PLAY_CARD` exista:

- `CombatEngineConfig.abilityCombo: ReadonlySet<AbilityId>` (nuevo, opcional, default vacío) — igual patrón "resuelto externamente" que `abilityCoreCosts`/`abilityCooldowns`/`abilityEffects`. Cualquier `abilityId` en este set genera Combo al activarse con éxito.
- **Restricción de contenido validada en el constructor:** toda entrada de `abilityCombo` debe pertenecer a una habilidad `side: 'LEADER'`. Justificación: el Enemigo tiene una única acción fija por turno (GDD §3.4, sin excepción textual) — no existe un "3er turno" de Enemigo que Combo pueda habilitar. Si contenido futuro quisiera dar Combo a una habilidad de Enemigo, sería una historia de diseño explícita (fuera de alcance aquí), no una feature silenciosa del motor.
- **Tope de bonus:** `actionsAllowedThisTurn` solo se incrementa **una vez** por turno (`comboBonusGrantedThisTurn`), aunque se activen varias habilidades con `COMBO` en la misma cadena. GDD §2.6 nunca describe un encadenamiento de 4ª/5ª acción — "permite una 3ª acción" se lee como techo, no como incremento acumulable.
- Cuando `PLAY_CARD` exista (H1.18), reutiliza el mismo contador `actionsTakenThisTurn`/`actionsAllowedThisTurn` y necesitará su propio `cardCombo: ReadonlySet<CardId>` análogo a `abilityCombo` — no se diseña aquí para no invadir alcance de H1.18, pero la estructura (contador de acciones desacoplado de qué comando concreto las genera) ya lo permite sin refactor.

### 0.3 "No repetir la misma habilidad en la cadena de Combo": ya garantizado por CD, pero se hace explícito

Dato clave ya presente en el motor: `ABILITY_BASE_COOLDOWN_MIN = 1` (H1.4) — **toda** habilidad, sin excepción, entra en cooldown ≥1 al activarse, y el cooldown solo baja al empezar un turno propio (nunca dentro del mismo turno). Consecuencia: **es estructuralmente imposible** activar la misma `abilityId` dos veces dentro del mismo turno hoy — el segundo intento siempre choca con `ABILITY_ON_COOLDOWN` antes de llegar a ningún código nuevo de esta historia.

Por tanto, "Tests muestran repetición de habilidad en Combo rechazada" (criterio de aceptación) ya se cumpliría gratis. Aun así, se añade una validación explícita y **anterior** en el orden (`ABILITY_ALREADY_ACTIVATED_THIS_TURN`, ver §3.4) por dos razones:
1. Da un código de error semánticamente correcto ("ya usada esta cadena/turno") en vez de uno incidental ("en cooldown") — mejor para tests/UI/logs.
2. Documenta la regla del GDD como invariante de primera clase del motor, no como coincidencia de otro sistema.

Esto obliga a migrar 1 test existente (§5.1) cuyo código de error esperado cambia de `ABILITY_ON_COOLDOWN` a `ABILITY_ALREADY_ACTIVATED_THIS_TURN` por el nuevo orden de validación.

### 0.4 Mecanismo de "deshacer turno enemigo": log de efectos reversibles, NO snapshot completo

Dos mecanismos posibles, evaluados explícitamente:

- **(A) Snapshot completo** de `CombatStateSnapshot` antes de cada turno de Enemigo, revertido por completo cuando se juega Contratiempo.
- **(B) Log de efectos reversibles**: cada mutación que ocurre durante el turno del Enemigo se registra con su valor "antes"/"después"; Contratiempo aplica el inverso solo de las entradas relevantes a su alcance.

**Se elige (B).** Razones, todas ancladas en el propio texto de GDD §2.7 y en el código ya existente:

1. **"Alcance según la carta: algunas revierten solo el daño, otras la carta de Dramaturgia entera."** Un snapshot es todo-o-nada por construcción — revertir *selectivamente* solo el daño de un turno que también movió Trama exigiría, de todos modos, diferenciar categorías de cambio dentro del snapshot, es decir, reinventar un log por categorías por dentro de la "solución" snapshot. (B) resuelve esto de forma nativa: cada entrada del log ya sabe si es `ATTACK` o `PLOT`.
2. **"No restaura el pool de Núcleos"** — con (A) hay que recordar excluir explícitamente `nucleoPool` del revertido (fácil de olvidar, fácil de romper en refactors futuros). Con (B), el pool de Núcleos simplemente **nunca se registra** en el log — la exclusión es estructural, no una excepción que alguien deba acordarse de aplicar.
3. **Coste marginal ~0**: `applyAttackEffect`/`applyPlotEffect` (H1.6) ya calculan y devuelven exactamente los valores "antes"/"después" que necesita el log, dentro de los eventos `LEADER_DAMAGED`/`SCENARIO_PLOT_CHANGED`. No hace falta clonar todo el estado del motor en cada turno de Enemigo (que además crecería con cada historia futura — Aliados, Secuaces, Energía... H1.15/H1.16/H1.18) — solo con capturar lo que ya se computa.
4. Precedente ya sentado en el propio proyecto: H1.6 diseñó los eventos de efecto para ser "autocontenidos" (rawAmount/absorbedByShield/appliedDamage/leaderDamageAfter, etc. — ver comentarios de esa spec), exactamente la propiedad que un log de reversión necesita.

**Ventana de vigencia ("1 turno atrás"):** se mantienen dos buffers, `currentEnemyTurnLog` (acumula durante el turno activo del Enemigo) y `previousEnemyTurnLog` (congelado = lo jugable por Contratiempo). La transición ocurre en `handleEndTurn` (§4.2): al pasar de ENEMY→LEADER se congela `currentEnemyTurnLog` en `previousEnemyTurnLog`; al pasar de LEADER→ENEMY se **invalida** `previousEnemyTurnLog` (la ventana de 1 turno ya se cerró) y se abre un `currentEnemyTurnLog` limpio. Esto modela "1 turno atrás" con exactitud sin fechar/timestampar nada.

**Consumo de un solo uso:** tras aplicar un Contratiempo, `previousEnemyTurnLog` se vacía inmediatamente (GDD: *"Lo cancelado se descarta"*) — un segundo Contratiempo en el mismo turno de Líder no puede revertir el mismo turno de Enemigo dos veces.

### 0.5 Contenido: las 4 `CardDefinition` CONTRATIEMPO existentes necesitan un keyword nuevo — SÍ, hace falta un campo nuevo

Revisadas las 4 cartas (`packages/data/cards/common-cards.json` ×2, `soldado-base-cards.json` ×1, `mago-base-cards.json` ×1): ninguna tiene ninguna keyword relacionada con "deshacer" — llevan `DEFENSA_X`, `TRAMA_X`, `ATAQUE_MAS_X`, que son keywords de efecto normal (igual que cualquier Evento), no de Contratiempo. Son datos de **antes** de que este mecanismo existiera (H1.9/H1.12 son historias de contenido de placeholder, anteriores a esta).

**Se añaden 2 `KeywordId` nuevos** (booleanos, sin `amount`, mismo patrón que `ARROLLAR`/`DEFENSOR`/`BERSERKER`):
- `DESHACER_DANO` → alcance `'DAMAGE_ONLY'` (solo revierte daño).
- `DESHACER_TURNO` → alcance `'FULL_TURN'` (revierte la carta de Dramaturgia entera: daño, Trama y CD de la habilidad usada).

**Regla de validación nueva en `CardDefinition`:** exactamente una de las dos debe estar presente **si y solo si** `type === 'CONTRATIEMPO'` (nunca en otro tipo de carta). Ver §6.2.

**Migración de datos** (§7): las 4 cartas existentes cambian sus keywords para llevar exactamente una de las dos nuevas, sustituyendo la keyword antigua no relacionada. No es un rebalanceo de números — es corregir un contrato de datos que nunca reflejó el mecanismo real. Balance fino de qué carta es `DAMAGE_ONLY` vs `FULL_TURN` es una decisión de diseño de contenido menor, justificada en §7, no bloqueante para Game Designer/Director si quisieran ajustarla después.

### 0.6 Fuera de alcance explícito de esta historia

- **`PLAY_CARD` genérico** (bajar cualquier carta de mano: Evento/Equipo/Aliado) sigue siendo H1.18. Esta historia solo añade `PLAY_CONTRATIEMPO`, un comando dedicado y mínimo (cardId + coste de Energía + revertido), porque Contratiempo es un criterio de aceptación explícito de H1.14 y su semántica (deshacer, no "resolver un efecto normal") es distinta de cualquier cosa que `PLAY_CARD` fuera a hacer. Cuando H1.18 diseñe `PLAY_CARD`, Contratiempo seguirá teniendo su propio comando (GDD ya lo trata como un "counter diferido", no un Evento normal).
- **Mano del jugador, robo, límite de 10 cartas, Auto-reglas** (GDD §2.2) — H1.18.
- **Excepción letal de Contratiempo** (GDD §2.7: *"si el golpe iba a ser letal, el Contratiempo salta en tiempo real"*) — requiere una condición de derrota (`leaderDamage` vs. vida máxima del Líder), que no existe hasta H1.18 (`LeaderDefinition.maxHealth` ni siquiera se ha definido para el Líder — solo Enemigo tiene `maxHealth`, H1.10). El `PLAY_CONTRATIEMPO` diseñado aquí cubre el camino **proactivo** (jugarlo como tu acción en tu turno), que es el caso primario descrito en GDD; el camino reactivo/interrupción quedará como ampliación de H1.18 sobre la misma infraestructura de log — no bloquea esta historia.
- **Secuaces (H1.16)** — el log de turno de Enemigo se diseña para 0-N entradas por turno (no asume "máximo 1"), así que ya es compatible si H1.16 añade una acción de Secuaz al turno de Enemigo — pero `ENEMY_BASE_ACTIONS_PER_TURN` sigue en 1 porque H1.16 especifica "solo 1 actúa por turno enemigo" (Enemigo o Secuaz, nunca ambos) — no cambia el total.

### 2.1 `packages/domain/combat/src/types/action.ts`

```ts
import type { CombatSide } from './turn';

/** GDD §2.1: "2 acciones por turno" del Líder. Constante de regla, no de contenido —
 *  mismo estatus que LEADER_SHIELD_MAX/UMBRAL_BONUS_THRESHOLD/ABILITY_BASE_COOLDOWN_MIN. */
export const LEADER_BASE_ACTIONS_PER_TURN = 2;

/** GDD §3.4: "Una acción por turno" del Enemigo. */
export const ENEMY_BASE_ACTIONS_PER_TURN = 1;

/** GDD §2.6: Combo "permite una 3ª acción" — techo de +1, no acumulable más allá en el
 *  mismo turno aunque se activen varias habilidades con keyword Combo (ver spec §0.2). */
export const COMBO_MAX_BONUS_ACTIONS_PER_TURN = 1;

export function baseActionsForSide(side: CombatSide): number {
  return side === 'LEADER' ? LEADER_BASE_ACTIONS_PER_TURN : ENEMY_BASE_ACTIONS_PER_TURN;
}

/** Snapshot de solo lectura del conteo de acciones del `turnOwner` ACTUAL — se resetea
 *  en cada END_TURN para el lado que recibe el turno (ver combat-engine.ts, handleEndTurn). */
export interface ActionsStateSnapshot {
  readonly side: CombatSide;
  readonly actionsTaken: number;
  readonly actionsAllowed: number;
  /** `true` si ya se concedió el bonus de Combo este turno (tope, ver COMBO_MAX_BONUS_ACTIONS_PER_TURN). */
  readonly comboBonusGranted: boolean;
}
```

```ts
        readonly leaderDamageBefore: number;
        readonly leaderDamageAfter: number;
        readonly leaderShieldBefore: number;
        readonly leaderShieldAfter: number;

    | {
        readonly kind: 'PLOT';
        readonly scenarioPlotBefore: number;
        readonly scenarioPlotAfter: number;
      };
}
```

### 3.4 Orden de validación final de `handleActivateAbility` (con las 2 inserciones nuevas)

1. `ABILITY_COST_UNKNOWN`
2. `NOT_YOUR_TURN`
3. **`NO_ACTIONS_REMAINING`** (NUEVO — ¿queda alguna acción este turno?)
4. **`ABILITY_ALREADY_ACTIVADA_THIS_TURN`** (NUEVO — ¿ya se usó esta habilidad este turno? ver §0.3)
5. `ABILITY_ON_COOLDOWN`
6. `NUCLEO_NOT_FOUND`
7. `NUCLEO_COLOR_MISMATCH`
8. Mutación (consumir Núcleo, resetear CD, incrementar contador de acciones, registrar en log si `side === 'ENEMY'`, aplicar efecto si existe, evaluar Combo, emitir eventos).

**Por qué `NO_ACTIONS_REMAINING` y `ABILITY_ALREADY_ACTIVATED_THIS_TURN` van antes de `ABILITY_ON_COOLDOWN`:** ambas son propiedades de "el turno actual en su conjunto" (cuántas acciones lleva, qué habilidades ya usó), no de la habilidad concreta aisladamente — encajan mejor junto a `NOT_YOUR_TURN` (otra pregunta sobre "¿tiene sentido este comando ahora?") que junto a los checks de Núcleo, que sí dependen del `nucleoInstanceId` concreto enviado.

Orden de validación de `handlePlayContratiempo` (nuevo comando, sin precedente que migrar):

1. `CONTRATIEMPO_CARD_UNKNOWN`
2. `NOT_YOUR_TURN` (`expected: 'LEADER'`, `actual: turnOwner` si no es Líder)
3. `NO_ACTIONS_REMAINING`
4. `CONTRATIEMPO_INSUFFICIENT_ENERGY`
5. `CONTRATIEMPO_NOTHING_TO_UNDO`
6. Mutación (gastar Energía, revertir según `undoScope`, invalidar la ventana, incrementar contador de acciones, emitir `CONTRATIEMPO_PLAYED`).

```ts
private handleActivateAbility(
  command: Extract<CombatCommand, { type: 'ACTIVATE_ABILITY' }>
): CombatCommandResult {
  const requirement = this.abilityCoreCosts.get(command.abilityId);
  if (!requirement) {
    return err({ code: 'ABILITY_COST_UNKNOWN', abilityId: command.abilityId });
  }

  if (command.side !== this.turnOwner) {
    return err({ code: 'NOT_YOUR_TURN', expected: this.turnOwner, actual: command.side });
  }

  // NUEVO H1.14 — ver spec §0.1/§3.4.
  if (this.actionsTakenThisTurn >= this.actionsAllowedThisTurn) {
    return err({
      code: 'NO_ACTIONS_REMAINING',
      side: this.turnOwner,
      actionsTaken: this.actionsTakenThisTurn,
      actionsAllowed: this.actionsAllowedThisTurn,
    });
  }

  // NUEVO H1.14 — ver spec §0.3.
  if (this.abilityIdsActivatedThisTurn.has(command.abilityId)) {
    return err({ code: 'ABILITY_ALREADY_ACTIVATED_THIS_TURN', abilityId: command.abilityId });
  }

  const remaining = this.remainingCooldowns.get(command.abilityId) ?? 0;
  if (remaining > 0) {
    return err({ code: 'ABILITY_ON_COOLDOWN', abilityId: command.abilityId, remaining });
  }

  const index = this.nucleoPool.findIndex((n) => n.id === command.nucleoInstanceId);
  if (index === -1) {
    return err({ code: 'NUCLEO_NOT_FOUND', nucleoInstanceId: command.nucleoInstanceId });
  }

  const nucleo = this.nucleoPool[index] as NucleoInstance;
  if (!satisfiesCoreCost(requirement, nucleo.color)) {
    return err({
      code: 'NUCLEO_COLOR_MISMATCH',
      nucleoInstanceId: command.nucleoInstanceId,
      requirement,
      actualColor: nucleo.color,
    });
  }

  // Solo a partir de aquí se muta estado.
  this.nucleoPool = [...this.nucleoPool.slice(0, index), ...this.nucleoPool.slice(index + 1)];

  const def = this.abilityCooldowns.get(command.abilityId) as AbilityCooldownDefinition;
  const cooldownBeforeThisActivation = this.remainingCooldowns.get(command.abilityId) ?? 0; // NUEVO H1.14
  this.remainingCooldowns.set(command.abilityId, def.baseCooldown);

  // NUEVO H1.14
  this.actionsTakenThisTurn += 1;
  this.abilityIdsActivatedThisTurn.add(command.abilityId);

  const events: CombatEvent[] = [];

  const activated: CombatEvent = {
    type: 'ABILITY_ACTIVATED',
    abilityId: command.abilityId,
    sourceId: command.sourceId,
    side: command.side,
    nucleoSpent: nucleo,
  };
  events.push(activated);
  this.eventBus.emit(activated);

  // NUEVO H1.14 — captura "antes" para el log de reversión de Contratiempo, SOLO si
  // side === 'ENEMY' (ver spec §0.4).
  let effectLogEntry: UndoableEnemyActionLogEntry['effect'] | undefined;

  const effectDef = this.abilityEffects.get(command.abilityId);
  if (effectDef) {
    if (effectDef.kind === 'ATTACK') {
      const leaderDamageBefore = this.leaderDamage;
      const leaderShieldBefore = this.leaderShield;
      const effectEvent = this.applyAttackEffect(command, effectDef, nucleo);
      events.push(effectEvent);
      this.eventBus.emit(effectEvent);
      effectLogEntry = {
        kind: 'ATTACK',
        leaderDamageBefore,
        leaderDamageAfter: this.leaderDamage,
        leaderShieldBefore,
        leaderShieldAfter: this.leaderShield,
      };
    } else {
      const scenarioPlotBefore = this.scenarioPlot;
      const effectEvent = this.applyPlotEffect(command, effectDef);
      events.push(effectEvent);
      this.eventBus.emit(effectEvent);
      effectLogEntry = { kind: 'PLOT', scenarioPlotBefore, scenarioPlotAfter: this.scenarioPlot };
    }
  }

  // NUEVO H1.14
  if (command.side === 'ENEMY') {
    this.currentEnemyTurnLog.push({
      abilityId: command.abilityId,
      sourceId: command.sourceId,
      cooldownBefore: cooldownBeforeThisActivation,
      ...(effectLogEntry ? { effect: effectLogEntry } : {}),
    });
  }

  // NUEVO H1.14 — Combo (GDD §2.6, ver spec §0.2). Se evalúa DESPUÉS del efecto para que
  // COMBO_TRIGGERED sea el último evento de esta activación.
  if (this.abilityCombo.has(command.abilityId) && !this.comboBonusGrantedThisTurn) {
    this.comboBonusGrantedThisTurn = true;
    this.actionsAllowedThisTurn += COMBO_MAX_BONUS_ACTIONS_PER_TURN;
    const comboEvent: CombatEvent = {
      type: 'COMBO_TRIGGERED',
      abilityId: command.abilityId,
      side: command.side,
      sourceId: command.sourceId,
      actionsAllowedThisTurn: this.actionsAllowedThisTurn,
    };
    events.push(comboEvent);
    this.eventBus.emit(comboEvent);
  }

  if (this.nucleoPool.length === 0) {
    this.nucleoPool = this.rollNewPool();
    const refilled: CombatEvent = {
      type: 'NUCLEO_POOL_ROLLED',
      pool: [...this.nucleoPool],
      priorityTurnOwner: this.turnOwner,
    };
    events.push(refilled);
    this.eventBus.emit(refilled);
  }

  return ok(events);
}
```

```ts
private handleEndTurn(): CombatCommandResult {
  const previousTurnOwner = this.turnOwner;
  this.turnOwner = previousTurnOwner === 'LEADER' ? 'ENEMY' : 'LEADER';
  this.turnNumber += 1;

  const events: CombatEvent[] = [];

  const turnEnded: CombatEvent = {
    type: 'TURN_ENDED',
    previousTurnOwner,
    nextTurnOwner: this.turnOwner,
    turnNumber: this.turnNumber,
  };
  events.push(turnEnded);
  this.eventBus.emit(turnEnded);

  this.tickCooldownsForSide(this.turnOwner);
  const cooldownsTicked: CombatEvent = {
    type: 'COOLDOWNS_TICKED',
    side: this.turnOwner,
    cooldowns: this.buildCooldownSnapshot(this.turnOwner),
  };
  events.push(cooldownsTicked);
  this.eventBus.emit(cooldownsTicked);

  // NUEVO H1.14 — reinicia el conteo de acciones para el lado que RECIBE el turno
  // (GDD §2.1/§3.4): el bonus de Combo NUNCA persiste entre turnos (GDD §2.6: "este turno").
  this.actionsTakenThisTurn = 0;
  this.actionsAllowedThisTurn = baseActionsForSide(this.turnOwner);
  this.comboBonusGrantedThisTurn = false;
  this.abilityIdsActivatedThisTurn = new Set();

  // NUEVO H1.14 — ventana de Contratiempo (GDD §2.7, "1 turno atrás"; ver spec §0.4).
  if (previousTurnOwner === 'LEADER') {
    // El Líder ya tuvo su oportunidad de jugar Contratiempo sobre el turno de Enemigo
    // anterior; esa ventana se cierra ahora, sin importar si se usó o no.
    this.previousEnemyTurnLog = [];
    this.currentEnemyTurnLog = [];
  } else {
    // previousTurnOwner === 'ENEMY': se congela lo acumulado durante SU turno como la
    // ventana disponible para el próximo turno de Líder.
    this.previousEnemyTurnLog = this.currentEnemyTurnLog;
    this.currentEnemyTurnLog = [];
  }

  return ok(events);
}
```

### 4.5 `handlePlayContratiempo` — método nuevo

```ts
private handlePlayContratiempo(
  command: Extract<CombatCommand, { type: 'PLAY_CONTRATIEMPO' }>
): CombatCommandResult {
  const def = this.contratiempoCards.get(command.cardId);
  if (!def) {
    return err({ code: 'CONTRATIEMPO_CARD_UNKNOWN', cardId: command.cardId });
  }

  // Contratiempo es exclusivo del Líder (GDD §2.7) — ver comentario en types/commands.ts.
  if (this.turnOwner !== 'LEADER') {
    return err({ code: 'NOT_YOUR_TURN', expected: 'LEADER', actual: this.turnOwner });
  }

  if (this.actionsTakenThisTurn >= this.actionsAllowedThisTurn) {
    return err({
      code: 'NO_ACTIONS_REMAINING',
      side: this.turnOwner,
      actionsTaken: this.actionsTakenThisTurn,
      actionsAllowed: this.actionsAllowedThisTurn,
    });
  }

  if (this.leaderEnergy < def.energyCost) {
    return err({
      code: 'CONTRATIEMPO_INSUFFICIENT_ENERGY',
      cardId: command.cardId,
      required: def.energyCost,
      available: this.leaderEnergy,
    });
  }

  if (this.previousEnemyTurnLog.length === 0) {
    return err({ code: 'CONTRATIEMPO_NOTHING_TO_UNDO', cardId: command.cardId });
  }

  // Mutación.
  this.leaderEnergy -= def.energyCost;
  this.actionsTakenThisTurn += 1;

  const revertedEntries = [...this.previousEnemyTurnLog];
  for (const entry of revertedEntries) {
    if (def.undoScope === 'FULL_TURN') {
      this.remainingCooldowns.set(entry.abilityId, entry.cooldownBefore);
    }
    if (!entry.effect) continue;
    if (entry.effect.kind === 'ATTACK') {
      // El daño SIEMPRE se revierte, en ambos alcances — GDD: "algunas revierten solo
      // el daño" ya implica que el daño se revierte también en FULL_TURN.
      this.leaderDamage = entry.effect.leaderDamageBefore;
      this.leaderShield = entry.effect.leaderShieldBefore;
    } else if (def.undoScope === 'FULL_TURN') {
      // Trama SOLO se revierte en alcance FULL_TURN — DAMAGE_ONLY nunca la toca.
      this.scenarioPlot = entry.effect.scenarioPlotBefore;
    }
  }

  // GDD §2.7: "Lo cancelado se descarta" — consumo de un solo uso, ver spec §0.4.
  this.previousEnemyTurnLog = [];

  const playedEvent: CombatEvent = {
    type: 'CONTRATIEMPO_PLAYED',
    cardId: command.cardId,
    sourceId: command.sourceId,
    undoScope: def.undoScope,
    energySpent: def.energyCost,
    leaderEnergyAfter: this.leaderEnergy,
    revertedEntries,
    leaderDamageAfter: this.leaderDamage,
    leaderShieldAfter: this.leaderShield,
    scenarioPlotAfter: this.scenarioPlot,
  };
  const events = [playedEvent];
  this.eventBus.emit(playedEvent);

  return ok(events);
}
```

- **Solo el Líder puede jugarlo:** con `initialTurnOwner: 'ENEMY'`, `PLAY_CONTRATIEMPO` rechazado con `NOT_YOUR_TURN` (`expected: 'LEADER'`).
- **Consume una acción:** jugar un Contratiempo cuando `actionsTakenThisTurn === actionsAllowedThisTurn` es rechazado con `NO_ACTIONS_REMAINING`.
- **Constructor:** lanza si `contratiempoCards` tiene `energyCost` negativo o no entero; lanza si `initialLeaderEnergy` está fuera de `[0, 5]`.
