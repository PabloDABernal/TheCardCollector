## 2. Diseño de la interfaz — decisión explícita

### 2.1 Usos futuros que la interfaz debe cubrir (contexto, no se implementan aquí)

- **Sorteo de pool de Núcleos** (`domain/combat`, H1.3+): tirar N dados con valor 1-4 (y color, si el color se modela como enum aparte) — necesita un entero acotado repetible.
- **Selección de secuaces** (`domain/combat`): elegir 1 elemento aleatorio de una lista de secuaces elegibles.
- **IA del Enemigo** (`domain/combat`): elegir una acción de una lista de acciones priorizadas/empatadas, o resolver un desempate aleatorio.
- Cualquier uso futuro de aleatoriedad en `domain/run` (p. ej. qué carta ofrecer en el pool de evolución si se decide aleatorizar en vez de determinista por catálogo).

Ninguno de estos casos se implementa en H1.2 — son consumidores futuros que justifican la forma de la interfaz.

### 2.2 Decisión: `next()` + helpers derivados como parte de la misma interfaz (no solo un float suelto)

**Se decide que `RandomSource` expone 3 métodos**, no solo `next()`:

```ts
export interface RandomSource {
  /** Float en [0, 1), igual contrato que Math.random(). Primitiva base. */
  next(): number;

  /** Entero en [minInclusive, maxExclusive). Lanza si maxExclusive <= minInclusive. */
  nextInt(minInclusive: number, maxExclusive: number): number;

  /** Elige un elemento uniformemente al azar de un array no vacío. Lanza si items.length === 0. */
  pick<T>(items: readonly T[]): T;
}
```

**Por qué no dejar solo `next()` y que cada consumidor derive `nextInt`/`pick` por su cuenta:**

- El motor de combate (H1.3+) va a necesitar `nextInt`/`pick` en varios puntos distintos (sorteo de Núcleos, selección de secuaces, IA del enemigo, futuros desempates). Si cada punto de consumo deriva su propio entero a partir de `next()` (p. ej. `Math.floor(rng.next() * n)`), el riesgo de sesgo/bug de redondeo se repite N veces en vez de una, y un test de "el motor de combate tira Núcleos uniformemente" tendría que volver a probar la derivación matemática en vez de confiar en una primitiva ya testeada una sola vez en `shared`.
- Mantener la superficie en `shared` (no en `combat`) es coherente con el principio ya usado para `EventBus`/`Result` en H1.1: una sola definición reutilizada por todos los paquetes de dominio, en vez de reimplementada por cada uno.
- El coste de tener 3 métodos en vez de 1 es mínimo (ambas implementaciones concretas comparten la derivación vía una clase base, ver §2.3) y elimina una clase entera de bugs de "off-by-one" en el redondeo de enteros repartidos por `domain/combat` y `domain/run`.

**Se descarta** la alternativa de interfaz mínima (`{ next(): number }`) por lo anterior. Queda cerrada para Programmer: no hay ambigüedad de diseño pendiente.

### 2.3 `BaseRandomSource` — clase abstracta para no duplicar la derivación

Para que `nextInt`/`pick` no se reimplementen por separado en `DefaultRandomSource` y `SeededRandomSource` (y diverjan en redondeo entre ambas), se define una clase abstracta intermedia que implementa `nextInt`/`pick` en términos de `next()`, y cada clase concreta solo implementa `next()`:

```ts
export abstract class BaseRandomSource implements RandomSource {
  abstract next(): number;

  nextInt(minInclusive: number, maxExclusive: number): number {
    if (maxExclusive <= minInclusive) {
      throw new Error(`nextInt: maxExclusive (${maxExclusive}) debe ser > minInclusive (${minInclusive})`);
    }
    return minInclusive + Math.floor(this.next() * (maxExclusive - minInclusive));
  }

  pick<T>(items: readonly T[]): T {
    if (items.length === 0) {
      throw new Error('pick: items no puede estar vacío');
    }
    return items[this.nextInt(0, items.length)] as T;
  }
}
```

`RandomSource` (la interfaz) es el tipo público que consumen `domain/combat`/`domain/run` (para poder inyectar cualquier implementación, real o de test, sin acoplarse a una clase concreta). `BaseRandomSource` es un detalle de implementación interno de `shared` — se exporta igualmente desde `index.ts` por si un test futuro necesita extenderla directamente (p. ej. un `RandomSource` de test que fuerza una secuencia exacta), pero el contrato que documenta la API pública del paquete es `RandomSource`.

## 4. `SeededRandomSource` — implementación determinista

### 4.1 Algoritmo: mulberry32

Se elige **mulberry32** (PRNG de 32 bits, dominio público, sin dependencias externas — el repo no tiene ninguna librería de RNG instalada y esta historia no añade ninguna dependencia nueva a `package.json`). Es el algoritmo determinista simple más habitual para este propósito en TypeScript/JavaScript: una sola función pura, estado de 32 bits, período largo (~2^32), suficiente para reproducibilidad de tests — no se necesita calidad criptográfica ni un período mayor para este uso.

**Pseudocódigo exacto (Programmer implementa literalmente esto, sin variarlo):**

```ts
export class SeededRandomSource extends BaseRandomSource {
  private state: number;

  constructor(seed: number) {
    super();
    // Fuerza a entero de 32 bits sin signo; asegura estado inicial determinista
    // incluso si `seed` llega como float o negativo.
    this.state = seed >>> 0;
  }

  next(): number {
    // mulberry32 — https://github.com/bryc/code/blob/master/jshash/PRNGs.md (dominio público)
    this.state = (this.state + 0x6d2b79f5) | 0;
    let t = this.state;
    t = Math.imul(t ^ (t >>> 15), t | 1);
    t ^= t + Math.imul(t ^ (t >>> 7), t | 61);
    return ((t ^ (t >>> 14)) >>> 0) / 4294967296;
  }
}
```

Notas de implementación obligatorias:
- `this.state` se actualiza en cada llamada a `next()` — es el único estado mutable de la clase.
- El resultado de `next()` siempre cae en `[0, 1)` porque se divide por `4294967296` (2^32) un entero sin signo de 32 bits (`>>> 0`).
- No usar `Math.random()` en ningún punto de `SeededRandomSource` — si aparece, la clase deja de ser determinista y rompe el criterio de aceptación.
- Dos instancias de `SeededRandomSource` construidas con la misma semilla y con el mismo número de llamadas a `next()`/`nextInt()`/`pick()` **deben** producir exactamente la misma secuencia de valores — es el criterio de aceptación central de la historia.

### 4.2 Tipo de semilla

`seed: number`. No se admite `string` en esta historia (mulberry32 opera sobre enteros de 32 bits; convertir un string a semilla numérica —p. ej. un hash— queda fuera de alcance y no lo pide el criterio de aceptación). Si en el futuro se necesita sembrar con string (p. ej. el id de una run), la conversión string→number es responsabilidad del código que instancia `SeededRandomSource`, no de esta clase.

## 6. Tests obligatorios

Archivo: `packages/domain/shared/src/random.test.ts`

Casos exactos que debe escribir Programmer (nombres de `describe`/`it` orientativos, pero deben cubrir exactamente estos escenarios):

```ts
import { describe, it, expect } from 'vitest';
import { DefaultRandomSource, SeededRandomSource } from './random';

describe('SeededRandomSource', () => {
  it('produce la misma secuencia de next() con la misma semilla', () => {
    const a = new SeededRandomSource(42);
    const b = new SeededRandomSource(42);

    const seqA = [a.next(), a.next(), a.next()];
    const seqB = [b.next(), b.next(), b.next()];

    expect(seqA).toEqual(seqB);
  });

  it('produce secuencias distintas con semillas distintas', () => {
    const a = new SeededRandomSource(1);
    const b = new SeededRandomSource(2);

    const seqA = [a.next(), a.next(), a.next()];
    const seqB = [b.next(), b.next(), b.next()];

    expect(seqA).not.toEqual(seqB);
  });

  it('nextInt es reproducible con la misma semilla y respeta el rango [min, max)', () => {
    const a = new SeededRandomSource(7);
    const b = new SeededRandomSource(7);

    const rollsA = Array.from({ length: 20 }, () => a.nextInt(1, 5)); // simula valores de Núcleo 1-4
    const rollsB = Array.from({ length: 20 }, () => b.nextInt(1, 5));

    expect(rollsA).toEqual(rollsB);
    for (const roll of rollsA) {
      expect(roll).toBeGreaterThanOrEqual(1);
      expect(roll).toBeLessThan(5);
    }
  });

  it('pick es reproducible con la misma semilla y solo devuelve elementos de la lista', () => {
    const secuaces = ['A', 'B', 'C'] as const;
    const a = new SeededRandomSource(99);
    const b = new SeededRandomSource(99);

    const picksA = Array.from({ length: 10 }, () => a.pick(secuaces));
    const picksB = Array.from({ length: 10 }, () => b.pick(secuaces));

    expect(picksA).toEqual(picksB);
    for (const p of picksA) {
      expect(secuaces).toContain(p);
    }
  });

  it('nextInt lanza si maxExclusive <= minInclusive', () => {
    const rng = new SeededRandomSource(1);
    expect(() => rng.nextInt(5, 5)).toThrow();
  });

  it('pick lanza si la lista está vacía', () => {
    const rng = new SeededRandomSource(1);
    expect(() => rng.pick([])).toThrow();
  });
});

describe('DefaultRandomSource', () => {
  it('next() devuelve siempre un valor en [0, 1)', () => {
    const rng = new DefaultRandomSource();
    for (let i = 0; i < 100; i++) {
      const v = rng.next();
      expect(v).toBeGreaterThanOrEqual(0);
      expect(v).toBeLessThan(1);
    }
  });
});
```

El primer test (`'produce la misma secuencia de next() con la misma semilla'`) es el que satisface literalmente el criterio de aceptación de backlog ("Al menos un test unitario muestra reproducibilidad con semilla fija"); los demás son cobertura adicional que ya deja cubiertos los usos futuros descritos en §2.1 sin necesidad de una historia de test aparte cuando lleguen H1.3+.
