Consume GDD §2.3 ("Azar Compartido"), §3.4, §3.5, §5.3 (`docs/GDD_The_Collector_v2.md`), y el código real de H1.1-H1.6 ya implementado y testeado en `packages/domain/combat/src/` y `packages/domain/shared/src/` (verificado línea a línea al escribir esta spec, no solo las specs anteriores). Mismo nivel de detalle que `docs/specs/H1.5_umbral.md`/`docs/specs/H1.6_trama_dano_separados.md`: sin código de producto salvo firmas/pseudocódigo exacto y tests literales.

## 0. Decisiones de diseño de sistema que cierra esta spec

### 0.1 "Dramaturgia" SÍ es un mazo real en el GDD — pero su mecánica de robo/mazo NO es alcance de H1.7

GDD §3.4 es explícito: *"Una acción por turno: siempre empieza robando la carta superior de Dramaturgia. La carta tiene icono ⚔️ o 📜"*. GDD §5.3 confirma que es un mazo de contenido real de 30 cartas (10 del Enemigo + 10 del Escenario + 10 comunes) con un mando de icono ⚔️/📜 emergente de qué cartas se incluyeron. **No es una invención a evitar: es un mazo/deck real que se roba y se descarta**, no un simple "listado de habilidades disponibles".

Dicho esto, el propio backlog ya reparte esta mecánica en historias distintas, y H1.7 no es ninguna de ellas:
- **Contenido del mazo** (qué cartas exactas tiene cada Enemigo/Escenario, sus 5 comunes) es H1.10/H1.11 ("Dramaturgia mínima cargable y ejecutable en IA" / "Dramaturgia cargable y jugable con combate").
- **Mecánica de runtime del mazo** (barajar, robar la carta superior, descartar, reponer cuando se agota) es H1.18 ("Sistema básico de juego de combate... inicializar combate (Líder, Enemigo, Escenario, **Dramaturgia**)... Turno enemigo (IA) ocurre automáticamente tras END_TURN del jugador").

H1.7, por diseño de la propia historia (backlog: *"cuando el Enemigo actúa, elige una carta de Dramaturgia (⚔️ o 📜)"* — en tiempo presente, como un dato ya disponible en ese instante, no como "construye el mazo y róbalo"), **recibe el icono de la carta ya robada como input externo**, exactamente igual que H1.5 recibió `NucleoValue` como input externo sin depender de cómo se genera un Núcleo en el pool real. Se modela como un tipo mínimo:

```ts
/** Icono de la carta de Dramaturgia YA robada por quien gestiona el mazo (H1.18, fuera de
 *  alcance aquí) — GDD §3.4. H1.7 NO construye el mazo, NO lo baraja, NO lo roba, NO lo
 *  descarta: recibe únicamente el resultado (⚔️ = 'ATTACK', 📜 = 'PLOT') como argumento
 *  explícito de `decideEnemyAbility` (§3), mismo patrón que H1.5 recibiendo `NucleoValue`
 *  sin depender de `nucleo-pool.ts`. */
export type DramaturgiaCardIcon = 'ATTACK' | 'PLOT';
```

**Consecuencia de alcance:** H1.7 no toca `CombatEngineConfig`, no añade ningún `CombatCommand`/`CombatEvent`, y no introduce ningún draw pile/discard pile. Esto es idéntico en espíritu a la decisión de H1.5 §0.1 ("capa de funciones puras, sin tocar `CombatEngine`").

### 0.2 "Firma" vs "básica": son conceptos distintos entre Ataque y Trama — modelados en un mapa nuevo, separado, solo para el Enemigo

GDD §3.5, leído literalmente y comparado con el criterio de aceptación del backlog (*"Ataque: firma > básica; Trama: max-CD > básica"*), describe **dos mecánicas distintas**, no una sola renombrada:

```
Cuando la carta es ⚔️ (Ataque):
  1. Habilidad firma  — si tiene Núcleo mínimo disponible
  2. Básica de Ataque — siempre disponible (⚫, CD1)

Cuando la carta es 📜 (Trama):
  1. Habilidad de Trama de mayor CD — si está desbloqueada
  2. Básica de Trama — siempre disponible (⚫, CD1)
```

- **Para Ataque**, "firma" es una **etiqueta de contenido** (una habilidad concreta marcada por quien diseña el Enemigo como su ataque "de firma/insignia"), no simplemente "la de mayor CD entre las que no son básica" — el GDD la nombra como una categoría propia, distinta de "básica", sin mencionar "mayor CD" en absoluto para esta rama.
- **Para Trama**, en cambio, el GDD **no usa la palabra "firma"** — usa explícitamente "de mayor CD", es decir, un criterio **calculado dinámicamente** sobre el CD base de las habilidades de Trama disponibles, sin necesitar ninguna etiqueta de contenido adicional. El propio backlog refuerza esta lectura llamándolo "max-CD", no "firma", en su criterio de aceptación.

**Modelado elegido:** un nuevo tipo `EnemyAbilityAiProfile` con dos campos — `branch` (`'ATTACK' | 'PLOT'`, qué icono de Dramaturgia activa esta habilidad) y `tier` (`'FIRMA' | 'BASICA' | 'STANDARD'`) — inyectado en un mapa **nuevo y separado** `ReadonlyMap<AbilityId, EnemyAbilityAiProfile>`, **fuera de `AbilityCooldownDefinition`** (que sigue usando el Líder, y que no necesita esta clasificación — el Líder no tiene "carta de Dramaturgia" ni prioridades de IA, GDD §3.1-§3.3) y **fuera de `AbilityEffectDefinition`** (H1.6; el `kind: 'ATTACK'|'PLOT'` de ese mapa es *opcional* y describe la mecánica numérica del efecto, no la prioridad de IA — una habilidad podría en teoría tener perfil de IA sin todavía tener un efecto numérico modelado, o viceversa; acoplar ambos mapas obligaría a que toda habilidad de IA tuviera ya su efecto resuelto, una dependencia que esta historia no necesita imponer).

**Invariantes de `tier` por `branch`** (ver `validateEnemyAbilityAiProfiles`, §3.4):
- `branch: 'ATTACK'` → `tier` solo puede ser `'FIRMA'` o `'BASICA'` (nunca `'STANDARD'` — el GDD no describe una tercera categoría de Ataque).
- `branch: 'PLOT'` → `tier` solo puede ser `'STANDARD'` o `'BASICA'` (nunca `'FIRMA'` — el GDD nunca usa esa palabra para Trama).
- Exactamente **una** entrada con `tier: 'BASICA'` por cada `branch` presente en el perfil de un Enemigo — refleja "CD1 doble" (GDD §3.4: una básica de Ataque + una básica de Trama, ambas ⚫) — y esa entrada debe tener `AbilityCooldownDefinition.baseCooldown === ABILITY_BASE_COOLDOWN_MIN` (1).
- Toda clave del mapa debe corresponder a una habilidad cuyo `AbilityCooldownDefinition.side` sea `'ENEMY'` (el Líder nunca tiene perfil de IA).

**Asunción explícita no bloqueante (declarada, no escalada):** se asume que el kit de Ataque de un Enemigo se modela como exactamente 2 habilidades relevantes para IA (una básica + como mucho una firma), sin una tercera categoría "de relleno". El GDD no describe explícitamente un tope al número de habilidades de Ataque totales de un Enemigo, pero tampoco describe una tercera prioridad más allá de firma/básica — si contenido futuro necesitara más de una habilidad "firma" simultánea, el desempate se resuelve igual que el de Trama (mayor CD, luego aleatorio — ver §0.4/§3.2). No se ha encontrado necesidad de escalar esto: es una extensión conservadora de un algoritmo ya explícito, no una decisión de negocio nueva.

### 0.3 Capa 1 (qué habilidad activar) y Capa 2 (qué Núcleo gastar): dos funciones puras independientes, sin tocar `CombatEngine`

Igual que H1.5 (`resolveAbilityUmbral`) y a diferencia de H1.6 (que sí conectó Umbral a una mutación real), H1.7 implementa **únicamente** dos funciones puras — `decideEnemyAbility` (Capa 1, §3.1) y `decideEnemyNucleoToSpend` (Capa 2, §3.2) — que reciben como argumentos explícitos todo lo que necesitan saber del tablero (candidatos de habilidad con su CD/perfil de IA, el pool de Núcleos actual, los "colores del jugador"), sin leer ni mutar ningún `CombatEngine`. Se testean pasándoles inputs de tablero construidos a mano, exactamente como los tests de `umbral.test.ts` construyen una `AbilityUmbralDefinition` y un `NucleoValue` literal sin levantar un motor.

**La conexión con `dispatch()` (traducir la decisión de la IA en un `ACTIVATE_ABILITY` real, disparado automáticamente al terminar el turno del jugador) es explícitamente responsabilidad de H1.18** — el propio criterio de aceptación de H1.18 lo dice literalmente: *"Turno enemigo (IA) ocurre automáticamente tras END_TURN del jugador"*. Añadir aquí un `CombatCommand` nuevo (tipo `AI_ACT`) o modificar `handleEndTurn()` para que dispare la IA automáticamente adelantaría ese trabajo sin que existan todavía ni el mazo de Dramaturgia real (§0.1) ni la condición de victoria/derrota (H1.18) que ese loop necesita para decidir cuándo parar. **Decisión de arquitectura explícita: H1.7 no añade ningún `CombatCommand`/`CombatEvent` nuevo, no toca `combat-engine.ts` ni `types/config.ts`.**

### 0.4 "Negar Núcleo alto > mayor valor": semántica resuelta citando el GDD, con 2 sub-decisiones menores declaradas (no bloqueantes)

Este es el punto marcado como más probable de ambigüedad en el encargo. Tras leer GDD §2.3 ("Sistema de Núcleos — **Azar Compartido**") junto a §3.5, la semántica queda resuelta con confianza, sin necesidad de escalar al Director:

- El pool de Núcleos es **compartido** entre Líder y Enemigo (§2.3, título literal de la sección) — cuando el Enemigo gasta un Núcleo, ese Núcleo desaparece del pool y **ya no está disponible para el Líder** hasta el próximo relanzado. Esto es exactamente lo que GDD §3.5 llama "negar": el Enemigo, al elegir gastar un Núcleo de un color que el Líder necesita (para pagar sus propias habilidades) y de valor alto (≥3, umbral de Umbral — GDD §12), le quita al jugador esa opción concreta en su próximo turno, "degradándolo" sin "apagarlo" (cita literal de §3.5).
- **"Colores del jugador"** se resuelve con datos que el motor YA tiene disponibles hoy: el conjunto de colores que aparecen en `CoreCostRequirement.colors` de alguna habilidad `side: 'LEADER'` en `abilityCoreCosts`/`abilityCooldowns` (las habilidades `kind: 'ANY'`/⚫ del Líder no "necesitan" ningún color concreto, así que no contribuyen a este conjunto). Se provee una función auxiliar (`derivePlayerColorsFromLeaderAbilities`, §3.3) que deriva este conjunto a partir de los mismos mapas que ya recibe `CombatEngineConfig` — no hace falta ningún dato nuevo de catálogo ni de `LeaderDefinition` (que no existe todavía, H1.8).
- El algoritmo completo de Capa 2 (§3.2) queda así, en el mismo orden que describe GDD §3.5:
  1. Entre los Núcleos válidos para el coste de la habilidad elegida, prioriza los que sean de un color de `playerColors` **y** valor ≥3 ("denegar").
  2. Si no hay ninguno: el de mayor valor disponible entre los válidos.
  3. Si hay empate en el paso 1 o en el paso 2 (o si `playerColors` está vacío, con lo que el paso 1 nunca aporta candidatos): cualquiera de los empatados, elegido vía `RandomSource.pick` — mismo patrón ya usado en el motor para desempates/azar (`rollNucleo`, selección de Secuaz H1.16).

**Sub-decisión 1 (declarada, no bloqueante):** GDD §3.5 encabeza esta capa como "Capa 2 — qué Núcleo gastar (**coste ⚫**)", sugiriendo que el ejemplo canónico son habilidades de coste `ANY`. H1.7 **generaliza** el mismo algoritmo a habilidades de coste `COLOR` específico (filtrando primero por `satisfiesCoreCost`, igual que ya hace `handleActivateAbility`) — si una habilidad "firma" tuviera coste 🔴 específico y hubiera varios Núcleos rojos disponibles, el mismo criterio (denegar → mayor valor → aleatorio) decide entre ellos. Esto no contradice al GDD (que no dice nada sobre qué hacer en ese caso), solo rellena un hueco no cubierto explícitamente con la extensión más simple y consistente. Si el Director prefiere que Capa 2 solo aplique a coste ⚫ y que el coste `COLOR` específico se resuelva con otra regla, es un ajuste de una línea en `decideEnemyNucleoToSpend` — no bloquea la implementación de esta historia.

**Sub-decisión 2 (declarada, no bloqueante):** dentro del grupo de "denegar" (color de jugador + valor ≥3), si hay varios Núcleos que cumplen, se prioriza entre ellos el de **mayor valor** (denegar "más fuerte" primero), con empate resuelto vía `RandomSource`. El GDD no dice explícitamente cómo desempatar dentro del propio grupo de "denegar" — esta es la lectura más consistente con el resto del algoritmo (siempre preferir mayor valor cuando hay que elegir entre válidos del mismo nivel de prioridad).

### 0.5 Ninguna otra ambigüedad de negocio detectada

Fuera de las asunciones explícitas ya declaradas (§0.2: kit de Ataque de 2 niveles firma/básica sin tercera categoría; §0.4: generalización a coste `COLOR` y desempate interno del grupo "denegar"), no se ha encontrado ninguna decisión de diseño de negocio que requiera escalarse al Director en esta historia — se resuelve todo citando el GDD/backlog con confianza, en particular gracias a que §2.3 ("Azar Compartido") deja sin ambigüedad qué significa "colores del jugador" y "denegar".

## 3. Funciones puras — `packages/domain/combat/src/enemy-ai.ts` (NUEVO)

### 3.1 `decideEnemyAbility` — Capa 1 (qué habilidad de Dramaturgia activar)

```ts
import type { RandomSource, AbilityId, CoreCostRequirement, NucleoColor } from '@collector/domain-shared';
import { satisfiesCoreCost } from '@collector/domain-shared';
import type { NucleoInstance } from './types/nucleo';
import type { AbilityCooldownDefinition } from './types/cooldown';
import { ABILITY_BASE_COOLDOWN_MIN } from './types/cooldown';
import type {
  DramaturgiaCardIcon,
  EnemyAbilityBranch,
  EnemyAbilityCandidate,
  EnemyAbilityDecision,
  EnemyAbilityAiProfile,
  EnemyNucleoDecision,
} from './types/enemy-ai';

/** Existe al menos un Núcleo en `pool` que satisface `requirement` (GDD §3.5: "si tiene
 *  Núcleo mínimo disponible"). Pura, reutiliza `satisfiesCoreCost` (H1.3). */
export function poolHasValidNucleo(requirement: CoreCostRequirement, pool: readonly NucleoInstance[]): boolean {
  return pool.some((n) => satisfiesCoreCost(requirement, n.color));
}

/**
 * Capa 1 (GDD §3.5): decide QUÉ habilidad del Enemigo se activa dado el icono de la
 * carta de Dramaturgia ya robada (§0.1). `candidates` debe incluir TODAS las
 * habilidades del Enemigo relevantes para IA (ambas ramas) — la función filtra
 * internamente por `icon`.
 *
 * Algoritmo (idéntico al árbol de GDD §3.5, ver spec §0.2/§0.4):
 *  - Rama ATTACK: prioridad 1 = candidatas `tier: 'FIRMA'`, CD listo (`remainingCooldown
 *    === 0`) Y con Núcleo válido disponible (`poolHasValidNucleo`); si hay más de una,
 *    desempate por `baseCooldown` más alto, y si sigue empatado, `RandomSource.pick`.
 *    Si el grupo de prioridad 1 está vacío, cae a la candidata `tier: 'BASICA'`.
 *  - Rama PLOT: prioridad 1 = candidatas `tier: 'STANDARD'` listas y con Núcleo válido;
 *    mismo desempate por `baseCooldown`/aleatorio. Si vacío, cae a `tier: 'BASICA'`.
 *  - La candidata BASICA de la rama pedida DEBE existir en `candidates` con
 *    `remainingCooldown === 0` (GDD §3.4: "siempre disponible") — si no se encuentra,
 *    lanza `Error` (invariante de contenido violada, no un caso de juego válido).
 */
export function decideEnemyAbility(
  icon: DramaturgiaCardIcon,
  candidates: readonly EnemyAbilityCandidate[],
  pool: readonly NucleoInstance[],
  randomSource: RandomSource
): EnemyAbilityDecision {
  const branch: EnemyAbilityBranch = icon;
  const readyInBranch = candidates.filter(
    (c) => c.aiProfile.branch === branch && c.remainingCooldown === 0
  );

  const priorityTier: EnemyAbilityAiProfile['tier'] = branch === 'ATTACK' ? 'FIRMA' : 'STANDARD';
  const priorityGroup = readyInBranch.filter(
    (c) => c.aiProfile.tier === priorityTier && poolHasValidNucleo(c.coreCost, pool)
  );

  const chosen =
    priorityGroup.length > 0
      ? pickHighestBaseCooldown(priorityGroup, randomSource)
      : pickBasica(readyInBranch, branch);

  return { abilityId: chosen.abilityId, branch, tier: chosen.aiProfile.tier };
}

function pickHighestBaseCooldown(
  group: readonly EnemyAbilityCandidate[],
  randomSource: RandomSource
): EnemyAbilityCandidate {
  const maxCooldown = Math.max(...group.map((c) => c.baseCooldown));
  const top = group.filter((c) => c.baseCooldown === maxCooldown);
  return top.length === 1 ? (top[0] as EnemyAbilityCandidate) : randomSource.pick(top);
}

function pickBasica(
  readyInBranch: readonly EnemyAbilityCandidate[],
  branch: EnemyAbilityBranch
): EnemyAbilityCandidate {
  const basica = readyInBranch.filter((c) => c.aiProfile.tier === 'BASICA');
  if (basica.length !== 1) {
    throw new Error(
      `decideEnemyAbility: se esperaba exactamente 1 candidata BASICA lista para la rama "${branch}" (GDD §3.4, "CD1 doble... siempre disponible"), encontradas ${basica.length}`
    );
  }
  return basica[0] as EnemyAbilityCandidate;
}
```

### 3.2 `decideEnemyNucleoToSpend` — Capa 2 (qué Núcleo gastar)

```ts
/**
 * Capa 2 (GDD §3.5, ver spec §0.4): decide QUÉ instancia de Núcleo del pool se gasta
 * para pagar `requirement` (el coste de la habilidad ya elegida por `decideEnemyAbility`).
 *
 * Algoritmo:
 *  1. `valid` = Núcleos de `pool` que satisfacen `requirement` (`satisfiesCoreCost`).
 *     Si `valid` está vacío, lanza `Error` (invariante violada — el caller debería haber
 *     comprobado `poolHasValidNucleo` antes de llegar aquí, vía Capa 1).
 *  2. `deny` = de `valid`, los de color ∈ `playerColors` Y `value >= 3` (GDD §12,
 *     `UMBRAL_BONUS_THRESHOLD`). Si `deny` no está vacío: de esos, el de mayor `value`
 *     (empate → `randomSource.pick`), razón `'DENY_PLAYER_COLOR'`.
 *  3. Si `deny` está vacío: de `valid`, el de mayor `value` (empate → `randomSource.pick`,
 *     razón `'ARBITRARY'`; sin empate, razón `'HIGHEST_VALUE'`).
 */
export function decideEnemyNucleoToSpend(
  requirement: CoreCostRequirement,
  pool: readonly NucleoInstance[],
  playerColors: readonly NucleoColor[],
  randomSource: RandomSource
): EnemyNucleoDecision {
  const valid = pool.filter((n) => satisfiesCoreCost(requirement, n.color));
  if (valid.length === 0) {
    throw new Error(
      'decideEnemyNucleoToSpend: no hay ningún Núcleo en el pool que satisfaga el coste requerido — el caller debe comprobar poolHasValidNucleo antes de invocar esta función'
    );
  }

  const deny = valid.filter((n) => playerColors.includes(n.color) && n.value >= 3);
  if (deny.length > 0) {
    const maxValue = Math.max(...deny.map((n) => n.value));
    const top = deny.filter((n) => n.value === maxValue);
    const nucleo = top.length === 1 ? (top[0] as NucleoInstance) : randomSource.pick(top);
    return { nucleo, reason: 'DENY_PLAYER_COLOR' };
  }

  const maxValue = Math.max(...valid.map((n) => n.value));
  const top = valid.filter((n) => n.value === maxValue);
  if (top.length === 1) {
    return { nucleo: top[0] as NucleoInstance, reason: 'HIGHEST_VALUE' };
  }
  return { nucleo: randomSource.pick(top), reason: 'ARBITRARY' };
}
```

### 3.3 `derivePlayerColorsFromLeaderAbilities` — helper para obtener `playerColors`

```ts
import type { CoreCostRequirement, NucleoColor, AbilityId } from '@collector/domain-shared';
import type { AbilityCooldownDefinition } from './types/cooldown';

/**
 * Deriva el conjunto de "colores del jugador" (GDD §3.5, spec H1.7 §0.4): los colores
 * que aparecen en el coste `kind: 'COLOR'` de alguna habilidad `side: 'LEADER'`. Las
 * habilidades `kind: 'ANY'` (⚫) del Líder no "necesitan" ningún color concreto y no
 * contribuyen. Pura, sin acceso a `CombatEngine` — opera directamente sobre los mismos
 * mapas que ya recibe `CombatEngineConfig` (`abilityCoreCosts`/`abilityCooldowns`).
 */
export function derivePlayerColorsFromLeaderAbilities(
  abilityCoreCosts: ReadonlyMap<AbilityId, CoreCostRequirement>,
  abilityCooldowns: ReadonlyMap<AbilityId, AbilityCooldownDefinition>
): readonly NucleoColor[] {
  const colors = new Set<NucleoColor>();
  for (const [abilityId, requirement] of abilityCoreCosts) {
    const def = abilityCooldowns.get(abilityId);
    if (def?.side !== 'LEADER') continue;
    if (requirement.kind === 'COLOR') {
      for (const color of requirement.colors) colors.add(color);
    }
  }
  return [...colors];
}
```

### 3.4 `validateEnemyAbilityAiProfiles` — invariantes de contenido (§0.2)

```ts
import type { AbilityId } from '@collector/domain-shared';
import type { AbilityCooldownDefinition } from './types/cooldown';
import { ABILITY_BASE_COOLDOWN_MIN } from './types/cooldown';
import type { EnemyAbilityAiProfile, EnemyAbilityBranch } from './types/enemy-ai';

/**
 * Valida las invariantes de contenido de un mapa de perfiles de IA (spec H1.7 §0.2).
 * Lanza `Error` con mensaje descriptivo en la primera violación encontrada, mismo estilo
 * que `CombatEngine.validateAbilityCooldownsConfig` (H1.4). NO se invoca desde el
 * constructor de `CombatEngine` en esta historia (§0.3: el mapa de perfiles de IA no
 * vive en `CombatEngineConfig` todavía) — es una función independiente para que
 * tests/H1.18 la llamen explícitamente al ensamblar contenido de Enemigo.
 *
 * Invariantes:
 *  1. Toda clave debe existir en `abilityCooldowns` con `side === 'ENEMY'`.
 *  2. `branch: 'ATTACK'` → `tier` ∈ {'FIRMA','BASICA'}; `branch: 'PLOT'` → `tier` ∈
 *     {'STANDARD','BASICA'} (spec §0.2).
 *  3. Para cada `branch` presente, exactamente una entrada con `tier: 'BASICA'`, y esa
 *     entrada debe tener `abilityCooldowns.get(id).baseCooldown === ABILITY_BASE_COOLDOWN_MIN`.
 */
export function validateEnemyAbilityAiProfiles(
  profiles: ReadonlyMap<AbilityId, EnemyAbilityAiProfile>,
  abilityCooldowns: ReadonlyMap<AbilityId, AbilityCooldownDefinition>
): void {
  const basicaCountByBranch = new Map<EnemyAbilityBranch, number>();

  for (const [abilityId, profile] of profiles) {
    const def = abilityCooldowns.get(abilityId);
    if (!def || def.side !== 'ENEMY') {
      throw new Error(
        `validateEnemyAbilityAiProfiles: "${String(abilityId)}" no existe en abilityCooldowns con side "ENEMY"`
      );
    }

    if (profile.branch === 'ATTACK' && profile.tier === 'STANDARD') {
      throw new Error(
        `validateEnemyAbilityAiProfiles: "${String(abilityId)}" es branch ATTACK con tier STANDARD — GDD §3.5 no define una tercera categoría para Ataque (solo FIRMA/BASICA), ver spec H1.7 §0.2`
      );
    }
    if (profile.branch === 'PLOT' && profile.tier === 'FIRMA') {
      throw new Error(
        `validateEnemyAbilityAiProfiles: "${String(abilityId)}" es branch PLOT con tier FIRMA — GDD §3.5 nunca usa "firma" para Trama (solo STANDARD/BASICA), ver spec H1.7 §0.2`
      );
    }

    if (profile.tier === 'BASICA') {
      if (def.baseCooldown !== ABILITY_BASE_COOLDOWN_MIN) {
        throw new Error(
          `validateEnemyAbilityAiProfiles: "${String(abilityId)}" es tier BASICA pero baseCooldown es ${def.baseCooldown} (esperado ${ABILITY_BASE_COOLDOWN_MIN}, GDD §3.4 "CD1 doble")`
        );
      }
      basicaCountByBranch.set(profile.branch, (basicaCountByBranch.get(profile.branch) ?? 0) + 1);
    }
  }

  for (const branch of ['ATTACK', 'PLOT'] as const) {
    const count = basicaCountByBranch.get(branch) ?? 0;
    if (count !== 1) {
      throw new Error(
        `validateEnemyAbilityAiProfiles: se esperaba exactamente 1 tier BASICA para branch "${branch}" (GDD §3.4, "CD1 doble"), encontradas ${count}`
      );
    }
  }
}
```
