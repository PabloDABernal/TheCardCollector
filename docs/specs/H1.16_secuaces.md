
# H1.16 — Secuaces del enemigo (presencia pasiva, acción selectiva)

> Spec técnica del Architect para H1.16 (`.ai-studio/memory/backlog.md`, líneas 226-236). Consume GDD §3.8 (también §3.4-§3.7 para el vocabulario compartido de Ataque/Trama/Aliados), decisions.md, y el código real de H1.3-H1.15 (`packages/domain/combat/src/combat-engine.ts` completo, todos sus `types/*.ts`, `enemy-ai.ts`/`types/enemy-ai.ts`, `packages/domain/catalog/src/types/enemy.ts`/`keyword.ts`/`dramaturgia-card.ts`, `packages/data/enemies/*.json`) verificados línea a línea.
>
> **Nota de secuencia:** sigue a H1.15 (Aliados/absorción, cerrada). Esta historia cierra explícitamente la deuda técnica de reversión de Contratiempo que H1.14/H1.15 dejaron documentada como "código muerto hasta H1.16" (ver H1.15 spec §0.8).

Estado del repo al escribir esta spec:

- `combat-engine.ts` implementa H1.3-H1.15 completo. **No existe ningún concepto de "Secuaz"** en `domain/combat` — ni tipo, ni estado, ni comando. `packages/data/enemies/bestia-base.json` y `espectro-base.json` ya tienen cartas de Dramaturgia con `effectDescription: "Invoca un secuaz menor (no ejecutable todavía... H1.16)"` / `"Maldición Menor"` — texto narrativo no ejecutable (mismo patrón que Combo/Contratiempo antes de H1.14), que esta historia **no está obligada a conectar automáticamente**: el mecanismo de resolver una carta de Dramaturgia entera (más allá de su `icon`) sigue siendo alcance de H1.18 (ver `dramaturgia-card.ts` comentario "no referencia ningún AbilityId, deliberado").
- `turn.ts` y `cooldown.ts` **ya anticipan literalmente esta historia**: "Aliados/Secuaces (H1.15/H1.16) actúan DENTRO del turno de uno de estos dos lados — no son un tercer valor de `CombatSide`, son `sourceId` distintos dentro del mismo lado"; "Aliados/Secuaces (H1.15/H1.16, fuera de alcance) usarían el `side` de su dueño (LEADER o ENEMY)". Esta spec sigue esa decisión ya tomada: el `side` de toda acción de un Secuaz es `'ENEMY'`.
- **`DEFENSOR` ya existe en `KeywordId`** (`catalog/types/keyword.ts`) desde H1.9/H1.12, usada hoy en 2 cartas `ALIADO` del jugador (`card-soldado-base-06`, `card-mago-base-06`). H1.15 §0.4 lo marcó explícitamente como "ambigüedad de contenido preexistente, fuera de alcance — GDD §12/§3.8 define Defensor únicamente para Secuaces del Enemigo… no tiene definición para un Aliado del jugador". **Esta historia es la que le da semántica real a Defensor**, pero solo para Secuaces (ver §0.5) — no toca las 2 cartas Aliado existentes.
- **Deuda técnica activa (bug #25, detectado por QA de H1.14, confirmado inalcanzable hasta ahora):** `handlePlayContratiempo` revierte `leaderDamage`/`scenarioPlot` sobrescribiendo con el valor "antes" de la ÚLTIMA entrada de cada tipo en el bucle sobre `revertedEntries`, en vez de acumular/usar la PRIMERA. Era código muerto porque `ENEMY_BASE_ACTIONS_PER_TURN = 1` y ninguna historia hasta H1.15 generaba una 2ª entrada `ATTACK`/`PLOT` de `side: 'ENEMY'` en el mismo turno. **Esta historia SÍ la genera** (ver §0.3) — el bug se vuelve alcanzable y se corrige aquí (§0.4).

## 0. Decisiones de diseño que cierra esta spec

### 0.1 Modelo de datos: `MinionDefinition`/`MinionInPlay`, mismo patrón que `AllyCardDefinition`/`AllyInPlay` (H1.15)

`domain/combat` no importa `domain/catalog`. Se introduce `types/minion.ts` con un tipo de configuración resuelto externamente e inyectado en `CombatEngineConfig.minionDefinitions` (mapa por `MinionDefinitionId`, alias de `string` — no se introduce un `EnemyId`/`CardId` nuevo porque un Secuaz no es una `CardDefinition` de catálogo hoy, es un concepto propio del Enemigo/Escenario, GDD §3.8: "definido por el enemigo y/o el escenario"), y un tipo de instancia en mesa (`MinionInPlay`), identificado con el mismo `CardInstanceId` ya generalizado por H1.15 ("copia concreta de algo en juego en el tablero", no solo cartas de Aliado).

```ts
export type MinionDefinitionId = string;

/** GDD §3.8: "efecto pasivo definido por el enemigo y/o el escenario (acumulable entre
 *  ambas fuentes, sin tope)". Magnitud FIJA, igual que un efecto PLOT (H1.6) — un pasivo
 *  no consume Núcleo, así que no hay Umbral que resolver. */
export interface MinionPassiveEffectDefinition {
  readonly kind: 'ATTACK' | 'PLOT';
  readonly amount: number; // entero >= 0, mismo piso que abilityEffects (H1.6 §0.5)
  /** Solo relevante si kind === 'ATTACK'; mismo significado que AbilityEffectDefinition.arrollar. Default false. */
  readonly arrollar?: boolean;
}

export interface MinionDefinition {
  readonly passiveEffect: MinionPassiveEffectDefinition;
  /** GDD §3.8: "acción especial" opcional. Si está presente, DEBE existir en
   *  abilityCoreCosts/abilityCooldowns/abilityEffects con side: 'ENEMY' (validado en el
   *  constructor, mismo estilo que AllyCardDefinition.abilityIds, H1.15 §3.6). */
  readonly specialActionAbilityId?: AbilityId;
  /** GDD §3.8: "Si ninguno tiene acción especial, uno al azar usa su ataque plano" y
   *  "Daño bajo". Magnitud fija, SIN Núcleo ni CD — ver §0.2. Entero >= 0. */
  readonly planoAttackAmount: number;
  /** GDD §3.8, keyword Defensor: "obliga a recibir/atacar a ese secuaz primero". Ver §0.5
   *  sobre por qué esta historia modela su CONTRATO pero no su integración end-to-end. */
  readonly isDefensor: boolean;
}

export interface MinionInPlay {
  readonly instanceId: CardInstanceId;
  readonly definitionId: MinionDefinitionId;
  /** Denormalizado en el momento de invocar (mismo patrón que AllyInPlay.isBerserker/
   *  maxLife, H1.15) — snapshots estables aunque el config sea inmutable de todas formas. */
  readonly passiveEffect: MinionPassiveEffectDefinition;
  readonly specialActionAbilityId?: AbilityId;
  readonly planoAttackAmount: number;
  readonly isDefensor: boolean;
}
```

**Sin campo de vida.** A diferencia de `AllyInPlay`, un Secuaz NO tiene vida en esta historia: no existe todavía ningún mecanismo por el que el jugador dañe al Enemigo o a sus Secuaces (`validateAbilityEffectsConfig` invariante 2 sigue exigiendo que todo efecto `ATTACK` pertenezca a una habilidad `side: 'ENEMY'` — no hay `enemyHealth`, ver `types/ability-effect.ts` comentario "no existe todavía enemyHealth"). Un Secuaz, una vez invocado, permanece en mesa indefinidamente en esta historia (GDD §3.8: "Sin límite de presencia" ya es coherente con esto — nada los retira salvo una historia futura que sí modele daño del jugador al tablero enemigo).

### 0.2 "Ataque plano" no consume Núcleo ni tiene CD — deliberado, distinto de `specialActionAbilityId`

GDD §3.8: *"Selección del que actúa: aleatorio con filtro de validez — entre los que pueden ejecutar su acción (no en CD, Núcleo disponible). Si ninguno tiene acción especial, uno al azar usa su ataque plano."* Esto describe DOS niveles:
1. **Acción especial** (`specialActionAbilityId`): es una habilidad real de catálogo, con su propio `coreCost`/`baseCooldown`/`effect` en los mismos mapas que ya usa el Enemigo (`abilityCoreCosts`/`abilityCooldowns`/`abilityEffects`) — reutiliza el 100% de la maquinaria existente (Umbral, CD, gasto de Núcleo, log de Contratiempo).
2. **Ataque plano**: el *fallback* cuando ningún Secuaz en mesa tiene una acción especial lista. GDD lo describe como una acción "de siempre disponible" sin mencionar CD ni Núcleo — se modela como magnitud fija (`planoAttackAmount`) sin coste, análogo a un efecto `PLOT`/pasivo (H1.6 §0.3: "magnitud fija… NO alimentada por Núcleo"), no como una `AbilityId` de catálogo.

### 0.3 Cómo actúa un Secuaz: nuevos comandos dedicados, mismo molde que `PLAY_ALLY`/`SET_DAMAGE_REDIRECT` (H1.15) — y aquí es donde se introduce la 2ª fuente ATTACK/PLOT de `side: 'ENEMY'` por turno

Dos comandos nuevos, ambos exclusivos de `turnOwner === 'ENEMY'`:

**`SUMMON_MINION`** — introduce un Secuaz en mesa (equivalente a lo que hoy hace, sin ejecutar, el `effectDescription` de una carta de Dramaturgia). Sin coste de acción/Núcleo (GDD no describe ninguno; "Sin límite de presencia" ya asume que invocar no compite con el resto de la economía de turno).

```ts
{ readonly type: 'SUMMON_MINION'; readonly minionDefinitionId: MinionDefinitionId; readonly sourceId: string }
```

**`RESOLVE_MINION_ACTION`** — decide Y ejecuta, en una sola llamada, la acción del turno de los Secuaces (selección aleatoria incluida). **Sin payload** — a diferencia de `ACTIVATE_ABILITY` (donde el CALLER elige `abilityId`/`nucleoInstanceId`), aquí la selección es responsabilidad del motor (GDD: "aleatorio con filtro de validez"), igual que `decideEnemyAbility`/`decideEnemyNucleoToSpend` (H1.7) ya son decisiones de IA, no del caller. Esta historia dispatchea esa decisión IA directamente contra el estado del motor (a diferencia de H1.7, que dejó sus funciones puras sin integrar en `dispatch` — aquí si se integra, porque el criterio de aceptación "Efecto pasivo es leído por `CombatEngine`" exige que el motor, no un harness externo, sea quien decide y mute):

```ts
{ readonly type: 'RESOLVE_MINION_ACTION' }
```

Gating: **1 sola vez por turno de Enemigo**, con un contador PROPIO (`minionActionResolvedThisEnemyTurn: boolean`), completamente independiente de `actionsTakenThisTurn`/`actionsAllowedThisTurn` (que siguen siendo, sin cambios, la única acción del Enemigo mismo — `ENEMY_BASE_ACTIONS_PER_TURN` permanece en `1`, NO se toca). Esto modela literalmente GDD §3.8 "iniciativa propia" (de los Secuaces) frente a GDD §3.4 "una acción por turno" (del Enemigo mismo): son dos presupuestos de acción distintos que coexisten en el mismo turno de `side: 'ENEMY'`. **Esta es precisamente la 2ª fuente de mutación `ATTACK`/`PLOT` de `side: 'ENEMY'` en el mismo turno** que H1.14/H1.15 dejaron marcada como inalcanzable — con `RESOLVE_MINION_ACTION` deja de serlo en cuanto hay ≥1 Secuaz en mesa con una acción resuelta como `ATTACK`/`PLOT`.

Resolución interna de `RESOLVE_MINION_ACTION` (ver `minion-ai.ts`, §5, para las funciones puras exactas):
1. Construir candidatos = `minionsInPlay` con `specialActionAbilityId` definido, CD listo (`remainingCooldowns.get(id) === 0`) y Núcleo válido en `nucleoPool` para su `coreCost`.
2. Si `candidatos` no vacío → `randomSource.pick(candidatos)` (uniforme, sin desempate por CD como en `decideEnemyAbility` — GDD §3.8 dice "aleatorio", no "de mayor CD"); a continuación, elegir Núcleo a gastar reutilizando `decideEnemyNucleoToSpend` (H1.7, sin cambios) contra `abilityCoreCosts.get(chosen.specialActionAbilityId)`; ejecutar el mismo camino que `handleActivateAbility` para ese `abilityId` (gasto de Núcleo, CD → `baseCooldown`, `applyAttackEffect`/`applyPlotEffect`, entrada en `currentEnemyTurnLog` con `origin: 'ABILITY'`, ver §0.6) — con `sourceId` = `instanceId` del Secuaz elegido.
3. Si `candidatos` vacío pero `minionsInPlay` no vacío → `randomSource.pick(minionsInPlay)` (TODOS son válidos para el fallback, GDD: "uno al azar usa su ataque plano" — sin filtro de CD/Núcleo, coherente con §0.2); aplicar `planoAttackAmount` como un `ATTACK` de magnitud fija: mismo `resolveDamageTarget()`/absorción por Escudo o Aliado que cualquier Ataque (H1.6/H1.15, sin cambios), `arrollar` siempre `false` para el ataque plano (GDD no le da esa keyword — es literalmente "plano"); entrada en `currentEnemyTurnLog` con `origin: 'MINION_PASSIVE'`-like pero etiquetada `origin: 'MINION_PLANO_ATTACK'` (ver §0.6) — sin `abilityId`/`cooldownBefore` (no hay habilidad ni CD implicados).
4. Si `minionsInPlay` vacío → no hace nada, pero **no es un error** (turno de Enemigo sin Secuaces en mesa es el caso normal antes de que se invoque el primero); emite un evento informativo `MINION_ACTION_SKIPPED` con `reason: 'NO_MINIONS_IN_PLAY'`.
5. Si `minionActionResolvedThisEnemyTurn` ya es `true` → error `MINION_ACTION_ALREADY_RESOLVED_THIS_TURN` (dispatch repetido en el mismo turno).
6. Si `turnOwner !== 'ENEMY'` → `NOT_YOUR_TURN` (`expected: 'ENEMY'`).

`minionActionResolvedThisEnemyTurn` se resetea a `false` en `handleEndTurn` exactamente donde se resetean `actionsTakenThisTurn`/`comboBonusGrantedThisTurn` hoy (para el lado que RECIBE el turno) — pero además solo tiene efecto real cuando ese lado es `'ENEMY'` (resetearlo cuando el Líder recibe turno es inocuo, simplemente no se usa hasta el próximo turno de Enemigo).

### 0.4 EL FIX del bug de Contratiempo — obligatorio en esta historia

**Diagnóstico exacto (línea a línea, código actual, `handlePlayContratiempo`):**

```ts
const revertedEntries = [...this.previousEnemyTurnLog];
for (const entry of revertedEntries) {
  if (def.undoScope === 'FULL_TURN') {
    this.remainingCooldowns.set(entry.abilityId, entry.cooldownBefore);
  }
  if (!entry.effect) continue;
  if (entry.effect.kind === 'ATTACK') {
    this.leaderDamage = entry.effect.leaderDamageBefore;          // <-- BUG: sobrescribe, no acumula
    if (entry.effect.target === 'LEADER') {
      this.leaderShield = entry.effect.leaderShieldBefore;        // <-- mismo BUG
    } else {
      /* restaura vida del Aliado — esto SÍ es correcto porque está keyed por allyInstanceId,
         no hay colisión salvo que el MISMO Aliado sea golpeado 2 veces en el turno (ver abajo) */
    }
  } else if (def.undoScope === 'FULL_TURN') {
    this.scenarioPlot = entry.effect.scenarioPlotBefore;           // <-- mismo BUG, rama PLOT
  }
}
```

Con `revertedEntries.length <= 1` (cierto para toda H1.3-H1.15, ver H1.15 spec §0.8) el bucle nunca sobrescribe nada — se ejecuta como mucho una vez por campo. **Con `RESOLVE_MINION_ACTION` (§0.3), `currentEnemyTurnLog` puede recibir una 2ª entrada `ATTACK` (o `PLOT`, si el Secuaz elegido usa una acción especial de rama Trama) en el mismo turno de Enemigo** — la del Enemigo mismo (vía `ACTIVATE_ABILITY`) y la del Secuaz (vía `RESOLVE_MINION_ACTION`). Si el jugador juega Contratiempo tras ese turno, `revertedEntries` tiene 2 entradas y el bucle actual asigna `this.leaderDamage = entry.effect.leaderDamageBefore` primero con el "antes" del Enemigo, luego lo SOBRESCRIBE con el "antes" del Secuaz — el resultado final dado dependde del "antes" de la ÚLTIMA entrada, no del acumulado real, y por tanto revierte de más o de menos según los valores concretos (ejemplo: Enemigo inflige 5 daño (0→5), Secuaz inflige 3 más (5→8); Contratiempo debería devolver `leaderDamage` a 0, pero el bucle actual lo deja en 5 — el "antes" de la entrada del Secuaz — perdiendo la reversión de los 5 del Enemigo).

**Fix exacto que debe aplicar el Programmer** — cambiar la semántica de "última entrada gana" a "primera entrada de cada contador gana" (equivalente a revertir al estado justo ANTES de la PRIMERA mutación de ese contador dentro de la ventana revertida — matemáticamente correcto sea cual sea el número de entradas, no solo 2):

```ts
let leaderDamageReverted = false;
let leaderShieldReverted = false;
let scenarioPlotReverted = false;
const allyLifeReverted = new Set<CardInstanceId>();

const revertedEntries = [...this.previousEnemyTurnLog];
for (const entry of revertedEntries) {
  if (def.undoScope === 'FULL_TURN' && entry.origin === 'ABILITY') {
    this.remainingCooldowns.set(entry.abilityId, entry.cooldownBefore);
  }
  if (!entry.effect) continue;

  if (entry.effect.kind === 'ATTACK') {
    if (!leaderDamageReverted) {
      this.leaderDamage = entry.effect.leaderDamageBefore;
      leaderDamageReverted = true;
    }
    if (entry.effect.target === 'LEADER') {
      if (!leaderShieldReverted) {
        this.leaderShield = entry.effect.leaderShieldBefore;
        leaderShieldReverted = true;
      }
    } else if (!allyLifeReverted.has(entry.effect.allyInstanceId)) {
      allyLifeReverted.add(entry.effect.allyInstanceId);
      this.alliesInPlay = this.alliesInPlay.map((a) =>
        a.instanceId === entry.effect.allyInstanceId ? { ...a, life: entry.effect.allyLifeBefore } : a
      );
    }
  } else if (def.undoScope === 'FULL_TURN' && !scenarioPlotReverted) {
    this.scenarioPlot = entry.effect.scenarioPlotBefore;
    scenarioPlotReverted = true;
  }
}
```

**Justificación de por qué "primera entrada gana" (y no "sumar deltas") es la forma correcta y más simple:** dado que `entry[i].leaderDamageBefore === entry[i-1].leaderDamageAfter` para toda `i > 0` (invariante estructural: son activaciones consecutivas del mismo turno de Enemigo, y `leaderDamage` solo lo mutan estas activaciones dentro de esa ventana — nada más escribe `leaderDamage` entre el primer y el último `ACTIVATE_ABILITY`/`RESOLVE_MINION_ACTION` del turno), revertir al `leaderDamageBefore` de la PRIMERA entrada de la cadena es equivalente a deshacer la suma completa de todos los deltas aplicados, sin necesidad de sumarlos explícitamente. Se generaliza igual para `leaderShield` (solo entre entradas `target: 'LEADER'`) y `scenarioPlot` (solo entradas `PLOT`, alcance `FULL_TURN`). La vida de Aliado (`allyLifeReverted`) usa la misma regla pero particionada por `allyInstanceId`, porque un turno con 2 ataques podría golpear a 2 Aliados distintos (cada uno se revierte a SU propio "antes", primera vez que aparece).

**Caso borde nuevo, cubierto por el mismo fix sin código adicional:** si el mismo Aliado es golpeado 2 veces en el turno (Enemigo + Secuaz, ambos redirigidos a él), `allyLifeReverted` se puebla en la primera aparición y la segunda entrada para ese `allyInstanceId` se ignora — revierte correctamente a la vida ANTES del primer golpe de ese turno, no a la de después del primero/antes del segundo.

**`cooldownBefore`/`origin`:** ver §0.6 — las entradas `MINION_PLANO_ATTACK` no tienen `abilityId`/`cooldownBefore` (no hay habilidad de catálogo detrás), así que el fix añade el guard `entry.origin === 'ABILITY'` antes de tocar `remainingCooldowns` (el código anterior asumía implícitamente que TODA entrada tenía `abilityId`, cierto hasta ahora porque solo `ACTIVATE_ABILITY` escribía en el log).

### 0.5 Keyword Defensor: contrato puro y testeado, NO integrado end-to-end (no existe todavía mecánica de "jugador ataca al Enemigo/Secuaz")

GDD §3.8: *"Keyword Defensor: obliga a recibir/atacar a ese secuaz primero."* Esto es el espejo, en el tablero del Enemigo, de lo que Berserker/redirección ya hacen en el tablero del Líder (H1.15) — pero para que tenga un efecto real en el motor hace falta una mecánica de "el jugador ataca al Enemigo/a un Secuaz", que **no existe en `domain/combat` hoy** (`validateAbilityEffectsConfig` invariante 2 sigue prohibiendo que un efecto `ATTACK` pertenezca a una habilidad `side: 'LEADER'`; no hay `enemyHealth`; H1.18 es quien introduce "condición de victoria: vida Enemigo ≤0" — sin un mecanismo de ataque del jugador contra el Enemigo, Defensor no tiene nada que interceptar todavía).

**Decisión, mismo espíritu que H1.15 §0.4 con DEFENSOR-en-Aliado:** esta historia declara el dato (`MinionDefinition.isDefensor`, `MinionInPlay.isDefensor`) y expone una función pura, testeada, lista para que una historia futura (cuando exista "el jugador ataca al Enemigo/tablero enemigo") la invoque sin rediseñar nada:

```ts
/**
 * GDD §3.8: si hay ≥1 Secuaz Defensor vivo en mesa, CUALQUIER ataque dirigido al tablero
 * del Enemigo (contrato futuro, fuera de alcance de esta historia — ver §0.5) debe
 * resolverse contra él, nunca contra el Enemigo directamente ni contra otro Secuaz. Si
 * hay más de uno (no esperado por contenido, pero no prohibido por tipos), se elige el
 * primero por orden de entrada en mesa — determinista, mismo criterio que el Berserker
 * (H1.15, resolveDamageTarget).
 *
 * "No se puede ignorar" (AC) se satisface por CONTRATO de firma: esta función no acepta
 * ningún parámetro para saltarse al Defensor — quien la invoque no tiene forma de pedir
 * "el objetivo elegido por el jugador" si hay un Defensor vivo, exactamente igual que
 * resolveDamageTarget (H1.15) no permite ignorar a un Berserker vivo.
 */
export function resolveDefenderMinion(minionsInPlay: readonly MinionInPlay[]): MinionInPlay | null {
  return minionsInPlay.find((m) => m.isDefensor) ?? null;
}
```

**No se llama desde ningún `dispatch()` de `CombatEngine` en esta historia** — no hay ningún comando hoy que modele "el jugador ataca al tablero enemigo" para conectarla. Se deja documentado como forward-declaration (mismo trato que H1.15 dio a `INCREASE_ALLY_MAX_HEALTH`/`ALLY_NO_WARMUP_ABILITY` en `evolution-template.ts`). Se recomienda a Coordinator abrir un ítem de backlog explícito ("conectar Defensor cuando exista ataque del jugador al Enemigo") en vez de dejarlo implícito.

### 0.6 `UndoableEnemyActionLogEntry` gana un discriminante `origin`

Hasta H1.15, toda entrada del log venía de `ACTIVATE_ABILITY` y siempre tenía `abilityId`/`cooldownBefore`. Con `RESOLVE_MINION_ACTION` aparecen entradas sin habilidad de catálogo detrás (el "ataque plano"). Se añade:

```ts
export type UndoableEnemyActionLogEntryOrigin = 'ABILITY' | 'MINION_PLANO_ATTACK';

export interface UndoableEnemyActionLogEntry {
  readonly origin: UndoableEnemyActionLogEntryOrigin; // NUEVO H1.16
  readonly sourceId: string;
  /** Solo presente si origin === 'ABILITY' (una acción especial de Secuaz SÍ tiene
   *  abilityId/cooldownBefore, igual que el Enemigo mismo; el ataque plano no). */
  readonly abilityId?: AbilityId;
  readonly cooldownBefore?: number;
  readonly effect?: /* misma unión ATTACK(LEADER|ALLY)/PLOT que H1.15, sin cambios */;
}
```

**Migración de código existente:** toda entrada creada por `handleActivateAbility` (Enemigo mismo o Secuaz vía acción especial) construye `origin: 'ABILITY'` con `abilityId`/`cooldownBefore` presentes (sin cambio de comportamiento). Las entradas de "ataque plano" (`RESOLVE_MINION_ACTION`, fallback) construyen `origin: 'MINION_PLANO_ATTACK'` sin esos 2 campos. La aplicación del pasivo (§0.7) construye una 3ª variante de origen — ver ahí.

### 0.7 Efecto pasivo: leído y aplicado por `CombatEngine` al INICIO de cada turno de Enemigo

GDD §3.8: *"mientras están en mesa, cada esbirro aporta un efecto pasivo… acumulable entre ambas fuentes, sin tope"*. AC: *"Efecto pasivo es leído por CombatEngine cada turno del enemigo."* Se aplica en `handleEndTurn`, en el mismo punto donde hoy se hace `tickCooldownsForSide(this.turnOwner)` — SOLO cuando `this.turnOwner` (el lado que RECIBE el turno) es `'ENEMY'`:

1. Sumar `passiveEffect.amount` de TODOS los `minionsInPlay` agrupado por `kind` (`ATTACK`/`PLOT`) — sin tope, sin filtro de CD/Núcleo (es presencia pasiva, no una acción).
2. Si la suma `ATTACK` > 0: resolver exactamente como cualquier Ataque (mismo `resolveDamageTarget()`/Escudo/Aliado que H1.6/H1.15), `arrollar` = `true` si CUALQUIER Secuaz en mesa declaró `arrollar: true` en su pasivo (unión simple, sin desglose por Secuaz — decisión de simplicidad, documentada, no bloqueante: contenido de un solo Secuaz por combate en el MVP hace esto irrelevante en la práctica).
3. Si la suma `PLOT` > 0: sumar directamente a `scenarioPlot` (dirección `INCREASE`, mismo criterio que cualquier `PLOT` de `side: 'ENEMY'`).
4. Emitir un único evento nuevo `MINION_PASSIVE_EFFECTS_APPLIED` con el desglose (`attackAmount`, `plotAmount`, `minionCount`) — 0 si no hay Secuaces en mesa (el evento se emite igual, informativo, con montos en 0; más simple para tests que un evento condicional).
5. Si `attackAmount > 0` o `plotAmount > 0`, empujar una entrada a `currentEnemyTurnLog` con `origin: 'MINION_PLANO_ATTACK'` reutilizado como "origen sin habilidad" (se documenta como un pequeño overload de nombre — alternativa: añadir un 3er valor `'MINION_PASSIVE'` a `UndoableEnemyActionLogEntryOrigin`; **se prefiere esta segunda opción**, más legible — `UndoableEnemyActionLogEntryOrigin = 'ABILITY' | 'MINION_PLANO_ATTACK' | 'MINION_PASSIVE'`) — para que Contratiempo también pueda revertir el pasivo del turno (GDD §2.7 no excluye ningún tipo de mutación del turno de Enemigo).

**Orden dentro de `handleEndTurn`:** aplicar el pasivo DESPUÉS de `tickCooldownsForSide`/reset de contadores de acción (mismo lugar donde H1.14 abre/cierra la ventana de Contratiempo) y ANTES de congelar `previousEnemyTurnLog` — de forma que el pasivo del turno que EMPIEZA quede dentro de la ventana revertible de Contratiempo del PRÓXIMO `END_TURN` del Líder, exactamente igual que cualquier `ACTIVATE_ABILITY`/`RESOLVE_MINION_ACTION` de ese mismo turno de Enemigo.

### 0.8 Fuera de alcance explícito

- **Conectar el `effectDescription` "invoca un secuaz" de una carta de Dramaturgia concreta a `SUMMON_MINION` automáticamente.** Sigue narrativo/no ejecutable (mismo estado que dejó H1.10) — se resuelve cuando H1.18 module la resolución completa de una carta de Dramaturgia, no solo su `icon`.
- **Contenido de catálogo (JSON) de un Secuaz real.** Esta historia es de motor puro (mismo patrón que H1.3-H1.7): `minionDefinitions` se inyecta en tests igual que `allyCards`/`contratiempoCards` se inyectan hoy sin `CatalogLoader` de por medio. No se toca `packages/data/enemies/*.json` ni se crea un `MinionDefinition` de catálogo — se recomienda a Coordinator un ítem de backlog de contenido si se quiere un Secuaz jugable end-to-end antes de H1.18.
- **Eliminar/matar Secuaces.** Sin mecanismo de daño del jugador al tablero enemigo (ver §0.5), no hay forma de que un Secuaz salga de mesa en esta historia.
- **Defensor conectado a un flujo real de "el jugador ataca".** Ver §0.5 — contrato puro y testeado, integración end-to-end fuera de alcance.
- **Combo del lado Enemigo/Secuaz.** `validateAbilityComboConfig` sigue exigiendo `side: 'LEADER'` para cualquier `abilityId` en `abilityCombo` — no se toca, un Secuaz nunca genera una 3ª acción.

### 4.3 `handleSummonMinion` — nuevo método

Validaciones, en orden: `MINION_DEFINITION_UNKNOWN` (si `minionDefinitionId` no está en `minionDefinitions`) → `NOT_YOUR_TURN` (`expected: 'ENEMY'`). ### 4.4 `handleResolveMinionAction` — nuevo método, contiene la lógica de §0.3 punto por punto

Validaciones en orden: `NOT_YOUR_TURN` (`expected: 'ENEMY'`) → `MINION_ACTION_ALREADY_RESOLVED_THIS_TURN`. Luego:

1. Filtrar `specialCandidates` = `minionsInPlay` con `specialActionAbilityId` definido, `remainingCooldowns.get(specialActionAbilityId) === 0`, y `poolHasValidNucleo(abilityCoreCosts.get(specialActionAbilityId)!, nucleoPool)` (reutiliza `poolHasValidNucleo`, `enemy-ai.ts`, sin cambios).
2. Si `specialCandidates.length > 0`: `chosen = randomSource.pick(specialCandidates)`; `nucleoDecision = decideEnemyNucleoToSpend(abilityCoreCosts.get(chosen.specialActionAbilityId!)!, nucleoPool, derivePlayerColorsFromLeaderAbilities(...), randomSource)` (mismas 2 funciones de `enemy-ai.ts`, H1.7, sin cambios); ejecutar el mismo camino de efecto que `handleActivateAbility` para `(chosen.specialActionAbilityId!, chosen.instanceId, nucleoDecision.nucleo)` — **reutilizar, no duplicar**, extrayendo de `handleActivateAbility` la porción "gastar Núcleo + resetear CD + resolver `abilityEffects` + construir `effectLogEntry`" a un método privado compartido (ver §4.5) invocado desde ambos flujos con `side: 'ENEMY'` fijo y `sourceId = chosen.instanceId`. Push a `currentEnemyTurnLog` con `origin: 'ABILITY'` (igual que hoy). Emitir `MINION_ACTION_RESOLVED { instanceId: chosen.instanceId, mechanism: 'SPECIAL_ACTION' }` DESPUÉS de los eventos del efecto compartido (mismo orden relativo que `COMBO_TRIGGERED` en H1.14, "último evento de la activación").
3. Si `specialCandidates.length === 0` y `minionsInPlay.length > 0`: `chosen = randomSource.pick(minionsInPlay)`; aplicar `planoAttackAmount` vía `resolveDamageTarget()` + la misma bifurcación Escudo/Aliado que `applyAttackEffectToLeader`/`applyAttackEffectToAlly` (H1.6/H1.15, invocadas directamente con `rawAmount = chosen.planoAttackAmount`, `effectDef` sintético `{ kind: 'ATTACK', arrollar: false }` — no hace falta un `AbilityId`/Núcleo real, así que estas 2 funciones deben aceptar `nucleo: NucleoInstance | undefined` o una sobrecarga sin Núcleo — ver nota de refactor en §4.5). Push a `currentEnemyTurnLog` con `origin: 'MINION_PLANO_ATTACK'`, sin `abilityId`/`cooldownBefore`. Emitir `MINION_ACTION_RESOLVED { mechanism: 'PLANO_ATTACK' }`.
4. Si `minionsInPlay.length === 0`: emitir solo `MINION_ACTION_SKIPPED { reason: 'NO_MINIONS_IN_PLAY' }`.
5. En los casos 2 y 3, `this.minionActionResolvedThisEnemyTurn = true` (en el caso 4, se deja en `false` — un turno sin Secuaces no "gasta" el turno, aunque en la práctica es irrelevante porque sin Secuaces nunca habrá candidatos hasta que se invoque uno).

**Nota de refactor no opcional:** `applyAttackEffectToLeader`/`applyAttackEffectToAlly` (H1.6/H1.15) reciben hoy `nucleo: NucleoInstance` únicamente para incluirlo en el evento (`nucleoSpent`). Es el único cambio retro-compatible: todo caller existente (H1.6-H1.15) sigue pasando un `NucleoInstance` real, nunca `null` — cero regresión de comportamiento, solo se amplía el tipo.

### 4.5 `handleEndTurn` — inserción de la presencia pasiva (§0.7)

```ts

if (this.turnOwner === 'ENEMY') {
  const attackAmount = this.minionsInPlay.reduce(
    (sum, m) => sum + (m.passiveEffect.kind === 'ATTACK' ? m.passiveEffect.amount : 0), 0);
  const plotAmount = this.minionsInPlay.reduce(
    (sum, m) => sum + (m.passiveEffect.kind === 'PLOT' ? m.passiveEffect.amount : 0), 0);
  const arrollarAny = this.minionsInPlay.some(
    (m) => m.passiveEffect.kind === 'ATTACK' && m.passiveEffect.arrollar === true);

  if (attackAmount > 0) {
    // resolver vía resolveDamageTarget()/applyAttackEffectToLeader|ToAlly, mismo patrón que §4.4 punto 3,
    // effectDef sintético { kind: 'ATTACK', arrollar: arrollarAny }, nucleo: null, rawAmount: attackAmount
    // -> push a currentEnemyTurnLog con origin: 'MINION_PASSIVE'
  }
  if (plotAmount > 0) {
    this.scenarioPlot = Math.max(0, this.scenarioPlot + plotAmount);
    // -> push a currentEnemyTurnLog con origin: 'MINION_PASSIVE'
  }

  const passiveEvent: CombatEvent = {
    type: 'MINION_PASSIVE_EFFECTS_APPLIED',
    minionCount: this.minionsInPlay.length,
    attackAmount,
    plotAmount,
    leaderDamageAfter: this.leaderDamage,
    scenarioPlotAfter: this.scenarioPlot,
  };
  events.push(passiveEvent);
  this.eventBus.emit(passiveEvent);
}
```

**Orden exacto respecto al resto de `handleEndTurn`:** después del reset de contadores de acción, ANTES del bloque que mueve `currentEnemyTurnLog` → `previousEnemyTurnLog` (para que el pasivo del turno de Enemigo que EMPIEZA quede dentro de esa misma ventana — ver §0.7).
