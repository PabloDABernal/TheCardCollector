## 0. Diagnóstico — por qué hay franjas negras enormes en desktop

`Phaser.Scale.FIT` (configurado en `CombatScreen.tsx`, `mode: Phaser.Scale.FIT, autoCenter:
Phaser.Scale.CENTER_BOTH`) escala el canvas UNIFORMEMENTE para que quepa dentro del contenedor
disponible, preservando el aspect ratio del viewport virtual (`COMBAT_SCENE_VIEWPORT` =
1080×2060, ratio ≈ 0.524 — un retrato MUY estrecho). El factor de escala real es
`min(anchoDisponible / 1080, altoDisponible / 2060)`.

En una ventana de escritorio típica (ej. 1600×900, ratio 1.78 — panorámica), el término que limita el
escalado es SIEMPRE la altura (`900/2060 = 0.437`) porque el viewport virtual es mucho más alto en
proporción que ancho: el canvas resultante mide `1080×0.437 ≈ 472px` de ancho, solo el **29.5% del
ancho de la ventana** — el resto queda como franja negra vacía a cada lado. Este es el bug reportado
por el Director Creativo. El causante NO es un error de CSS ni de centrado — es que el viewport
virtual está diseñado 100% para retrato móvil (decisión original de H2.6/H2.1, "móvil primero") y
nunca se adaptó para pantallas panorámicas.

**Una solución completa** (viewport realmente panorámico, ej. 1700×1300, con topología de zonas
reorganizada de pila vertical a bandas horizontales) exigiría rediseñar la cadena de derivación
COMPLETA de `board-layout.ts` (§H5.1, ~200 líneas, todos los invariantes de H4/H5.1 recalculados
contra una geometría distinta) — alto riesgo, alto coste, y el propio Director pidió explícitamente
**secuenciar** el trabajo ("intenta centrarte en que se vea bien en navegador y luego pelearemos para
móvil" implica iteración, no una única solución perfecta de una vez). Esta historia entrega una mejora
real y acotada ahora (§1-§3 abajo); deja documentada como trabajo futuro explícito la opción de
rediseño topológico completo (§4).

## 1. Palanca 1 — ensanchar `COMBAT_SCENE_VIEWPORT` (1080 → 1280), altura sin cambios

`COMBAT_SCENE_VIEWPORT.height` (2060) NO cambia — es la dimensión que H4/H5.1 ya validaron
exhaustivamente (todos los invariantes verticales: Líder dentro de viewport, márgenes mínimos,
`board-layout.test.ts`). Solo cambia `width`, de 1080 a **1280**.

```ts
// packages/combat-scene/src/view/board-layout.ts
export const COMBAT_SCENE_VIEWPORT = { width: 1280, height: 2060 } as const; // ANTES: { width: 1080, height: 2060 }
```

### 1.1 Por qué esto es seguro (riesgo bajo, mismo patrón que el FIX P0 de H4)

La inmensa mayoría de las constantes X del archivo son YA fórmulas derivadas del ancho del viewport,
no literales sueltos:

- `x: 540` (SCENARIO_POSITION/ENEMY_POSITION/LEADER_POSITION/HAND_ROW_POSITION/todos los `PANEL_ZONES`)
  → pasa a `x: 640` (= `COMBAT_SCENE_VIEWPORT.width / 2`), un único punto de cambio si se sustituye el
  literal `540` por `COMBAT_SCENE_VIEWPORT.width / 2` en cada sitio donde aparece (Programmer: grep
  `x: 540` en `board-layout.ts`, son ~6 apariciones).
- `NUCLEO_PANEL_WIDTH = Math.round(COMBAT_SCENE_VIEWPORT.width * NUCLEO_PANEL_WIDTH_RATIO)` — YA es
  una fórmula; con `width=1280` pasa de 778px a 922px SIN tocar la línea — la mesa de Núcleos crece
  físicamente, dando más espacio real a los dados (uno de los focos explícitos del encargo:
  "espacio respirable alrededor" de H5.1 se acentúa, no se rompe).
- Ninguna constante Y depende de `width` — cero riesgo de romper el eje vertical ya validado.
- Precedente directo: el propio archivo documenta que H4 ("FIX URGENTE P0") ya subió `height` de 1920
  a 2060 con este mismo criterio de "recalcular fórmulas, ajustar tests si algún margen queda
  negativo" — mismo procedimiento, aplicado ahora al eje X en vez del Y.

### 1.2 Qué SÍ requiere revisión manual (no son fórmulas automáticas)

- `TILE_SEPARATION_PX` (140), `ABILITY_ICON_SEPARATION_PX` (200), `MINIONS_ROW_X_ORIGIN`/
  `ALLIES_ROW_X_ORIGIN` (200) — literales absolutos en px, no ratios. Con 200px más de ancho
  disponible (1280 vs 1080), Programmer puede (no debe) aumentarlos moderadamente para que
  cartas/habilidades se separen más (mejora legibilidad, ver §2) — pero el test de "sin solapes
  horizontales" (`board-layout.test.ts`) debe seguir pasando con el máximo de elementos del catálogo
  actual (4 habilidades por lado, hasta N Secuaces en mesa).
- `SIDE_ACTION_RAIL_X` (`docs/specs/H5.7_hud_lider_discreto.md` §3.1, 76px, calculado sobre el margen
  de 151px de la versión de 1080px) — el margen lateral libre crece a `(1280-922)/2 = 179px`; Programmer
  recalcula `SIDE_ACTION_RAIL_X` contra el nuevo margen (mismo criterio "centrado dentro del margen
  libre" de esa spec, con más aire disponible).

### 1.3 Resultado numérico esperado

Con `width=1280`, en la misma ventana de referencia 1600×900: `min(1600/1280, 900/2060) = min(1.25,
0.437) = 0.437` (sigue limitado por altura, sin cambio en el factor de escala) → canvas
`1280×0.437 ≈ 559px` de ancho, **34.9% del ancho de ventana** (antes 29.5%) — mejora real pero
modesta por sí sola, PORQUE el cuello de botella sigue siendo la ALTURA del viewport virtual (2060,
deliberadamente sin tocar). Esta palanca por sí sola no "resuelve" el problema — se combina con §2/§3
para que el resultado se perciba como una mejora genuina, no solo aritmética.

## 3. Palanca 3 — el margen lateral residual deja de ser negro vacío

Con `width=1280` (§1), en desktop panorámico sigue quedando un margen lateral considerable
(`.combat-screen-canvas-area` menos el ancho real del canvas). Hoy ese margen es `background: #000`
puro (heredado del `<body>`/contenedor raíz) — se sustituye por 2 cambios de bajo riesgo, sin tocar
Phaser ni `board-layout.ts`:

### 3.1 Fondo temático en vez de negro plano

```css
/* apps/shell/src/screens/CombatScreen.css — MODIFICADO */
.combat-screen-canvas-area {
  background: radial-gradient(ellipse at center, var(--binder) 0%, var(--ink) 100%);
  /* ANTES: heredaba #000 del body sin declaración propia */
}
```

Reutiliza los tokens ya definidos (`--binder`/`--ink`, `apps/shell/src/ui/tokens.css`) — el margen dejará
de leerse como "pantalla rota"/"contenido cortado" y pasa a leerse como marco decorativo intencional,
mismo lenguaje visual que el resto de la UI (`COLOR_BINDER` ya es el fondo de todos los paneles/modales).

### 3.2 `CombatLogPanel` se convierte en sidebar persistente en viewports anchos (≥1100px CSS reales)

Reutiliza el componente YA EXISTENTE (`apps/shell/src/combat/log/CombatLogPanel.tsx`, H4) sin cambios
de lógica — solo un breakpoint de layout que activa un `variant` de presentación distinto, usando el
margen lateral libre para algo funcional en vez de decorativo puro (coherente con el criterio explícito
del encargo: "espacio usado funcionalmente", no solo relleno visual).

```ts
// apps/shell/src/combat/log/CombatLogPanel.tsx — MODIFICADO
export interface CombatLogPanelProps {
  readonly entries: readonly CombatLogEntry[];
  /** NUEVO H5.8 — por defecto `'peek'` (comportamiento H4 sin cambios: franja inferior +
   *  bottom-sheet expandible). `'sidebar'`: panel lateral SIEMPRE expandido (sin franja peek, sin
   *  toggle), pensado para el margen lateral libre en desktop ancho — mismas entradas, mismo
   *  `colorForTone`, solo cambia el contenedor/posición. */
  readonly variant?: 'peek' | 'sidebar';
}
```

`CombatScreen.css` decide el `variant` vía un hook simple de breakpoint (`useIsWideViewport()`, mismo
patrón que el ya existente `useIsCompactViewport()` de `CombatHud.tsx` pero invertido — `matchMedia`
sobre `(min-width: 1100px)`), pasado desde `CombatScreen.tsx` a `CombatLogPanel`. En `variant='sidebar'`:
posición `position: fixed; right: 0; top: 0; bottom: 0; width: 320px`, panel siempre visible con scroll
interno (sin franja peek, sin pulso — el espacio ya está permanentemente visible, no necesita "atraer
la mirada" con una animación); en `variant='peek'`, comportamiento IDÉNTICO a H4 (sin cambios).

## 4. Fuera de alcance explícito — rediseño topológico completo (futuro)

Un viewport GENUINAMENTE panorámico (ej. 1700×1300, ratio >1) que llene la mayoría del ancho en
pantallas panorámicas exigiría reducir la altura virtual muy por debajo de 2060, lo que rompe la pila
vertical de 7 filas actual (Enemigo→Secuaces→Escenario→Mesa→Aliados→Mano→Líder) — no cabría sin
reorganizar al menos 2 de esas filas en columnas horizontales (ej. Enemigo+Escenario lado a lado
arriba de la mesa). Esto es un rediseño real de topología, con superficie de riesgo alta sobre TODOS
los invariantes de H4/H5.1 — se deja fuera de esta historia explícitamente, documentado aquí para que
Coordinator pueda convertirlo en una historia propia (`H6.x` o similar) si el Director Creativo lo pide
tras validar esta mejora intermedia. **No se elimina el "franjas negras" al 100% en esta historia** —
se reduce su proporción (§1) y se rellena de intención visual/funcional el resto (§3), lo cual es la
mejora que razonablemente cabe en el alcance "P0 bloqueante, corrección rápida" de este paquete.
