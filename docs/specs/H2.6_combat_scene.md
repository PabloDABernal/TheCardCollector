## 0. Qué resuelve esta historia (y qué NO)

### 0.1 Dentro de alcance de H2.6

1. Sustituir `HelloCombatScene` (H2.1, escena de prueba desechable) por una nueva clase de producción
   `CombatScene extends Phaser.Scene`, que será la base sobre la que H2.7 (`InputAdapter`) y H2.8

4. Configurar `Scale Manager` en modo `FIT` con viewport virtual `1080×1920` (`docs/architecture_stack.md`

5. Adaptar el harness standalone de `combat-scene` (`main.ts`) — hoy el único "caller" de la escena, ya que

### 0.2 Fuera de alcance de H2.6

- **Ninguna renderización de tablero, Núcleos, cartas en mano/mesa, HUD.** Eso es H2.8. `create()` no llama a
  `this.add.sprite/rectangle/text` para pintar estado de combate — solo prepara cámara/fondo/Scale Manager.
- **Ningún `InputAdapter`.** Eso es H2.7. `CombatScene` no registra listeners de `pointerdown`/`pointerup`

- **Ningún `<CombatScreen>`/`<PhaserMount>` de React, ni construcción del `CombatBridge` desde `apps/shell`.**
  Eso es H2.9. El harness `main.ts` de esta historia es un **sustituto temporal** de ese futuro punto de

- **Ningún test de `update()` custom.** Ver §2.3 — `update()` es intencionalmente un no-op documentado.

## 2. `CombatScene` — contrato y ciclo de vida

```ts
// packages/combat-scene/src/scenes/CombatScene.ts (contrato, no implementación completa)
import Phaser from 'phaser';
import type { CombatBridge, Unsubscribe } from '@collector/combat-bridge';
import { createEffectsDirector, JUICE_CONFIG, RECIPE_REGISTRY } from '../juice';

/** Viewport virtual de diseño — mobile-first, ver docs/architecture_stack.md §4.2. Misma resolución que
 *  main.ts (H2.1) ya usaba para el propio `Phaser.Game`; ahora también gobierna el `Scale Manager`. */
export const COMBAT_SCENE_VIEWPORT = { width: 1080, height: 1920 } as const;

/** Data que `CombatScene` espera recibir vía `scene.init(data)` (docs/architecture_stack.md §2.3: "el
 *  CombatEngine se crea en React/factory y se inyecta a Phaser vía scene.init(data)"). CombatScene NUNCA
 *  construye su propio CombatBridge/CombatEngine — los recibe ya construidos. */
export interface CombatSceneInitData {
  readonly bridge: CombatBridge;
}

export class CombatScene extends Phaser.Scene {
  constructor() {
    super('CombatScene');
  }

  /** Fase 1 del ciclo de vida Phaser — recibe `CombatSceneInitData`. Únicamente asigna `this.bridge`;
   *  no dispatch, no suscripciones, no side-effects (ver §2.1). Lanza si `data.bridge` falta — fallo
   *  temprano y explícito en vez de un `create()` silenciosamente roto más tarde. */
  init(data: CombatSceneInitData): void;

  /** No-op explícito en esta historia (ver §2.2) — sin assets propios de CombatScene todavía (H2.8 los
   *  añadirá: atlas de cartas, spritesheet de dados, fuentes). */
  preload(): void;

  /** Fase 3 — ver §2.1 para la secuencia completa (cámara, EffectsDirector.attach, registro de cleanup). */
  create(): void;

  /** No-op explícito (ver §2.3) — Phaser ya gestiona su loop de render; el juice es reactivo a eventos
   *  del CombatBridge, no a polling por frame. Documentado para que H2.7/H2.8 no asuman que aquí hay
   *  lógica oculta ni añadan código a update() por costumbre sin justificarlo explícitamente. */
  update(_time: number, _delta: number): void;
}
```

### 2.1 Secuencia de `create()`

1. `this.cameras.main.setBackgroundColor('#12141c')` (o el color de fondo provisional que decida Programmer —
   sin exigencia visual concreta en esta historia; solo evita el fondo transparente/negro por defecto de
   Phaser mientras H2.8 no pinta el tablero real).
2. Instanciar el `EffectsDirector` real: `const effectsDirector = createEffectsDirector(JUICE_CONFIG,
   RECIPE_REGISTRY)` (mismos símbolos ya exportados por `packages/combat-scene/src/juice/index.ts` desde
   H2.4/H2.5 — sin wiring nuevo de esos dos, solo consumo).
3. `const unsubscribe: Unsubscribe = effectsDirector.attach(this.bridge, this)` — conecta el
   `EffectsDirector` al `CombatBridge` inyectado en `init()` y a esta misma escena.
4. Registrar el cleanup en el evento de ciclo de vida `shutdown` de la escena (ver §2.4):
   `this.events.once(Phaser.Scenes.Events.SHUTDOWN, () => unsubscribe())`.
5. **No** se llama a `bridge.getSnapshot()` ni se pinta nada del estado de combate — eso es H2.8. `create()`
   termina aquí; la escena queda "viva" (cámara lista, `EffectsDirector` suscrito) esperando a que H2.7/H2.8
   añadan input y renderizado.

### 2.2 `preload()` — no-op explícito

Sin assets propios de `CombatScene` en esta historia (ningún atlas/spritesheet/fuente todavía — las 4 recetas
de H2.5 usan placeholders de `Rectangle`/`Text` generados en runtime, no requieren `preload`). Se deja el
método presente y vacío (no se omite de la clase) para que quede explícito en el código que la ausencia de
carga de assets es una decisión de esta historia, no un olvido — mismo criterio que `JUICE_CONFIG` (H2.4)
usa para entradas sin receta (`[]` explícito en vez de omitir la clave).

### 2.3 `update()` — no-op explícito, confirmado

`update()` no contiene lógica. Justificación explícita (para que no quede ambiguo en la review de H2.6/H2.7):

- Phaser ya gestiona el loop de render (RAF interno) — no hace falta un `update()` custom para que los tweens
  de las recetas de H2.5 avancen; el `Tween Manager`/`Timeline` de Phaser se actualiza automáticamente cada
  frame en el ciclo interno de la escena, independientemente de si `Scene.update()` tiene código o no.
- El "juice" es **reactivo a eventos** del `CombatBridge` (`subscribeSceneEvents` → `EffectsDirector` →
  receta), nunca por **polling** de estado en cada frame. No hay ninguna razón de diseño para leer
  `bridge.getSnapshot()` dentro de `update()`.
- H2.7 (InputAdapter) tampoco necesita `update()`: Phaser expone eventos de input (`pointerdown`, etc.) vía
  listeners registrados en `create()`/`attach()`, no vía polling en `update()`.
- Si una historia futura necesitase lógica por-frame genuina (p.ej. una barra de progreso continua no
  dirigida por eventos), se añadiría entonces con su propia justificación explícita — no se anticipa aquí
  (YAGNI), pero se documenta la firma con parámetros nombrados (`_time`, `_delta`, prefijo `_` por convención
  de "parámetro de interfaz no usado intencionalmente") para que quede claro que Programmer no la olvidó, la
  decidió vacía.

### 2.4 `shutdown`/`destroy` — sin fugas de listeners

`Phaser.Scene` no expone un método `shutdown()` de instancia por defecto que el propio Scene Manager invoque
directamente como override — el patrón correcto de Phaser 3 es suscribirse al **evento** de ciclo de vida
`Phaser.Scenes.Events.SHUTDOWN` (emitido por `this.sys.events`/`this.events`, alias del mismo emisor) cuando la
escena para (`scene.stop()`/`scene.start()` de otra escena la reemplaza) o `Phaser.Scenes.Events.DESTROY`
cuando el `Phaser.Game` completo se destruye (`game.destroy()`).

- **Decisión:** registrar el `unsubscribe()` del `EffectsDirector.attach(...)` en `Phaser.Scenes.Events.SHUTDOWN`
  (no en `DESTROY`) — `SHUTDOWN` cubre tanto "la escena para porque el juego termina" como "la escena se
  reinicia para un combate nuevo sin destruir el `Phaser.Game`" (caso relevante para H2.9: un modal de
  resultado que reinicia `CombatScene` con un nuevo `CombatBridge` para el siguiente combate de la run, sin
  recargar la página). Usar solo `DESTROY` dejaría el listener vivo (fuga) en ese escenario de reinicio.
- `this.events.once(...)` (no `.on(...)`): la suscripción de cleanup debe dispararse una única vez; si la
  escena se reinicia (`init()`/`create()` se llaman de nuevo), `create()` vuelve a registrar un `once` fresco
  sobre el nuevo `unsubscribe` de la nueva instancia de `attach(...)`.
- No hace falta guardar `unsubscribe` como propiedad de instancia (`this.unsubscribeEffects`) — basta con la
  variable local capturada por el closure del listener de `SHUTDOWN`, ya que solo se usa una vez y no se
  expone fuera de `create()`.

## 3. `build-default-combat-bridge.ts` — construcción del `CombatBridge` para el harness standalone

Sustituye la responsabilidad de `build-hello-engine.ts` (H2.1), migrando su lógica de construcción de
catálogo/engine, pero devolviendo un `CombatBridge` (H2.3) en vez de un `CombatEngine` crudo + snapshot:

```ts
// packages/combat-scene/src/build-default-combat-bridge.ts (contrato, no implementación completa)
import { createId, SeededRandomSource } from '@collector/domain-shared';
import type { LeaderId, EnemyId, ScenarioId } from '@collector/domain-shared';
import { CatalogLoader } from '@collector/domain-catalog';
import { CombatEngine, buildCombatEngineConfig } from '@collector/domain-combat';
import { createCombatBridge } from '@collector/combat-bridge';
import type { CombatBridge } from '@collector/combat-bridge';
import { loadRawContent } from './load-raw-content';

// Mismos 3 ids de contenido 2×2×2 que build-hello-engine.ts (H2.1) usaba — sin cambio de contenido,
// decisions.md "2026-07-06" ya fija reutilizar el 2×2×2 de H1 para H2.8-H2.9; este harness hace lo mismo
// desde antes, sin crear contenido nuevo.
const DEFAULT_LEADER_ID = 'leader-soldado-base';
const DEFAULT_ENEMY_ID = 'enemy-bestia-base';
const DEFAULT_SCENARIO_ID = 'scenario-bosque-encantado-base';
const HARNESS_SEED = 1;

/** Construye un CombatBridge real (motor + puente, sin snapshot expuesto por separado — quien lo necesite
 *  llama a `bridge.getSnapshot()`) contra el contenido real 2×2×2. Único consumidor hoy: `main.ts` (harness
 *  standalone de `combat-scene`, §4) — H2.9 sustituirá esta función por la construcción equivalente que
 *  viva en `apps/shell` (mismo patrón, catálogo real vía `loadRawContent`/fetch en vez de duplicar aquí). */
export async function buildDefaultCombatBridge(): Promise<CombatBridge> {
  const rawInput = await loadRawContent();
  const catalog = await new CatalogLoader(rawInput).load();

  const leader = catalog.leaders.get(createId<'LeaderId'>('LeaderId', DEFAULT_LEADER_ID) as LeaderId)!;
  const enemy = catalog.enemies.get(createId<'EnemyId'>('EnemyId', DEFAULT_ENEMY_ID) as EnemyId)!;
  const scenario = catalog.scenarios.get(createId<'ScenarioId'>('ScenarioId', DEFAULT_SCENARIO_ID) as ScenarioId)!;

  const randomSource = new SeededRandomSource(HARNESS_SEED);
  const config = buildCombatEngineConfig({ catalog, leader, enemy, scenario, randomSource });
  const engine = new CombatEngine(config);

  return createCombatBridge(engine);
}
```

- Mismo patrón exacto de `loadRawContent` (dual Node/navegador, H2.1) — sin tocarlo.
- `leaderMaxHealth`/`enemyMaxHealth` (que `HelloCombatResult` exponía junto al snapshot) **no** se necesitan
  aquí — `CombatScene` de esta historia no pinta vida/daño (H2.8 lo hará, leyendo `catalog`/`LeaderDefinition`
  de la misma forma que hacía `build-hello-engine.ts`, o resolviendo esos máximos donde H2.8 decida; fuera de
  alcance de H2.6 decidirlo ahora).

## 4. `main.ts` — harness standalone reescrito

```ts
// packages/combat-scene/src/main.ts (contrato, no implementación completa)
import Phaser from 'phaser';
import { CombatScene, COMBAT_SCENE_VIEWPORT } from './scenes/CombatScene';
import { buildDefaultCombatBridge } from './build-default-combat-bridge';

const game = new Phaser.Game({
  type: Phaser.AUTO,
  width: COMBAT_SCENE_VIEWPORT.width,
  height: COMBAT_SCENE_VIEWPORT.height,
  parent: 'app',
  scale: {
    mode: Phaser.Scale.FIT,
    autoCenter: Phaser.Scale.CENTER_BOTH,
    width: COMBAT_SCENE_VIEWPORT.width,
    height: COMBAT_SCENE_VIEWPORT.height,
  },
  scene: [], // deliberadamente vacío: CombatScene se añade/arranca a mano abajo, para poder pasarle
             // `CombatSceneInitData` en el `start()` (Phaser no permite inyectar `data` de init a una
             // escena listada directamente en `scene: [...]` de la config del Game).
});

game.scene.add('CombatScene', CombatScene);

void buildDefaultCombatBridge().then((bridge) => {
  game.scene.start('CombatScene', { bridge });
});
```

- **Scale Manager a nivel de `Phaser.Game`, no de `CombatScene`:** el modo `FIT`/viewport virtual se configura
  en el `config` del `Phaser.Game` (`scale: {...}`), no dentro de la escena — es una propiedad global del
  juego, no de una escena individual. `CombatScene` expone la constante `COMBAT_SCENE_VIEWPORT` (§2) para que
  `main.ts` (hoy) y `apps/shell`/`<PhaserMount>` (H2.9, mañana) usen el mismo número sin duplicar el literal
  `1080×1920`, evitando que diverjan si se ajusta el viewport en el futuro.
- Este `main.ts` es explícitamente el "harness standalone" mencionado en §0.1.5 — su forma (construir bridge,
  `game.scene.start(key, { bridge })`) es deliberadamente el mismo contrato que `<CombatScreen>` usará en
  H2.9, para que esa historia futura no descubra sorpresas de integración.

## 5. Verificación

### 5.1 Test unitario — lógica de inicialización sin canvas real (`combat-scene.test.ts`)

Mismo espíritu que `effects-director.test.ts` (H2.4, §5 de esta spec de referencia): no se instancia un
`Phaser.Game` real ni se depende de canvas/WebGL. Se instancia `new CombatScene()` directamente (el
constructor de `Phaser.Scene` no requiere un `Phaser.Game` real para construirse — solo falla si se intentan
usar propiedades como `this.add`/`this.cameras` antes de que el `SceneManager` las inicialice), y se
sobrescriben a mano las superficies mínimas de Phaser que `create()` toca, mismo patrón que `FakeJuiceScene`
(H2.5 §1.2) pero aplicado sobre la propia instancia bajo test:

```ts
// combat-scene.test.ts (contrato, no implementación completa)
// @vitest-environment node
import { describe, it, expect, vi } from 'vitest';
import type { CombatBridge } from '@collector/combat-bridge';
import { CombatScene } from './scenes/CombatScene';

function createFakeCombatSceneSurface(scene: CombatScene) {
  const shutdownListeners: Array<() => void> = [];
  (scene as any).cameras = { main: { setBackgroundColor: vi.fn() } };
  (scene as any).events = { once: (evt: string, cb: () => void) => evt === 'shutdown' && shutdownListeners.push(cb) };
  return { fireShutdown: () => shutdownListeners.forEach((cb) => cb()) };
}

describe('CombatScene — init/create/shutdown (H2.6)', () => {
  it('init(data) guarda el CombatBridge inyectado sin dispatch ni side-effects', () => {
    const scene = new CombatScene();
    const bridge = { subscribeSceneEvents: vi.fn(() => () => {}) } as unknown as CombatBridge;
    scene.init({ bridge });
    expect(bridge.subscribeSceneEvents).not.toHaveBeenCalled(); // create() aún no corrió
  });

  it('init(data) sin bridge lanza de forma explícita', () => {
    const scene = new CombatScene();
    expect(() => scene.init({} as any)).toThrow();
  });

  it('create() suscribe EffectsDirector al bridge inyectado exactamente una vez', () => {
    const scene = new CombatScene();
    const { } = createFakeCombatSceneSurface(scene);
    const unsubscribe = vi.fn();
    const bridge = { subscribeSceneEvents: vi.fn(() => unsubscribe) } as unknown as CombatBridge;
    scene.init({ bridge });
    scene.create();
    expect(bridge.subscribeSceneEvents).toHaveBeenCalledTimes(1);
  });

  it('shutdown de la escena invoca el unsubscribe de EffectsDirector.attach', () => {
    const scene = new CombatScene();
    const { fireShutdown } = createFakeCombatSceneSurface(scene);
    const unsubscribe = vi.fn();
    const bridge = { subscribeSceneEvents: vi.fn(() => unsubscribe) } as unknown as CombatBridge;
    scene.init({ bridge });
    scene.create();
    fireShutdown();
    expect(unsubscribe).toHaveBeenCalledTimes(1);
  });

  it('reconstrucción real: buildDefaultCombatBridge + CombatScene arrancan sin lanzar (sanity funcional)', async () => {
    const { buildDefaultCombatBridge } = await import('./build-default-combat-bridge');
    const bridge = await buildDefaultCombatBridge();
    const scene = new CombatScene();
    createFakeCombatSceneSurface(scene);
    scene.init({ bridge });
    expect(() => scene.create()).not.toThrow();
  });
});
```

el `CombatEngine`/catálogo real y verificar que no lanza — ahora contra `buildDefaultCombatBridge()` +
`CombatScene` reales, sin volver a aseverar la forma exacta del snapshot (eso seguía siendo responsabilidad de
H1/H1.19, ya cubierta ahí; repetirlo aquí sería redundante).
