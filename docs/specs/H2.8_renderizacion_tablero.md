# Spec H2.8 — Renderización de tablero, Núcleos, Líder/Enemigo/Escenario, Aliados/Secuaces y mano de cartas (componentes Phaser)

> Spec técnica del Architect para Programmer. Historia origen: `.ai-studio/memory/backlog.md`, Épica E2,
> "H2.8: Renderización de tablero, Núcleos, cartas en mano y en mesa (componentes Phaser)". Depende de H2.3
> (cerrada: `CombatBridge`), H2.4/H2.5 (cerradas: `EffectsDirector`/`JuiceConfig`/recetas reales), H2.6
> (cerrada: `CombatScene`), H2.7 (cerrada: `InputAdapter` de gestos genéricos). Precede a H2.9 (dispatch real
> desde inputs del jugador), H2.10 (cooldowns visuales animados), H2.11 (daño/Trama con floating text
> animado), H2.12 (animaciones de Núcleo gastado/rolleado). Es la historia más grande de la Épica E2 — esta
> spec es más extensa de lo habitual a propósito.

---

## 0. Qué resuelve esta historia (y qué NO)

### 0.1 El hallazgo que condiciona todo el alcance: no existe "mano" en el modelo de dominio

Antes de diseñar nada, una lectura literal de `CombatStateSnapshot` (`packages/domain/combat/src/types/snapshot.ts`)
y de `CombatCommand.PLAY_CARD` (`commands.ts`, comentario H1.18) revela que **el motor de combate no tiene
concepto de mano ni de mazo de robo**. `PLAY_CARD` baja cualquier carta de `leader.cardPoolIds` directamente
por `cardId` de catálogo, gateada solo por `leaderEnergy`/acciones — nunca por "¿la tengo en la mano ahora
mismo?". El propio comentario de `commands.ts` lo dice explícito: *"Baja una carta EVENTO/EQUIPO de
`cardPoolIds` (sin concepto de mano/mazo...)"*. Lo mismo aplica a `PLAY_ALLY`/`PLAY_CONTRATIEMPO`: cualquier
carta del pool fijo del Líder (~10 cartas, `H1.9`/`H1.12`), siempre disponible, gateada solo por Energía y
(para Aliado/Contratiempo) el propio efecto.

**Consecuencia de diseño:** "cartas en mano" en esta historia no es "un subconjunto robado del mazo que
cambia con el tiempo" (eso no existe todavía en el motor), sino **el pool fijo completo del Líder**,
resuelto una vez desde el catálogo al construir la escena — igual de "siempre visible" que ya lo es en la UI
del CLI (`packages/cli/src/renderer.ts` no imprime mano tampoco, imprime cooldowns/Núcleos/estado). Esta
spec renderiza ese pool fijo como tiles estáticos de "mano", con datos reales de catálogo (nombre, coste,
tipo) — no placeholders inventados. La `CombatBridge`/`CombatStateSnapshot` no exponen catálogo, así que la
data de estos tiles se resuelve **fuera** de `CombatBridge`, en el mismo punto donde ya se resuelve
`leaderMaxHealth`/`enemyMaxHealth` (ver §2).

### 0.2 Dentro de alcance de H2.8

Vertical slice completo de renderizado de **estado** (sin animación de transición propia — eso ya existe
vía juice H2.4/H2.5, o se refina en H2.10-H2.12):

1. **Tablero de fondo** (`createBoard`): zonas visuales fijas (Núcleos, Líder, Enemigo, Escenario, Aliados en
   mesa, Secuaces en mesa, mano) sin lógica de estado — capa puramente decorativa/de layout.
2. **Pool de Núcleos** (`createCorePool`/re-render): un tile por `NucleoInstance` de `snapshot.nucleoPool`,
   coloreado por `NucleoColor` real y con su `value` (1-4, o 0 si un debuff lo fijó ahí, GDD/decisions.md) como
   texto — la pieza más icónica del feel (forcetable.net/strawtable.net), con datos 100% reales del snapshot.
3. **Líder / Enemigo / Escenario** (`createRoleView` ×3): un game object "de rol" persistente por cada uno
   (nunca destruido/recreado, ver §3.4), nombrado con los mismos `FOCUS_ID_LEADER`/`FOCUS_ID_ENEMY`/
   `FOCUS_ID_SCENARIO` que `effects-director.ts` (H2.4) ya exporta y usa — para que las recetas de juice
   (`hitImpact`, `screenShake`) encuentren y animen exactamente este objeto vía `getByName` (ver §1). Cada uno
   lleva un texto HUD con datos reales: vida/daño, escudo, energía, nivel (Líder); daño/fase (Enemigo);
   Trama/fase (Escenario). Sin animación de floating text propia (eso es H2.11) — el texto se actualiza en
   el sitio, sin tween.
4. **Aliados en mesa** (`createAlliesView`): un tile por `AllyInPlay` de `snapshot.alliesInPlay`, nombrado
   con su `allyInstanceId` real (para que `ALLY_DAMAGED`/`ALLY_ENTERED_PLAY` en `resolveJuiceTarget` lo
   encuentren), mostrando vida real (`life`/`maxLife`) y si es Berserker.
5. **Secuaces en mesa** (`createMinionsView`): un tile por `MinionInPlay` de `snapshot.minionsInPlay`,
   nombrado con su `instanceId` real (para `MINION_SUMMONED`), mostrando si es Defensor.
6. **Mano del Líder** (`createCardHandView`): un tile estático por carta del pool fijo del Líder (§0.1),
   resuelto desde catálogo (nombre, coste, tipo EVENTO/EQUIPO/ALIADO/CONTRATIEMPO), nombrado con la
   convención `card-{cardId}` (ver §1.4) para que `CARD_PLAYED`/`ALLY_ENTERED_PLAY`/`CONTRATIEMPO_PLAYED`
   puedan encontrar exactamente ese tile — nunca el avatar del Líder (ver §1.3, el bug que esta convención
   evita). Atenuado visualmente (alpha reducido) si `leaderEnergy < energyCost` en el snapshot actual — sin
   más lógica de disponibilidad que esa.
7. **HUD de turno/energía/acciones** (texto simple, sin panel React): turno actual, dueño del turno,
   energía del Líder, acciones usadas/permitidas.
8. **CD resumido (texto plano, sin iconos ni tweens)**: una línea de texto por Líder/Enemigo con el estado
   de sus cooldowns (`snapshot.cooldowns`), mismo dato que ya imprime `renderCooldowns` del CLI — sin ningún
   tratamiento visual más allá de texto (iconos/barras/tween-on-tick son H2.10, explícitamente).
9. **Sincronización reactiva con el snapshot**: `BoardView.render(snapshot)` se llama una vez al construir
   la escena y de nuevo cada vez que el `CombatBridge` emite cualquier evento (§3.5) — sin loop de `update()`,
   mismo criterio "reactivo a eventos, nunca por polling" que H2.6 ya fijó para `EffectsDirector`.
10. `targetId` real (Núcleo/Aliado/Secuaz/carta de mano) en todo game object interactivo, consumible por el
    `InputAdapter` de H2.7 vía `getData('targetId')` — sin que esta historia traduzca esos gestos a
    `PlayerIntent`/dispatch todavío (eso es H2.9, ver §0.4).
11. Extensión de `CombatSceneInitData`/`buildDefaultCombatBridge` (H2.6) con el contexto de catálogo
    resuelto que el renderizado necesita (`BoardViewContext`, §2) — sin el cual `leaderMaxHealth`,
    `enemyMaxHealth`, nombres de carta/habilidad y el pool de mano no son derivables solo del `CombatBridge`.
12. Extracción de `NameLookup`/`buildNameLookup` (hoy solo en `packages/cli`) a `packages/domain/catalog`
    para que `combat-scene` lo reutilice sin duplicar lógica de resolución de nombres (§2.3) — cumple la
    instrucción de reutilizar componentes existentes en vez de reinventar.

### 0.3 Fuera de alcance de H2.8 (diferido explícitamente, con destino)

- **Ninguna animación de transición propia de este tablero.** `hitImpact`/`cardFlip`/`diceRoll`/`screenShake`
  YA EXISTEN (H2.4/H2.5) y ya están conectados (H2.6). Esta historia solo garantiza que esas recetas
  encuentren game objects **reales** en vez de placeholders efímeros (§1, §3.4) — no reimplementa ni
  modifica ninguna receta.
- **Ninguna animación específica de "Núcleo gastado desaparece"/"Núcleo rolleado entra" más allá de lo que
  `diceRoll` (H2.5) ya hace y de que el re-render (§3.5) refleje el pool correcto tras el evento.** El
  criterio literal de backlog ("al recibir CORE_SPENT el visual desaparece, al CORE_ROLLED aparecen nuevos")
  queda satisfecho por el mecanismo general de re-render (§3.5): el pool de Núcleos se redibuja completo
  contra `snapshot.nucleoPool`, que ya refleja el Núcleo gastado ausente / los nuevos presentes tras
  `ABILITY_ACTIVATED`/`NUCLEO_POOL_ROLLED`. La animación de *transición* (fundido de salida de un dado
  concreto) es H2.12, explícitamente.
- **Ningún floating text de daño/Trama, ni tween de "número que sube/baja parpadeando".** Eso es H2.11. Los
  textos HUD de esta historia se actualizan de golpe (`setText(...)`), sin tween.
- **Ningún icono/barra de cooldown, ni tween al bajar CD.** Eso es H2.10 (criterio propio: "tween de escala +
  color rojo→verde" en `COOLDOWNS_TICKED`). Esta historia solo expone el dato en texto plano (§0.2 punto 8).
- **Ningún dispatch de `CombatCommand` desde gestos del jugador.** El `InputAdapter` (H2.7) ya clasifica
  gestos; los tiles de esta historia ya llevan `targetId` real — pero **nadie traduce todavía** un `TAP`
  sobre `card-{cardId}` en un `PLAY_CARD` real. Eso es H2.9 ("conectar la interacción React (botones, menú)
  con el flujo de intents [...] usuario puede clickear una carta en mano → comando PLAY_CARD se dispatch").
  Ver §0.4 para el límite exacto y por qué.
- **Ningún robo/deck de cartas real** — no existe en el motor (§0.1); no se inventa aquí.
- **Ninguna evolución del contrato `PLAY_CARD`/`PLAY_ALLY`/`PLAY_CONTRATIEMPO` en `domain/combat`.** Cero
  cambios a `packages/domain/*` — esta historia es puramente de `combat-scene` + una extracción interna en
  `domain/catalog` (§2.3, sin cambio de comportamiento, solo de ubicación).

### 0.4 Límite exacto con H2.9 — por qué el dispatch no entra aquí

H2.9 es explícitamente "React aloja Phaser, **dispatch de comandos**, reacción visual" — su criterio de
aceptación literal incluye "Usuario puede clickear una carta en mano → comando `PLAY_CARD` se dispatch al
engine" como parte de integrar `<CombatScreen>`/`<PhaserMount>` con el flujo de intents completo. Meter el
dispatch en H2.8 obligaría a re-abrir esta historia cuando H2.9 defina cómo `InputAdapter.subscribe(...)`
se traduce a `PlayerIntent` de dominio (esa traducción todavía no existe, H2.7 §0.1 la difirió aquí mismo
"a H2.8/H2.9" sin comprometerse a cuál) — y H2.8 ya es la historia más grande de la épica. Se traza así:

- **H2.8 (esta historia):** el estado se **lee** del snapshot y se **pinta**; los tiles llevan `targetId`
  real y están listos para ser tocados, pero tocar un tile hoy no dispara ningún comando — como mucho, un
  `console.debug`/overlay de depuración opcional (mismo criterio no-bloqueante que el `debug-rect` de H2.7).
- **H2.9:** añade la capa de traducción `PointerGesture` → `PlayerIntent` de dominio → `bridge.dispatch(...)`,
  usando exactamente los `targetId`/nombres que esta historia fija (`card-{cardId}`, `allyInstanceId`,
  `nucleo.id`, `FOCUS_ID_LEADER`/`ENEMY`/`SCENARIO`) — sin inventar una convención nueva, sin rehacer
  ninguno de los game objects de esta historia.

## 1. Convenciones de identidad — `name` (juice) vs `data.targetId` (input)

Dos mecanismos independientes, ya fijados por historias previas, que **todo** game object persistente de
esta historia debe respetar simultáneamente donde aplique:

- **`name` + `scene.children.getByName(name)`** — usado por `resolveOrCreatePlaceholder`/
  `resolveOrCreateCardPlaceholder` (`juice/recipes/placeholder.ts`, H2.5) para que `hitImpact`/`cardFlip`
  animen un game object **real** en vez de crear un placeholder efímero. La clave de búsqueda es el
  `focusId` que `resolveJuiceTarget` (`effects-director.ts`, H2.4) calcula por tipo de evento (§1.3 de
  H2.4/tabla abajo).
- **`data.targetId` (`setData('targetId', ...)`) + `scene.input.hitTestPointer(...)`** — usado por
  `InputAdapter` (H2.7 §2.1) para resolver sobre qué game object cayó un gesto. Es un mecanismo
  **independiente** del anterior (una escena podría, en teoría, tener un `targetId` distinto de su `name`),
  pero en esta historia se fija la misma cadena para ambos en todo caso salvo Núcleos (ver tabla), para
  evitar dos convenciones a mantener sincronizadas sin motivo.

### 1.1 Tabla de identidad por tipo de game object

| Game object | `name` (para `getByName`, juice) | `data.targetId` (para `InputAdapter`) | Por qué |
|---|---|---|---|
| Líder (rol) | `FOCUS_ID_LEADER` ('leader') | `FOCUS_ID_LEADER` | `resolveJuiceTarget` usa `'leader'` para `LEADER_DAMAGED` |
| Enemigo (rol) | `FOCUS_ID_ENEMY` ('enemy') | `FOCUS_ID_ENEMY` | ídem `ENEMY_DAMAGED`/`PHASE_CHANGED(source:'ENEMY')` |
| Escenario (rol) | `FOCUS_ID_SCENARIO` ('scenario') | `FOCUS_ID_SCENARIO` | ídem `SCENARIO_PLOT_CHANGED`/`PHASE_CHANGED(source:'SCENARIO')` |
| Núcleo (dado en pool) | *(sin nombre — ver §1.2)* | `nucleo.id` (`NucleoInstanceId` real) | `NUCLEO_POOL_ROLLED.focusId` es siempre `undefined` (§1.2); solo `InputAdapter` necesita identificar el dado concreto |
| Aliado en mesa | `ally.instanceId` (`CardInstanceId`) | `ally.instanceId` | `resolveJuiceTarget` usa `event.allyInstanceId` para `ALLY_DAMAGED`/`ALLY_ENTERED_PLAY` |
| Secuaz en mesa | `minion.instanceId` | `minion.instanceId` | `resolveJuiceTarget` usa `event.instanceId` para `MINION_SUMMONED` |
| Carta de mano | `` `card-${cardId}` `` (§1.4) | `` `card-${cardId}` `` | convención NUEVA de esta historia — ver §1.3/§1.4 |

### 1.2 Por qué los Núcleos no llevan `name`

`resolveJuiceTarget('NUCLEO_POOL_ROLLED')` devuelve `{ event }` sin `focusId` (`effects-director.ts` línea
~36) — la receta `diceRoll` (H2.5) nunca hace `getByName` sobre un Núcleo individual: itera
`target.event.pool` y crea/destruye sus propios rectángulos efímeros de dado en cada llamada (`dice-roll.ts`,
ver §3.6 para el análisis completo de por qué esto es seguro). Nombrar los tiles persistentes del pool no
aporta nada a la resolución de juice — solo necesitan `data.targetId` para que H2.9 sepa, más adelante, qué
`NucleoInstanceId` seleccionó el jugador al tocar un dado.

### 1.3 El bug que esta spec evita: `sourceId === 'leader'` colisionaría con el rol del Líder

`CARD_PLAYED`/`CONTRATIEMPO_PLAYED` usan `resolveJuiceTarget → focusId: event.sourceId`. Si el futuro
dispatcher (H2.9) reutilizase `sourceId: 'leader'` (como hoy hacen los tests/harness de dominio, donde
`sourceId` es solo un string informativo de auditoría — `commands.ts`: *"es informativo/de auditoría en
H1.3, no participa en la validación"*), `cardFlip` haría `getByName('leader')` y **encontraría y animaría el
propio rol del Líder** (`createRoleView` de esta historia) como si fuera una carta — flip de escala + swap de
color sobre el avatar del Líder, en vez de sobre el tile de la carta jugada. Cosmético pero confuso, y un
acoplamiento accidental entre dos conceptos no relacionados.

**Decisión (contrato que esta historia fija para H2.9):** cuando H2.9 dispatch `PLAY_CARD`/`PLAY_ALLY`/
`PLAY_CONTRATIEMPO`, el `sourceId` del comando **debe** ser `` `card-${cardId}` `` (el mismo `name`/
`targetId` del tile de mano tocado, §1.4) — nunca `'leader'` ni ningún otro literal. `sourceId` sigue siendo
puramente informativo para el motor (no cambia su validación, `commands.ts` ya lo documenta así), así que
fijar esta convención no requiere ningún cambio en `domain/combat`. Esta spec deja constancia explícita para
que H2.9 no reintroduzca la colisión; Programmer de H2.8 no necesita hacer nada más que nombrar los tiles de
mano así (§1.4) — el dispatch en sí no existe todavía en esta historia.

### 1.4 Convención `card-{cardId}` para tiles de mano

```ts
export function cardTileName(cardId: CardId): string {
  return `card-${cardId}`;
}
```

Determinista y estable (mismo `cardId` de catálogo → mismo `name`/`targetId` siempre, sin importar cuántas
veces se reconstruya el tile) — se exporta desde `packages/combat-scene/src/view/card-hand-view.ts` (§3.3)
para que H2.9 la reutilice sin reinventar el prefijo.

## 2. Datos de catálogo que `combat-scene` necesita y no puede derivar de `CombatBridge`

### 2.1 `BoardViewContext` — nuevo contrato

```ts
// packages/combat-scene/src/view/board-view-context.ts (contrato, no implementación completa)
import type { NameLookup } from '@collector/domain-catalog';
import type { CardId } from '@collector/domain-shared';

/** Dato mínimo de una carta del pool fijo del Líder (§0.1) necesario para pintar su tile de mano —
 *  NO es `CardDefinition` completa (evita acoplar `combat-scene/view` a todo el esquema de catálogo,
 *  igual que `AllyCardDefinition`/`PlayableCardDefinition` en `domain/combat` ya "resuelven" solo lo
 *  que su capa necesita en vez de reexportar el catálogo crudo). */
export interface HandCardViewData {
  readonly cardId: CardId;
  readonly name: string;
  readonly energyCost: number;
  readonly cardType: 'EVENTO' | 'EQUIPO' | 'ALIADO' | 'CONTRATIEMPO';
}

/** Contexto de presentación resuelto UNA VEZ contra el catálogo (mismo momento/lugar que
 *  `buildCombatEngineConfig`, ver §2.2) — análogo directo a `RenderContext` de `packages/cli/src/renderer.ts`,
 *  reutilizando su mismo `NameLookup` (§2.3) en vez de reinventar resolución de nombres. */
export interface BoardViewContext {
  readonly nameLookup: NameLookup;
  readonly leaderMaxHealth: number;
  readonly enemyMaxHealth: number;
  readonly scenarioPlotDefeatThreshold: number;
  /** Pool fijo completo del Líder (§0.1), en el mismo orden que `leader.cardPoolIds` — orden estable
   *  para que el layout de mano (§3.3) no "salte" entre renders. */
  readonly leaderCardPool: readonly HandCardViewData[];
}
```

### 2.2 `CombatSceneInitData` (H2.6) — extendida

```ts
// packages/combat-scene/src/scenes/CombatScene.ts (extensión sobre H2.6)
export interface CombatSceneInitData {
  readonly bridge: CombatBridge;
  readonly boardContext: BoardViewContext; // NUEVO H2.8
}
```

`init(data)` (H2.6 §2.1, ya lanza si falta `bridge`) gana la misma validación temprana para
`boardContext` — lanza `Error` explícito si falta, mismo criterio de "fallo temprano" ya establecido.

### 2.3 Extracción de `NameLookup` a `domain/catalog` — sin cambio de comportamiento

`packages/cli/src/name-lookup.ts` (`NameLookup`, `buildNameLookup`) es hoy puro (`Catalog`,
`LeaderDefinition`, `EnemyDefinition` → mapa de nombres) y no depende de nada específico de CLI. Se mueve
tal cual a `packages/domain/catalog/src/name-lookup.ts`, exportado desde el barrel de `domain-catalog`.
`packages/cli/src/name-lookup.ts` pasa a ser un **re-export** de una línea (`export * from
'@collector/domain-catalog'`, o el equivalente nombrado) — cero cambio de comportamiento, cero cambio a los
tests existentes de CLI (siguen importando del mismo path relativo si Programmer lo prefiere, o se
actualizan a importar directo de `@collector/domain-catalog`; cualquiera de las dos es válida, la exigencia
es que la lógica viva en un solo sitio). `combat-scene` importa `NameLookup`/`buildNameLookup` directamente
de `@collector/domain-catalog` — sin duplicar la función.

### 2.4 `buildDefaultCombatBridge` (H2.6) — contrato de retorno extendido

```ts
// packages/combat-scene/src/build-default-combat-bridge.ts (extensión sobre H2.6)
export interface DefaultCombatSetup {
  readonly bridge: CombatBridge;
  readonly boardContext: BoardViewContext;
}

export async function buildDefaultCombatBridge(): Promise<DefaultCombatSetup> {
  const rawInput = await loadRawContent();
  const catalog = await new CatalogLoader(rawInput).load();
  const leader = catalog.leaders.get(/* ... igual que hoy ... */)!;
  const enemy = catalog.enemies.get(/* ... */)!;
  const scenario = catalog.scenarios.get(/* ... */)!;

  const randomSource = new SeededRandomSource(HARNESS_SEED);
  const config = buildCombatEngineConfig({ catalog, leader, enemy, scenario, randomSource });
  const engine = new CombatEngine(config);
  const bridge = createCombatBridge(engine);

  const nameLookup = buildNameLookup({ leader, enemy, catalog }); // @collector/domain-catalog, §2.3
  const leaderCardPool: HandCardViewData[] = leader.cardPoolIds.map((cardId) => {
    const card = catalog.cards.get(cardId)!; // garantizado por CatalogLoader, igual que catalog-adapter.ts
    return { cardId, name: card.name, energyCost: card.cost.energy, cardType: card.type };
  });

  const boardContext: BoardViewContext = {
    nameLookup,
    leaderMaxHealth: config.leaderMaxHealth,
    enemyMaxHealth: config.enemyMaxHealth,
    scenarioPlotDefeatThreshold: config.scenarioPlotDefeatThreshold,
    leaderCardPool,
  };

  return { bridge, boardContext };
}
```

**Cambio de firma que se propaga a `main.ts` (H2.6):** donde hoy hace
`buildDefaultCombatBridge().then((bridge) => game.scene.start('CombatScene', { bridge }))`, pasa a
`buildDefaultCombatBridge().then(({ bridge, boardContext }) => game.scene.start('CombatScene', { bridge,
boardContext }))`. Mismo patrón exacto que H2.9 replicará en `apps/shell` más adelante (H2.6 §3 ya lo
anticipaba: *"H2.9 sustituirá esta función por la construcción equivalente que viva en `apps/shell`"*) —
ese reemplazo futuro hereda el contrato `DefaultCombatSetup`, no lo rompe.

### 3.2 `role-view.ts` — Líder/Enemigo/Escenario (creados UNA VEZ, nunca destruidos)

```ts
// packages/combat-scene/src/view/role-view.ts (contrato, no implementación completa)
import type Phaser from 'phaser';
import type { CombatStateSnapshot } from '@collector/domain-combat';
import { FOCUS_ID_LEADER, FOCUS_ID_ENEMY, FOCUS_ID_SCENARIO } from '../juice'; // re-exportados por H2.4
import type { BoardViewContext } from './board-view-context';

export interface RoleView {
  /** Actualiza el texto HUD en el sitio (sin tween, §0.3) contra el snapshot actual. Nunca

```

Escenario) — sin exigencia visual concreta más allá de ser visualmente distinguibles; mismo nivel de detalle

### 3.4 `allies-view.ts` / `minions-view.ts` — colecciones que solo crecen, nunca se destruyen

Justificación de diseño (evita el conflicto #3 de la tarea — juice animando un objeto que el renderizador
persistente acaba de destruir a mitad de tween):

- `AllyInPlay`/`MinionInPlay` **nunca se eliminan** de `alliesInPlay`/`minionsInPlay` una vez invocados (H1.15
  §0.6, H1.16 §0.1 — un Aliado muerto queda con `life === 0`, un Secuaz permanece indefinidamente). Por tanto
  **no existe ningún caso en el MVP actual donde un tile de Aliado/Secuaz deba desaparecer** — la colección de
  tiles solo puede crecer, nunca encogerse. Esto hace segura la estrategia "crear si falta, actualizar si ya
  existe, nunca destruir":

```ts
// packages/combat-scene/src/view/allies-view.ts (contrato, no implementación completa)
import type Phaser from 'phaser';
import type { CombatStateSnapshot } from '@collector/domain-combat';

export interface AlliesView {
  /** Por cada AllyInPlay del snapshot: si no existe un game object con `name === instanceId`
   *  (mismo mecanismo de búsqueda que `resolveOrCreatePlaceholder`, reutilizado — ver nota abajo),
   *  se crea en la siguiente posición libre de la fila (`ALLIES_ROW_Y`); si ya existe, solo se
   *  reposiciona a su índice actual (por si el orden se hubiera alterado, defensivo) y se actualiza
   *  su texto de vida. Nunca se llama `.destroy()` sobre un tile de Aliado. */
  syncFromSnapshot(snapshot: CombatStateSnapshot): void;
}

export function createAlliesView(scene: Phaser.Scene): AlliesView;
```

`minions-view.ts` es el análogo exacto para `MinionInPlay`/`minionsInPlay`/`MINIONS_ROW_Y`, mostrando
`isDefensor` como una etiqueta de texto adicional.

**Nota de reutilización:** en vez de reimplementar "busca por `name`, crea si falta" desde cero,
`allies-view.ts`/`minions-view.ts`/`card-hand-view.ts` (para el caso — poco probable pero posible — de que
un `ALLY_ENTERED_PLAY` dispare `cardFlip` ANTES de que `syncFromSnapshot` corra por primera vez y ya exista
un placeholder de `juice/recipes/placeholder.ts` con ese `name`) reutilizan la misma función exportada
`resolveOrCreateCardPlaceholder`/`resolveOrCreatePlaceholder` de `juice/recipes/placeholder.ts` como
mecanismo de "resolver-o-crear" — evita que la carrera entre el primer `cardFlip` de un Aliado recién entrado
y el primer `syncFromSnapshot` posterior produzca DOS game objects con el mismo `name` (Phaser permite
nombres duplicados; `getByName` devuelve el primero que encuentra, así que un duplicado sería un bug visual
silencioso, no una excepción — por eso se reutiliza la misma función en vez de crear una paralela). Ver §3.7
para el veredicto final sobre `placeholder.ts` (no se modifica, se re-exporta).

### 3.6 Por qué el pool de Núcleos persistente NUNCA compite con `diceRoll`

`diceRoll` (`dice-roll.ts`, H2.5) itera `target.event.pool` y para cada Núcleo hace
`scene.add.rectangle(...)` (un rectángulo NUEVO, sin `name`, sin buscar nada existente vía `getByName`) y lo
`.destroy()` al terminar su propio tween — es 100% autocontenido y efímero, nunca toca ni conoce los tiles
persistentes de `nucleo-pool-view.ts`. Esto significa que cuando `NUCLEO_POOL_ROLLED` se dispara:

1. `EffectsDirector` (canal `subscribeSceneEvents`) dispara `diceRoll`, que hace aparecer sus propios dados
   efímeros "rodando" durante 500ms sobre `DEFAULT_PLACEHOLDER_POSITION` (o `params.poolOrigin` si se pasa).
2. `BoardView` (canal `subscribeHudEvents`, §3.8) llama `nucleoPoolView.syncFromSnapshot(bridge.getSnapshot())`,
   que destruye los tiles persistentes viejos y crea los nuevos ya en su posición final (`NUCLEO_POOL_ROW_Y`).

Ambos pueden ocurrir en cualquier orden relativo sin ningún error ni fuga — no comparten ningún game object.
El efecto visual neto (dados "rodando" que aterrizan en su sitio final) es una consecuencia feliz del orden
temporal, no algo que esta historia deba orquestar explícitamente. Si el feel pide sincronizar mejor el
punto de aterrizaje del tween de `diceRoll` con `NUCLEO_POOL_ROW_Y`, es un ajuste de **parámetro**
(`poolOrigin`) en `JUICE_CONFIG`/`RECIPE_REGISTRY`, fuera de alcance de renderizado — nota para H2.12.

### 3.7 Veredicto sobre `juice/recipes/placeholder.ts`: se mantiene sin cambios funcionales

Contestando explícitamente la pregunta abierta de la tarea: **`placeholder.ts` NO se retira ni se reescribe.**
Su propio mecanismo (`getByName` primero, crear placeholder de emergencia solo si no existe) es exactamente
"buscar primero un sprite real y solo crear placeholder si no existe" — ya está implementado desde H2.5, sin
saberlo, como un mecanismo genérico de fallback. Esta historia lo aprovecha tal cual dándole sprites reales

## 4. Wiring en `CombatScene.create()` (extensión sobre H2.6/H2.7)

```ts
// CombatScene.create() — extensión de H2.8 sobre el código ya existente de H2.6/H2.7
create(): void {
  this.cameras.main.setBackgroundColor('#12141c');

  const effectsDirector = createEffectsDirector(JUICE_CONFIG, RECIPE_REGISTRY);
  const unsubscribeEffects = effectsDirector.attach(this.bridge, this);

  this.inputAdapter = createInputAdapter();
  const unsubscribeInput = this.inputAdapter.attach(this);

  // NUEVO H2.8 — construcción + pintura inicial + suscripción de sincronización.
  const boardView = createBoardView(this, this.boardContext);
  boardView.render(this.bridge.getSnapshot());
  const unsubscribeBoard = this.bridge.subscribeHudEvents(() => {
    boardView.render(this.bridge.getSnapshot());
  });

  this.events.once(Phaser.Scenes.Events.SHUTDOWN, () => {
    unsubscribeEffects();
    unsubscribeInput();
    unsubscribeBoard();
  });
}
```

### 4.1 Por qué `subscribeHudEvents` y no `subscribeSceneEvents` para el re-render de estado

`subscribeSceneEvents` (H2.3) está documentado como "canal para consumidores juice (Phaser/`EffectsDirector`)"
y `subscribeHudEvents` como "canal para consumidores HUD no-juice (React overlay [...]) — vida, Trama,
turno, resultado final". `BoardView` es, semánticamente, exactamente ese segundo rol — un consumidor que
refleja **estado**, no que dispara **animación transitoria** — aunque viva físicamente dentro de Phaser en
vez de React. Se decide reinterpretar el canal `subscribeHudEvents` por su función (estado vs. juice), no
por su ubicación física (React vs. Phaser): ambos canales ya reciben el mismo stream de eventos sin
duplicación (H2.3 §3.2), así que suscribirse al canal "HUD" desde dentro de Phaser es válido y no rompe
ningún contrato existente — solo generaliza su interpretación. Se documenta aquí para que no se lea como una
inconsistencia sin explicar.

### 4.2 `init(data)` — extensión

```ts
init(data: CombatSceneInitData): void {
  if (!data?.bridge) {
    throw new Error('CombatScene.init: falta "bridge" ...'); // ya existente, H2.6
  }
  if (!data?.boardContext) {
    throw new Error('CombatScene.init: falta "boardContext" en CombatSceneInitData — necesario para render de tablero (H2.8).');
  }
  this.bridge = data.bridge;
  this.boardContext = data.boardContext; // NUEVO — propiedad de instancia, mismo patrón que this.bridge
}
```
